## 1. Identidade do projeto (por que molda o sistema de skills)

> "huu designs pipelines that make *thinking* agents follow a *deterministic* process. […] No LLM planner invents scope; the human underwrites the method, the agent supplies the intelligence." — AGENTS.md, Identity

Implicação direta: o sistema de skills deve seguir a mesma filosofia do produto — **o humano subscreve o método** (checkpoints, curadoria, diffs para revisão) e **o agente fornece a inteligência** (geração de rascunho, evolução proposta, nunca merge autônomo).

**Meta-fato decisivo:** as 9 skills legadas foram *geradas por um pipeline huu* — rodapé do `agent-skills.md`: "Generated by huu_audit-and-improve-skills v4.3.0". Ou seja, o repositório já viveu o cenário da pesquisa citada no prompt (Gloaguen et al., ETH Zurich, arXiv:2602.11988 — contexto LLM sem curadoria degrada sucesso e custo). O redesign atual corrige isso com curadoria humana obrigatória em cada fase.

## 4. Domínios e contratos

### 4.1 Schema de pipeline (v2)
- `WorkStep`: `type:'work'` opcional (compat v1), `name`, `prompt`, `files[]` (vazio = projeto inteiro), `scope: 'project'|'per-file'|'flexible'` (ausente = flexible/legado), `modelId?`, `next?`.
- `CheckStep`: `type:'check'` obrigatório, `condition` (com substituição do token `$runs`), `outcomes[]` (`label`, `nextStepName`, exatamente UM `default:true` por check — imposto por `validateTopology` via superRefine do Zod), `maxRuns?`.
- Defaults exatos: `DEFAULT_CHECK_MAX_RUNS=5` · `DEFAULT_MAX_NODE_EXECUTIONS=50` · `DEFAULT_CARD_TIMEOUT_MS=600_000` · `DEFAULT_SINGLE_FILE_CARD_TIMEOUT_MS=300_000` · `DEFAULT_MAX_RETRIES=1`.
- Formatos aceitos: `huu-pipeline-v2`, `huu-pipeline-v1` e legado `programatic-agent-pipeline-v1`.
- Guia humano: `docs/pipeline-json-guide.md` (âncora `#conditional-steps-check-nodes` confirmada).

### 4.2 Orchestrator (⚠ verificar linhas na Fase 3)
- Fluxo: preflight → loop de stages → decomposição (`task-decomposer`: 1 task/arquivo OU 1 rodada projeto-inteiro) → pool com worktree por agente → merge do stage → próximo. Cursor caminha o grafo respeitando `next`/outcomes.
- CheckStep: judge roda com `agentId` **reservado 9998** no worktree de integração; verdict = último bloco JSON da saída; cap `maxRuns` → dispara o outcome `default:true`; cada visita vira entrada em `OrchestratorState.checkRuns` → persistida em `RunManifest.checkRuns` (cards kanban em TUI e web).
- AutoScaler (`auto` é o default): alvo = headroom de RAM (disponível − margem 10%/512MiB) ÷ pegada por agente via EMA (seed 250MiB, clamp 128MiB–2GiB). `--concurrency=N` / `--no-auto-scale` pinam `manual`.
- MEMORY GUARD (ambos os modos): ≥95% RAM/CPU → mata o agente MAIS NOVO (`startedAt`), requeue na FRENTE da fila, contador `requeues` (badge `↻N`).

> "The killed-attempt marker is the consumable `killedAgentIds` Set in `orchestrator/index.ts` — never a status flag (see `requeue.test.ts` for the race + stale-flag regression)." — AGENTS.md

### 4.3 Camada git
- Worktrees em `.huu-worktrees/<runId>/` (respeita `HUU_WORKTREE_BASE`; abs = literal, rel = relativo ao repo). Exclusão via `.git/info/exclude`.
- Branches por `branch-namer.ts`: `huu/<runId>/agent-N` + `huu/<runId>/integration`. IDs reservados: 9998 judge, 9999 agente de integração.
- Merge determinístico: ordem ASCENDENTE de agentId, `--no-ff`. **Worktree de integração nunca rebobina** — loops de check acumulam commits sobre o HEAD atual.
- Commits internos com `--no-verify`; push com retry exponencial (≤3). **Stub não resolve conflito** — aborta na hora; só agentes reais têm essa permissão.

### 4.4 Backends (descoberta: AGENTS.md está DESATUALIZADO aqui)
- `backends/registry.ts` despacha `kind → factory` (interface AgentFactory). Kinds reais no código: `'pi' | 'copilot' | 'azure' | 'stub'` (enum do RunConfig).
- **Azure é backend real** (`src/lib/azure.ts`, `llm-client-factory.ts`, `docs/azure-backend.md` em pt-BR), mas a tabela de arquitetura do AGENTS.md lista só pi/copilot/stub → corrigir quando o sistema de skills tocar os docs.
- Resolução de API key (cadeia): `/run/secrets/<nome>` → `<VAR>_FILE` → env → config persistida (`~/.config/huu/config.json`) → prompt na TUI.
- Detecção de "thinking" por prefixo de modelId (anthropic/claude*, deepseek-r1*, openai/o1*, gemini-2.5*, glm-z1).

### 4.5 Isolamento de portas
- `port-allocator` (orchestrator): janela contígua por agente, base 55100, window 10, probe TCP contra overlap.
- `writeAgentEnvFile` → `.env.huu` (0600) com `PORT`, `HUU_PORT_*`, `HUU_PORT_REMAP` (CSV `de:para`, wildcard `*:`) e `LD_PRELOAD` (Linux) / `DYLD_INSERT_LIBRARIES` (macOS) apontando para o shim.
- `writeAgentBinShim` → `.huu-bin/with-ports` (bash: source `.env.huu` + exec). **Gotcha central:** o LD_PRELOAD só vale se o processo for lançado VIA with-ports — injetar no `process.env` do orchestrator quebraria o isolamento por agente.
- Compilação cacheada em `.huu-cache/native-shim/<os>-<arch>/`; em Docker, `HUU_NATIVE_SHIM_PATH` aponta o pré-compilado. Matriz de cobertura: `docs/PORT-SHIM.md` (fonte da verdade).

### 4.6 Runtime Docker
- Ordem de bypass do `decideReexec` (primeira que casa vence): `HUU_IN_CONTAINER=1` → `--yolo` → `--no-docker` → `HUU_NO_DOCKER=1|true` → `--help|-h` → subcomandos nativos (`init-docker`, `status`, `prune`) → senão, re-exec.
- cidfile em `/tmp/huu-cids/cid-<pid>-<rand>.id`; `prune` mata órfãos com pai morto. Imagem default `ghcr.io/frederico-kluser/huu:latest`, override `HUU_IMAGE`. Rede com MTU auto-detectado p/ VPN (`huu-net-mtu<N>`). Secrets via `--mount` readonly (excluídos do `-e`).
- `--web` exige `--yolo` fora do container — gate em `src/cli.tsx:20–33`, ANTES do decideReexec (rejeita antes de docker pull).

### 4.7 Pipelines default (7)
- Fonte: `src/lib/default-pipelines/<nome>.ts` + `registry.ts`; só `testSuite` tem `_default:true`. Materialização idempotente: `pipeline-bootstrap.ts` checa `existsSync` ANTES de escrever — **nunca sobrescreve** (deletar `pipelines/<nome>.pipeline.json` re-materializa).
- `knowledge-protocol.ts` exporta: `knowledgeProtocol()` (bloco read-before/append-after), `reportJudgeCondition()` (condição do judge dos 5 audits report-only), `persistenceCheck()` (ajuste de `.gitignore`), notas de ordering/campos opcionais.
- Contrato guardado por `registry.test.ts`: nomes/arquivos únicos, schema+topologia válidos, 1 judge por audit com default `approved` e `maxRuns ≤ 3`, marcador "REPORT-ONLY" no prompt, caps (`maxRetries ≤ 3`, `maxNodeExecutions ≤ 50`).

> Audits são report-only: escrevem SÓ em `.huu/audits/**` + no máximo UM ajuste de `.gitignore`; "They never touch README.md, package.json, […] or any production source." — AGENTS.md. Exceções por design (setup pipelines): `huu Test Suite` e `huu Agent Knowledge`.

### 4.8 FSM + TUI
- `screen-fsm.ts`: reducer PURO compartilhado por TUI e web (15 estados: welcome, faq, pipeline-assistant, pipeline-editor, pipeline-import[-paste|-custom], pipeline-export, saved-pipelines, backend-selector, model-selector, api-key, timeout-prompt, run, summary). I/O fica no chamador (app.tsx / web/session.ts).
- Tokens em `src/ui/theme.ts`: ai (magenta), aiAccent, border/cursor (cyan), success, warning, error, info (blue).

> "`theme.ai` (magenta) is reserved for AI-driven UI: Smart Select […], Pipeline Assistant, Project Recon, agent logs. Non-AI components must not introduce magenta." — AGENTS.md (confirmado no código; exceção documentada: card `conflict_resolving`, que É um agente LLM)

- Gotchas Ink: `cardHeight()` do RunKanban orça linhas por card — adicionar linha renderizada sem atualizar o orçamento estoura `maxCardRows`; `FULL_CLEAR='\x1b[3J'` antes de re-render para evitar artefatos no resize; `useInput` com refs estáveis (re-attach não pode engasgar com o tick ~1Hz do SystemMetricsBar); RunKanban memoizado.

### 4.9 Modo web
- `server.ts`: HTTP+WS em 127.0.0.1, token UUID por processo (`?t=`), comparação timing-safe, heartbeat 30s, serve `dist-static` com anti path-escape.
- `session.ts`: 1 WebSession por conexão — espelho do app.tsx (resolve I/O e despacha eventos no FSM puro). `orchestrator-bridge.ts` (StateCoalescer): ~8Hz (125ms), flush imediato em frames terminais.
- `ws-protocol.ts`: **18 tipos client→server** (nav, pipeline.*, backend.select, model.*, apiKey.submit, files.scan, assistant.prompt, recon.start, run.{start,abort,setConcurrency,setAutoScale}, ping) e **13 server→client** (hello, screen, state, pipelines, models, files, assistant.*, recon.*, apiKey.required, result, error). Mensagem nova exige entrada em `CLIENT_MSG_TYPES` + type guard.
- `webui/`: Atomic Design estrito (atoms ⊂ molecules ⊂ organisms ⊂ templates ⊂ pages), estado via hooks próprios (`useWs` + WsContext/`useWsSession` — **não encontrado: zustand/redux**), Tailwind `darkMode:'class'` com token `ai` (fuchsia, espelho do TUI).

## 5. Convenções de código, teste e docs

**Código** (evidências do levantamento em ~10 arquivos):
- Arquivos kebab-case; ESM puro com **sufixo `.js` em imports relativos** (moduleResolution Bundler); **sem barrels** (`orchestrator/index.ts` importa 21 exports nomeados direto dos vizinhos).
- Exports nomeados como regra. ⚠ Divergência a curar: skill legada dizia "NO default exports"; o levantamento atual observou default export em componentes React — verificar no código antes de fixar a regra na skill nova.
- Erros: `throw new Error` + Zod para validação; `process.exit` só em entrypoints CLI; logger de debug engole erros de disco silenciosamente (by design).
- `lib/types.ts` é a fonte única de tipos (~25 interfaces); discriminated unions + type guards (`isCheckStep`); JSDoc denso em exports; comentários em EN/PT misturados.
- Pureza de módulo no topo do `cli.tsx`: hoisting de imports faz efeito colateral de módulo rodar TAMBÉM no caminho do wrapper Docker — corpos de módulo devem ser puros.

**Testes**: vitest `describe/it/expect`; colocados (`<módulo>.test.ts`); **git real** em `mkdtempSync` + cleanup (sem mock de git); factories stub ad-hoc (`makeKillableFactory`, `StubAssistantChat`); sem fake timers nos analisados. Regressões nomeadas que funcionam como spec: `requeue.test.ts` (kill→TODO→re-run com contador) e `registry.test.ts` (contrato dos 7 defaults).

**Commits/CI**: Conventional Commits estrito (`feat|fix|docs|chore|refactor|merge`; scopes observados: pipelines, cli, orchestrator, docker, azure, ui,web, smoke, tui, kanban, backend…). *Não há CI* — gate manual:

> "Run `npm run typecheck && npm test` before every commit. **There is no automated CI**" — AGENTS.md (hook opt-in: `git config core.hooksPath .githooks`)

**Docs/idioma**: raiz é pt-BR-primeiro (`README.md` pt + `README.en.md`, idem MANIFESTO); `docs/` mistura EN com variantes `.pt-BR.md` (ci, onboarding, operations). CHANGELOG em Keep a Changelog 1.1.0 com `[Unreleased]`. Para o sistema de skills isso abre a questão de idioma (seção 9).

## 6. Gotchas de alto sinal (resumo curado — candidatos diretos a skill)

1. `killedAgentIds` é um Set CONSUMÍVEL, nunca flag de status (race de flag obsoleta — `requeue.test.ts`).
2. Worktree de integração nunca rebobina; loops de CheckStep acumulam commits.
3. `pipeline-bootstrap` nunca sobrescreve → editar módulo default NÃO atualiza JSON já materializado do usuário.
4. Exatamente um `outcome.default:true` por CheckStep (validateTopology rejeita o resto).
5. LD_PRELOAD do shim só funciona via `.huu-bin/with-ports` (sourcing), nunca via env global do orchestrator.
6. Gate `--web`+`--yolo` roda ANTES do decideReexec (cli.tsx:20–33) — mover quebra a UX de erro.
7. `cardHeight()` ↔ linhas renderizadas do card devem andar juntos (overflow silencioso do kanban).
8. `theme.ai`/fuchsia só para UI dirigida por IA (TUI e webui).
9. Mensagem WS nova = tipo + `CLIENT_MSG_TYPES` + type guard + handler na session + acumulador no front (procedimento multi-arquivo).
10. Imports relativos exigem `.js`; corpo de módulo puro no caminho do cli.
11. Commits internos `--no-verify`; stub aborta em conflito de merge.
12. Sem CI: typecheck+test+smokes são responsabilidade de quem commita.
