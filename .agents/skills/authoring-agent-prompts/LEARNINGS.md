- [2026-06-28][source:inference][task:test-suite-code-frozen][probation] To track a known bug WITHOUT turning the suite red, skip/disable/ignore (`@Disabled`, `t.Skip`, `#[ignore]`, `test.todo`) are WRONG — they run ZERO assertions and silently park the test, hiding regressions. Only the xfail FAMILY runs the desired-behavior assertion and (strict) flips red when the bug is fixed: pytest `@pytest.mark.xfail(strict=True)` + `xfail_strict`, vitest `test.fails`, RSpec `pending`. Go/Rust/JUnit5 have NO native xfail → characterization (assert ACTUAL output) only. Put the per-stack idiom in a doc the fan-out reads (huu-tests.md), not hardcoded in each prompt.
- [2026-06-28][source:inference][task:test-suite-code-frozen][probation] Banning the easy path (edit source to green) pushes a small model toward worthless-green tests (assertion-free / tautological / self-mocked). Pair the ban with a mechanical anti-cheat JUDGE clause (≥1 concrete equality assertion per new test; ban weak-only sole assertions; unit-under-test never self-mocked) and a per-test mutation-strength self-check ("would an assertion fail if one operator in the function flipped?"). Make freeze + anti-cheat clauses HARD — never waived by the `$runs>=2` lean-approve shortcut, which is for ONE stubborn green-blocker only.
- [2026-06-28][source:inference][task:test-suite-code-frozen][probation] A multi-branch step that says "pick EXACTLY ONE" but where one branch is itself "do BOTH b1 and b2" and another "ALSO add X" is self-contradicting for a small model — it does only one sub-act and orphans the rest. Phrase such logic as a decision TREE (one action per leaf; "ALWAYS do both", conditional "THEN only if …"), and do NOT reuse A/B/C labels across adjacent steps for non-aligned meanings (a model pattern-matches the prior step's letters). Pull lookups (id-collision dedup) out of the inline branch into a one-line rule.
