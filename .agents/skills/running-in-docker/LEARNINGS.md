- [2026-06-25][source:inference][task:npm-deploy-v2.0.0][probation] Multi-arch GHCR build+push in this env needs a buildkit builder whose CONTAINER is on the host network: `docker buildx create --name huu-host --driver docker-container --driver-opt network=host --bootstrap`. The pre-existing `huu-builder` had only the worker RUN-step `network=host`, but its container ran on `bridge` — so BuildKit's OWN registry pulls (the `# syntax=docker/dockerfile:1.7` frontend + `node:20-slim` base) failed with TLS-handshake-timeout to registry-1.docker.io, even though the daemon reaches Docker Hub fine (`docker pull` works). Register arm64 emulation first: `docker run --privileged --rm tonistiigi/binfmt --install arm64`. Use `--provenance=false` to keep the manifest to exactly amd64+arm64 (no `unknown/unknown` attestation entries).
- [2026-06-25][source:inference][task:npm-deploy-v2.0.0][probation] GHCR push needs `docker login ghcr.io -u <user> --password-stdin` with a NON-expired classic PAT carrying `write:packages`. Diagnose a `denied: denied` push fast: `curl -sS -D - -o /dev/null -H "Authorization: token <PAT>" https://api.github.com` → `401` means the token is invalid/expired (NOT a scope problem; a valid token returns 200 + an `x-oauth-scopes:` header to check for `write:packages`). The cached buildx build makes a re-push after re-login take seconds.
- [2026-06-25][source:inference][task:web-ui][probation] Web-UI mode runs the server INSIDE the container, so the host browser needs the port published: `buildDockerArgv` grew a `publishPorts?: number[]` → `-p <n>:<n>` (added BEFORE the image, like `--network`), threaded from `reexecInDocker`'s `ReexecOptions`. cli.tsx top-level decides web-vs-cli with the pure `decideInterfaceMode` (loaded there because it pulls no React/Ink — safe on the wrapper path) and pins `process.env.HUU_WEB_PORT` so host+container agree; `HUU_WEB_PORT/HUU_WEB_HOST/HUU_WEB_TOKEN/HUU_CLI` were added to the passthrough set. Front-end (web/CLI) is ORTHOGONAL to the Docker bypass: `decideReexec` is unchanged, so `huu --yolo` = web native, `huu --cli` = TUI in Docker. Inside the container the server must bind `0.0.0.0` regardless of `HUU_WEB_HOST` (the published port needs it) — `resolveWebHost` forces it when `HUU_IN_CONTAINER=1`. Dockerfile `EXPOSE 4888` is doc-only; the `-p` is what publishes.

- [2026-07-02][source:user][task:kernel-memory-ceilings][probation] Kernel ceilings (ROADMAP Fase 2.1) landed in BOTH runtimes; four non-obvious facts: (1) the container previously ran UNLIMITED (`buildDockerArgv` had zero resource flags) — it could take 100% of host RAM; now `buildMemoryLimitArgs()` emits `--memory = host total − osReserveBytes()` (→ cgroup memory.max), `--memory-swap = memory + HUU_SWAP_MAX_MB` (allowance ON TOP of memory, docker semantics: memory-swap is the SUM), `--pids-limit 8192`; escapes `HUU_DOCKER_MEMORY_MB` / `HUU_NO_MEM_LIMIT=1`. (2) the operations.md claim that a negative `HUU_OOM_SCORE_ADJ` "takes effect as root inside the container" was WRONG — the container runs `--user <uid>:<gid>` (non-root), so writing a NEGATIVE adj no-ops there exactly like natively (needs CAP_SYS_RESOURCE). The effective lever is the opposite direction: `oom-child-watcher.ts` sweeps /proc every 2 s and RAISES huu's descendants (agent tool children) to +500 — raising needs no privilege (empirically pinned by a real-/proc test) — so kernel OOM kills a vitest worker (→ task retry), not the orchestrator/desktop. (3) native Linux self-wrap: `cgroup-self-wrap.ts` re-execs into `systemd-run --user --scope` at the SAME module-purity slot as the docker gate (top of cli.tsx, before heavy imports), gated by `decideCgroupWrap` (skips container/help/status/prune/wrapped/HUU_NO_CGROUP) and a definitive probe (`systemd-run --user --scope -- true`, ~tens of ms — existence of the binary is NOT enough, the user session bus is the real dependency); re-exec command = `process.execPath + process.execArgv + argv.slice(1)` so tsx dev loaders survive; the PROBE must carry the FULL -p property set (old systemd errors on unknown MemorySwapMax — a bare probe passes while the real wrap dies), the wrapper must FORWARD SIGINT/SIGTERM/SIGHUP to the child (kill <pid> otherwise orphans the scoped app) and map exit signals faithfully (128+N via os.constants.signals — the kernel OOM this feature creates is SIGKILL/137, not 143). docker --memory needs a `docker info {{.MemoryLimit}}` probe (rootless without memory delegation REJECTS the flags — degrade to unlimited, never block). (4) SIZE THE SCOPE BY THE OS RESERVE, NOT THE DIAL: `MemoryHigh = total − reserve`, `MemoryMax = total − reserve/2`. Sizing by the dial double-dials — the sampler is cgroup-aware, so inside the scope `ramTotalBytes` BECOMES memory.max and the internal budget would compute dial% × dial-sized-total (50% dial → 30% effective). Reserve-sized scope = host-safety line fixed at boot; the dial stays the live-tunable utilization target within it. All `HUU_RAM_PERCENT`/`HUU_GUARD_*`/reserve/cap knobs are now in the docker env passthrough Set (a host `HUU_RAM_PERCENT` was silently ignored in-container before).
