- [2026-06-25][source:inference][task:sequential-projects-queue][probation] Browser-driven sequential runs ride the existing single-run server: WebRunManager throws "already in progress" → HTTP 409 while active, then flips phase done/error + nulls the orch on settle, so the next /api/run is accepted. Pattern: on a terminal SSE phase, archive the final `state`, then POST the next run; retry on a transient 409 (≤30× / 350ms covered every observed gap). Guard double-archive with a latched current.runId + a processed-runId Set (SSE replays the last frame to new connections).
- [2026-06-25][source:inference][task:sequential-projects-queue][probation] server.startRun derives the backend from `provider` (openrouter→pi, azure→azure) and only falls back to body.backend when NO provider is sent; it IGNORES opts.lockedBackend. So `--web --stub` does NOT make browser-initiated runs use stub — POST `{backend:'stub'}` with no provider (stub.requiresApiKey=false) to drive a real no-LLM run for local verification.
- [2026-06-25][source:inference][task:sequential-projects-queue][probation] Per-card cost exists ONLY on AgentStatus; StageIntegration (merge) and CheckRun (judge) have no `cost` field — their LLM cost folds into OrchestratorState.totalCost (authoritative project total). - [2026-06-25][source:inference][task:sequential-projects-queue][probation] No-browser verification recipe that worked: (1) keep DOM-free logic in its own module (db.js touches indexedDB/document only inside functions) and import it in Node; (2) `fake-indexeddb/auto` in a SCRATCH npm project (never add to project deps) to round-trip the IndexedDB store; (3) boot `--web --stub` + curl assets/`/api/bootstrap` for the ESM graph + content-types; (4) drop the wrapped `.fixtures/smoke-pipeline.json` into `pipelines/` (parsePipelineFromJson accepts `{_format,pipeline}`) and drive `{backend:'stub', runDirectory:<temp git repo>}` to capture a real OrchestratorState. The stub run's worktrees land under the runDirectory's `.huu-worktrees/`, so a temp repo keeps the project clean.
- [2026-06-25][source:inference][task:run-log-realtime-stream][probation] The single `/events` SSE stream now carries TWO frame shapes distinguished by `frame.type`: `run` (throttled state snapshot, coalesced to ≤1/120 ms, last frame replayed on connect, per-agent logs capped at 200) and `agent-stream` (`{agentId,channel,text}`, raw output firehose — NOT throttled, NOT replayed, one frame per coalesced line, mirrored to `console.log`). The server's `broadcastAgentStream` writes the firehose DIRECTLY to every `sseClient` (bypasses the snapshot throttle) and short-circuits when `sseClients.size === 0`. Wiring: `WebRunManager` constructor gained a 3rd arg `onAgentOutput` that subscribes to `Orchestrator.subscribeAgentOutput`; `app.js` `connectSse().onmessage` branches on `frame.type`. Pattern to reuse: append-only / high-frequency data gets its own un-throttled frame type, never bloats the `run` snapshot.
- [2026-06-25][source:inference][task:run-log-realtime-stream][probation] `client/app.js` ships ONLY via the build's `cp -R src/web/client/. dist/web/client/` step (package.json `build`), because the server serves `clientDir()` = `./client/` relative to the module (dev→src, prod→dist). So a client edit is live under `npm run dev` (tsx) instantly but needs `npm run build` to reach the Docker/prod image — `tsc`/tests are GREEN either way (the file isn't compiled), so a stale-UI regression is silent. `node --check src/web/client/app.js` is the only syntax gate for the vanilla-JS client.
