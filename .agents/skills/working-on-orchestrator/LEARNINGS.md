Fixed in resource-monitor.ts via vm_stat reclaimable pages (free+inactive+purgeable+speculative). Symptom signature to recognize: status running, activeAgents 0, cost 0, no spawn logs.

- [2026-06-24][source:inference][task:greedy-max-mode][probation] Adding a scaler mode beyond auto/manual touches TWO non-obvious spots besides AutoScaler: (1) the per-tick target recompute in executeTaskPool is gated by `getMode() === 'auto'` — a new mode that needs a dynamic instanceCount must be added to that condition or it stays frozen at its seed; (2) the mode's enable method MUST call `portAllocator.setMaxAgents(AUTO_SCALE_MAX_INSTANCES)` like enableAutoScale(), else real concurrency is silently pinned at the manual port window (MAX_INSTANCES=20) however high targetConcurrency() returns. Added 'greedy' (UI label MAX) this way: targetConcurrency early-returns min(active+pending, maxAgents), shouldSpawn gates at the destroy threshold (both CPU and RAM, to match shouldDestroy's OR) and honors COOLDOWN for damping.
- [2026-06-24][source:inference][task:greedy-max-mode][probation] AutoScaleStatus.enabled is `this.enabled && mode === 'auto'` — it means "auto is DRIVING the target", NOT "scaler is on". The memory guard (shouldDestroy → destroyAgent: kill newest by startedAt, requeue to front of TODO with a requeues++ counter, killedAgentIds skips retry accounting) is mode-agnostic and runs in the poll loop for EVERY mode, so a new mode inherits kill-newest+requeue for free with zero guard changes. Consumers that read `enabled` as a proxy for "is auto" break once a third mode exists — branch on `mode` instead.
