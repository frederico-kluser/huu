- [2026-06-25][source:inference][task:npm-deploy-v2.0.0][probation] `npm publish` (`huu-pipe`) requires a 2FA OTP (account = "auth and writes"). NOT detectable beforehand: `npm whoami` succeeds, `npm access get status` only shows public/private, `npm profile get` doesn't reveal it — it surfaces only at publish time as `npm error code EOTP`. The agent can't generate it: have the user run `npm publish --otp=<6-digit>` via the `!` session-command prefix (fresh code, ~30s window; a placeholder/expired code re-errors EOTP). `npm publish` does NOT touch git, so a failed OTP leaves a clean tree to retry.
- [2026-06-25][source:user][task:npm-deploy-v2.0.0][probation] Versioning at 1.x+: a BREAKING change (removing the `huu --web` front-end) shipped as a MAJOR bump (1.4.0 → 2.0.0), NOT minor. The "0.x.x convention: breaking → minor" line in CLAUDE.md/CHANGELOG/this skill governed only the 0.x phase; past 1.0.0, breaking rides MAJOR (scripts/deploy.ts's own prompt already says "major — breaking changes em 1.0.0+"). User explicitly chose 2.0.0. The CHANGELOG header convention line is now stale for 1.x+.
- [2026-06-25][source:inference][task:npm-deploy-v2.0.0][probation] ALWAYS `rm -rf dist && npm run build` before `npm publish` after any source REMOVAL: `tsc` never prunes orphaned outputs and `prepack` only strips `*.test.*`/`*.spec.*`, so a stale `dist/` keeps compiled files whose source was deleted. Caught here: `dist/web/**` + `dist/cli-web.*` (the removed web mode) were about to ship inside 2.0.0. Gate the real publish behind `npm publish --dry-run` + `find dist -path '*web*'` (or equivalent) to confirm the tarball matches current source.
- [2026-06-25][source:inference][task:npm-deploy-v2.0.0][probation] `npm run deploy` (scripts/deploy.ts) is canonical but INTERACTIVE (readline: bump 1/2/3, confirm, Docker y/N) and pushes git BEFORE `npm publish`. For an agent run, do the steps manually with publish-BEFORE-push ordering (an OTP/network failure then leaves no dangling release commit+tag), mirroring deploy.ts's CHANGELOG move (`[Unreleased]` → `[X.Y.Z] - YYYY-MM-DD`) + compare-link update (`[Unreleased]: …/compare/vX…HEAD` and a new `[X.Y.Z]: …/compare/vPREV…vX`).
- [2026-06-26][source:inference][task:npm-deploy-v3.0.0][probation] GHCR push auth can be PRE-VALIDATED before the slow (emulated-arm64) multi-arch build, so an expired stored login fails in ~1s instead of after minutes: when `~/.docker/config.json` holds an INLINE base64 auth for ghcr.io (no credsStore/credHelper), extract it and `curl -s -o /dev/null -w '%{http_code}' -H "Authorization: Basic <auth>" "https://ghcr.io/token?service=ghcr.io&scope=repository:frederico-kluser/huu:pull,push"` → 200 = valid for push, 401/403 = expired. Keep the secret out of the transcript: assign the base64 to a shell var, print ONLY the HTTP code. At v3.0.0 the v2.x `docker login ghcr.io` was STILL valid AND the `huu-host` buildx builder + `qemu-aarch64` binfmt were already set up — so NO fresh PAT was needed at the gate. The skill's "ask for a fresh PAT" assumes logged-out; CHECK FIRST (`grep ghcr.io ~/.docker/config.json` + the probe) and skip the ask when the stored login still authenticates. npm was the ONLY hard blocker (OTP).
- [2026-06-26][source:inference][task:npm-deploy-v3.0.0][probation] `docker buildx build --push` pushes straight to the registry from the builder container and does NOT load the image into the local daemon — so the post-publish `./scripts/smoke-image.sh ghcr.io/frederico-kluser/huu:X.Y.Z` FAILS with "imagem … não existe local. Rode 'docker build -t huu:local .' primeiro" (the script only inspects the LOCAL image store, never pulls). Fix: `docker pull ghcr.io/frederico-kluser/huu:X.Y.Z` FIRST, then smoke (the pull also re-confirms the pushed multi-arch digest). v3.0.0's published-image smoke then passed clean (huu --help, tini PID 1, git, safe.directory wildcard, bind-mount worktree consistency, idle-safe HEALTHCHECK; size 686MB).

- [2026-06-26][source:user][task:changelog-v1.1.0-split][probation] huu does NOT publish GitHub Releases — distribution is npm + GHCR + git TAG + CHANGELOG, and this skill's flow never runs `gh release create`. The ONLY GitHub Release that ever existed (`v1.3.0`, 2026-06-10) was stale and still showed as "Latest" on the repo's Releases page while shipping was at v3.0.0; deleted on user request with `gh release delete v1.3.0 --yes` (NO `--cleanup-tag`, so the git tag — referenced by the CHANGELOG body + compare links — is preserved). Lesson: don't create GitHub Release OBJECTS as part of cutting a version (the annotated git tag IS the release here), and a lingering one silently mis-badges "Latest"; `gh release list` to audit.
- [2026-06-29][source:inference][task:npm-deploy-v3.1.0][probation] v3.1.0 (MINOR: web max-time-per-agent global Settings + the four merged web branches). npm-only by user scope — the GHCR half of the parallel flow was simply SKIPPED (it's optional; no harm). TWO refinements to the branch dance: (1) By release time `main` was ALREADY ahead of `development` — the repo's REMOTE AUTOMATION had merged `development` → `main` (twice: `Merge branch 'development'` commits), so `main..development` was EMPTY. So I did NOT merge dev→main myself; I cut the `chore(release): vX.Y.Z` commit directly ON `main` (where the content already sat), pushed `main` + tag, THEN fast-forwarded `development` UP to the release commit (`git checkout development && git merge --ff-only main && git push origin development`) — `development`'s tip was an ancestor of the release commit, so a clean FF. End state: all of main/development/origin both = the release commit at 3.1.0. Skipping the dev FF-back would strand development's stale `[Unreleased]` to collide with main's `[3.1.0]` on the next auto-merge. (2) CHANGELOG HYGIENE: merging several feature branches that each appended under `[Unreleased]` left a DUPLICATE `### Added` subsection (one near the top, one after `### Fixed`). Keep-a-Changelog wants ONE subsection per type per release — consolidate the duplicate `### Added`/`### Changed` blocks (preserve order Added→Changed→Fixed) when moving `[Unreleased]` → `[X.Y.Z]`. The 2FA-OTP gate worked exactly as the v2.0.0 entry documents (`! npm publish --otp=<fresh>` → `+ huu-pipe@3.1.0`; `npm view huu-pipe version` confirms 3.1.0).
