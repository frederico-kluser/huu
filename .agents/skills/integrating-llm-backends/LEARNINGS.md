- [2026-06-25][source:user][task:openrouter-key-401-shadow][promoted] "Valid key still 401s" is almost always the resolver picking a DIFFERENT key than the one the user set: env var (step 3) outranks the Options/config.json store (step 4), so a stale `OPENROUTER_API_KEY` (commonly exported from a shell profile or a sourced `~/.secrets`) shadows what the user saved. Diagnose with `checkOpenRouterReachable` vs a raw `curl https://openrouter.ai/api/v1/auth/key` — same endpoint, so a 200 on curl + 401 in huu means key MISMATCH, not a bad key. `dev`/`start` run native (`HUU_NO_DOCKER=1`), so the resolver sees the host env directly.

- [2026-06-25][source:user][task:invert-key-precedence][probation] SUPERSEDES the precedence half of the two `[task:openrouter-key-401-shadow]` entries above: the resolver was INVERTED so the explicitly SAVED store now OUTRANKS the env var. New order in `resolveApiKeyWithSource`: secret-mount → **stored** → env-file → env → none. The `shadowsStored` field was renamed to `storedOverridesEnv` (true only when `source==='stored'` AND a different non-empty ambient env value exists — an ignored env var). `keyRemedyHint`'s foot-gun message moved from `case 'env'` to `case 'stored'` (env now only reached as the fallback when nothing is saved). The web already preferred the browser key (`run-manager.ts` `params.apiKey || resolveApiKey`), so only the terminal resolver changed. Consumers updated: `docker-reexec` host note (`storedOverridesEnv`), `OptionsScreen` ⚠ row, `orchestrator/index.ts` 401 hint, `api-key.test.ts`.

- [2026-06-25][source:inference][task:run-log-realtime-stream][probation] The pi event-mapper (`backends/pi/event-mapper.ts`) historically consumed `message_update` NOT AT ALL — only agent_start / tool_* / message_end(usage) / agent_end / retry. But `session.subscribe` (factory.ts) receives EVERY `AgentSessionEvent`, and during generation pi fires MANY `message_update`s, each carrying `assistantMessageEvent` (a pi-agent-core `AssistantMessageEvent`: `text_delta`/`thinking_delta`/`{text,thinking,toolcall}_{start,end}`/start/done/error, each with a cumulative `partial: AssistantMessage`). Dropping them meant the agent's actual streamed reply+reasoning was captured NOWHERE and the run log only advanced at tool boundaries (looked frozen mid-generation — the "run log only works when I click stop" report). Fix: map `message_update` → new AgentEvent `{type:'stream',channel:'assistant'|'thinking',delta}` for `text_delta`/`thinking_delta` ONLY (cumulative `partial` + structural start/end/toolcall sub-events are log noise). `message_end` stays the usage/token source (unchanged); file WRITES go through the tool path, so assistant text is prose/narration, not file dumps. Backend-agnostic seam: any backend can now emit `stream` and get the live run log + console firehose for free (the stub does, which is how the web SSE test exercises it without a real LLM).
