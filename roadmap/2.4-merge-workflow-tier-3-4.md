## Best-Practice Guardrails (Git + AI + Automation)

- Use `ort` (default) for two-head merges; `recursive` is effectively an alias in modern Git.
- Use `-X ours` / `-X theirs` only as conflict-hunk preference inside `ort` merges.
- Do **not** confuse `-s ours` with `-X ours`: `-s ours` discards the other tree entirely.
- There is no `-s theirs` merge strategy; `theirs` is a strategy option (`-X theirs`) for conflict hunks.
- Use `octopus` only for multi-head, mostly non-conflicting merges; do not route Tier 3/4 through octopus.
- Prefer machine-readable conflict detection (`git merge-tree --write-tree`) before touching working tree.
- Keep AI conflict resolution behind validation gates (parseability, scope checks, tests, human review flags).
- Track resolution outcomes to learn per-file heuristics over time (rerere-like memory, but explicit and auditable).

## 1) Tier 3: Deterministic Heuristic Resolver

### O QUE E

Tier 3 is a **deterministic mechanical resolver** executed after Tier 2 conflict detection and before Tier 4 AI calls.  
It chooses a resolution side (`ours` or `theirs`) using a weighted policy that includes:

- last-touch signal,
- file ownership history,
- per-file past resolution success rates,
- file risk class (low-risk docs/config vs critical code paths),
- optional criticality hints from architecture or test ownership.

`last-touch-wins` remains useful but only as one feature in a composite decision.

### O QUE NAO E

- It is not semantic reasoning about business logic.
- It is not a blanket `-s ours` tree overwrite.
- It is not a replacement for Tier 4/human review on high-risk or ambiguous conflicts.
- It does not auto-resolve binary, rename/delete, or structurally ambiguous conflicts without explicit policy allowlist.

### COMO IMPLEMENTAR

1. **Collect conflict metadata**
   - Preflight with `git merge-tree --write-tree --name-only --no-messages <base> <incoming>` (or equivalent branch refs).
   - On actual merge failure, parse `simple-git` merge summary (`err.git.conflicts`) and `git.status().conflicted`.

2. **Classify conflict type and risk**
   - Low risk examples: markdown docs, changelog, generated artifacts with deterministic regeneration path.
   - High risk examples: auth, security, migration scripts, payment, infra deploy config.

3. **Compute side score**
   - Example weighted formula:
     - `last_touch_weight = 0.30`
     - `ownership_weight = 0.25`
     - `history_success_weight = 0.25`
     - `file_risk_weight = 0.20`
   - Decision must include confidence score.

4. **Apply deterministic threshold**
   - If `abs(score_ours - score_theirs) >= 0.25` **and** confidence `>= 0.75` **and** file risk is low/medium, Tier 3 can apply.
   - Otherwise escalate to Tier 4.

5. **Apply file resolution**
   - Use per-file resolution commands (for conflicted files only), stage, and continue merge.
   - Persist attempt + outcome in SQLite.

```ts
type Side = 'ours' | 'theirs';

interface Tier3Signals {
  filePath: string;
  conflictType: string;
  lastTouchSide?: Side;
  ownershipScore: { ours: number; theirs: number };
  historyScore: { ours: number; theirs: number };
  riskClass: 'low' | 'medium' | 'high';
}

function chooseTier3Side(s: Tier3Signals): { side: Side | 'escalate'; confidence: number } {
  if (s.riskClass === 'high') return { side: 'escalate', confidence: 0 };

  const lastTouch = s.lastTouchSide === 'ours' ? { ours: 1, theirs: 0 } :
    s.lastTouchSide === 'theirs' ? { ours: 0, theirs: 1 } :
    { ours: 0.5, theirs: 0.5 };

  const ours = 0.30 * lastTouch.ours + 0.25 * s.ownershipScore.ours + 0.25 * s.historyScore.ours + 0.20;
  const theirs = 0.30 * lastTouch.theirs + 0.25 * s.ownershipScore.theirs + 0.25 * s.historyScore.theirs + 0.20;
  const delta = Math.abs(ours - theirs);
  const confidence = Math.min(1, delta + 0.5);

  if (delta < 0.25 || confidence < 0.75) return { side: 'escalate', confidence };
  return { side: ours >= theirs ? 'ours' : 'theirs', confidence };
}
```

### CRITERIO DE ACEITACAO

- Tier 3 decisions are reproducible for the same input conflict bundle.
- High-risk conflicts never auto-resolve in Tier 3.
- Every Tier 3 attempt is recorded (decision, confidence, outcome).
- Failed or low-confidence Tier 3 transitions to Tier 4 without manual operator intervention.

### RISCOS

- **Risk:** biased preference can silently drop meaningful changes.  
  **Mitigation:** confidence threshold + risk gating + required test pass.
- **Risk:** rename/delete and binary conflicts are mishandled by simplistic file-level rules.  
  **Mitigation:** conflict-type gating; non-supported types bypass Tier 3.
- **Risk:** stale heuristics drift from current code ownership.  
  **Mitigation:** time-decay scoring and minimum-sample thresholds.

## 2) Tier 4: AI Resolver

### O QUE E

Tier 4 is a **semantic conflict resolver** invoked only when Tier 3 cannot safely decide.  
It asks an LLM to generate a minimal merge resolution patch for conflicted files, then validates before accepting.

### O QUE NAO E

- It is not "apply model output directly to main".
- It is not unrestricted code refactoring outside conflicted scope.
- It is not exempt from tests, validation, and mandatory human visibility.

### COMO IMPLEMENTAR

1. **Build a conflict context package (mandatory minimum)**
   - merge metadata: merge base SHA, ours SHA, theirs SHA, queue item id;
   - conflict body: base/ours/theirs versions or conflict-marker blocks;
   - local context: +/- N lines around each conflict hunk;
   - file metadata: path, language, ownership, risk class;
   - history: prior resolutions for same file + fingerprint + success outcomes;
   - code constraints: project conventions and relevant architecture guardrails;
   - verification context: failing tests/lint errors linked to conflict scope.

2. **Prompt contract**
   - Force model to output only:
     - unified diff touching conflicted files, or
     - complete content for each conflicted file in strict JSON envelope.
   - Require explicit rationale + confidence.

3. **Validation pipeline (hard gates)**
   - Parse output format.
   - Reject edits that touch non-conflicted files.
   - Apply in isolated worktree/index.
   - Run targeted verification (tests/lint/typecheck subset, then full gate if needed).
   - If validation fails: one bounded retry with structured feedback; then escalate human.

4. **Persist audit data**
   - model id, prompt hash, token usage, resolution confidence, validation outcome.
   - mark all Tier 4 outputs as human-review-required in queue/TUI metadata.

```ts
interface ConflictContextBundle {
  queueItemId: string;
  mergeBaseSha: string;
  oursSha: string;
  theirsSha: string;
  files: Array<{
    path: string;
    language: string;
    riskClass: 'low' | 'medium' | 'high';
    conflictHunks: Array<{ base: string; ours: string; theirs: string; surrounding: string }>;
    history: Array<{ strategy: string; outcome: 'success' | 'failed'; resolvedSide?: 'ours' | 'theirs' }>;
  }>;
  constraints: string[];
  failingChecks: string[];
}
```

### CRITERIO DE ACEITACAO

- Tier 4 receives complete context bundle (not only diff + history summary).
- Resolver output is machine-parseable and scoped to conflicted files.
- No Tier 4 merge is accepted without passing validation gates.
- Every Tier 4 result is flagged for human review visibility in TUI/logs.
- On repeated failure, item moves to human escalation path automatically.

### RISCOS

- **Risk:** hallucinated edits outside intended scope.  
  **Mitigation:** strict output schema + path allowlist + patch scope validator.
- **Risk:** false confidence from LLM output.  
  **Mitigation:** confidence is advisory only; tests and policy gates are authoritative.
- **Risk:** cost/latency spikes with oversized prompts.  
  **Mitigation:** capped context windows, prioritized snippets, bounded retries.

### Tier 4 Context Sufficiency Verdict

Previous definition ("conflict diff + per-file history") is **insufficient** for reliable semantic resolution.  
Minimum viable context must include base/ours/theirs content, surrounding code, risk metadata, and verification feedback.

## 3) Conflict History Tracking in SQLite

### O QUE E

A durable conflict telemetry layer that stores:

- what conflicted,
- what strategy was attempted (Tier 3 / Tier 4),
- what was chosen,
- whether it passed validation,
- whether human override was required.

This data feeds Tier 3 heuristics and operational reporting (hotspot files, failure patterns).

### O QUE NAO E

- It is not full repository snapshot storage.
- It is not unbounded log accumulation.
- It does not replace Git history; it complements it for decision support.

### COMO IMPLEMENTAR

Use normalized tables plus indexes:

```sql
CREATE TABLE IF NOT EXISTS merge_conflicts (
  id TEXT PRIMARY KEY,
  queue_item_id TEXT NOT NULL,
  file_path TEXT NOT NULL,
  conflict_fingerprint TEXT NOT NULL,
  conflict_type TEXT NOT NULL,
  merge_base_sha TEXT NOT NULL,
  ours_sha TEXT NOT NULL,
  theirs_sha TEXT NOT NULL,
  detected_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS merge_resolution_attempts (
  id TEXT PRIMARY KEY,
  conflict_id TEXT NOT NULL,
  tier INTEGER NOT NULL, -- 3 or 4
  strategy TEXT NOT NULL, -- x-ours, x-theirs, ai-patch, manual
  selected_side TEXT, -- ours/theirs when applicable
  confidence REAL,
  outcome TEXT NOT NULL, -- success/failed/escalated
  model_id TEXT,
  prompt_hash TEXT,
  applied_commit_sha TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY (conflict_id) REFERENCES merge_conflicts(id)
);

CREATE INDEX IF NOT EXISTS idx_merge_conflicts_path ON merge_conflicts(file_path);
CREATE INDEX IF NOT EXISTS idx_merge_conflicts_fingerprint ON merge_conflicts(conflict_fingerprint);
CREATE INDEX IF NOT EXISTS idx_merge_attempts_conflict ON merge_resolution_attempts(conflict_id);
```

Recommended heuristic query shape:

```sql
SELECT
  mc.file_path,
  mra.strategy,
  COUNT(*) AS attempts,
  SUM(CASE WHEN mra.outcome = 'success' THEN 1 ELSE 0 END) * 1.0 / COUNT(*) AS success_rate
FROM merge_conflicts mc
JOIN merge_resolution_attempts mra ON mc.id = mra.conflict_id
WHERE mc.file_path = ?
GROUP BY mc.file_path, mra.strategy
HAVING attempts >= 3
ORDER BY success_rate DESC, attempts DESC;
```

### CRITERIO DE ACEITACAO

- Every detected conflict writes one `merge_conflicts` row.
- Every Tier 3/Tier 4 attempt writes one `merge_resolution_attempts` row.
- System can query top-conflicting files and strategy success rates by file.
- Heuristic lookup uses data only when minimum sample threshold is met.

### RISCOS

- **Risk:** DB growth and slow queries over time.  
  **Mitigation:** retention policy + periodic aggregation.
- **Risk:** weak fingerprints produce poor history matching.  
  **Mitigation:** stable fingerprint from normalized hunks + conflict type + file path.
- **Risk:** history overfits old project patterns.  
  **Mitigation:** decay older data and cap strategy confidence contribution.

## 4) Human Escalation Path (Pause Queue + Notify TUI)

### O QUE E

A deterministic, auditable path for unresolved or unsafe merges:

- pause merge queue item,
- notify operator with actionable payload,
- support explicit resume/retry/manual/abort actions.

### O QUE NAO E

- It is not a silent failure fallback.
- It is not a global hard-stop for the entire orchestration by default.
- It is not a full manual merge editor implementation in this stage.

### COMO IMPLEMENTAR

1. **State machine**
   - `pending -> running -> blocked_human -> resolved | failed | aborted`

2. **Escalation triggers**
   - Tier 3 below confidence threshold.
   - Tier 4 parse/validation failure.
   - high-risk conflict type.
   - retry budget exhausted.

3. **Escalation payload in SQLite `messages`**
   - `type = escalation`
   - include queue id, file list, attempted tiers, confidence, recommended actions.

4. **Operator actions**
   - `retry_tier4`
   - `accept_tier3_candidate`
   - `manual_resolution_committed`
   - `abort_merge_item`

5. **Audit trail**
   - every transition writes merge result + action provenance (who/when/why).

```ts
interface EscalationPayload {
  queueItemId: string;
  reason: 'tier3_low_confidence' | 'tier4_validation_failed' | 'high_risk_conflict' | 'retry_budget_exhausted';
  conflictedFiles: string[];
  attempted: Array<{ tier: 3 | 4; strategy: string; confidence?: number; outcome: string }>;
  recommendedActions: Array<'retry_tier4' | 'accept_tier3_candidate' | 'manual_resolution_committed' | 'abort_merge_item'>;
}
```

### CRITERIO DE ACEITACAO

- Any unresolved conflict transitions to `blocked_human` with persisted escalation message.
- TUI/CLI can list pending escalations with all required context.
- Operator action deterministically moves queue item to next valid state.
- No merge commit is finalized while item is `blocked_human`.

### RISCOS

- **Risk:** deadlock on queue items waiting forever.  
  **Mitigation:** SLA timer + reminder/escalate-to-owner policy.
- **Risk:** operator overload from noisy escalations.  
  **Mitigation:** deduplicate similar escalations and prioritize by risk.
- **Risk:** missing context at escalation time slows manual recovery.  
  **Mitigation:** enforce payload completeness contract before state transition.
