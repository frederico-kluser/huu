# 0.3 WorktreeManager

> Phase 0 — Foundation (Act 1: Setup)
>
> Objetivo: prover isolamento de filesystem por agent via Git Worktrees, com controle de concorrência seguro para refs compartilhadas, lifecycle previsível e API de uso simples para o orquestrador.

---

## 1) Verificação de coerência com `ARCHITECTURE.md`

### Diagnóstico (antes desta revisão)
- O arquivo estava **coerente na direção**, mas **incompleto em detalhamento técnico** para implementação sem dúvidas.
- Faltavam contratos explícitos de concorrência, parsing robusto de `worktree list`, critérios de aceite e política de lifecycle (stale/prune/lock reason).

### Coerência exigida pela arquitetura (Decision #2 e #11)
- 1 worktree por agent, com isolamento real de diretório.
- Operações em refs compartilhadas serializadas (mutex/semaphore).
- `simple-git` como camada primária, usando `raw()` para `git worktree`.
- Proibição operacional: não executar `git gc`/`prune` de forma agressiva durante operações ativas.

### Resultado desta revisão
- Especificação atualizada para fechar as lacunas acima.
- Etapa **pronta para implementação** (ainda não implementada neste arquivo de roadmap).

### Dependências para iniciar
- `0.1 Project Scaffolding` (obrigatória; `simple-git`, TypeScript e estrutura de pastas disponíveis).

### O que esta etapa É
- Infra de isolamento de código por agent via worktrees.
- API única para criar/listar/remover worktrees e obter `SimpleGit` isolado.
- Segurança de concorrência para refs compartilhadas no mesmo processo Node.
- Política explícita de lifecycle para detectar e limpar metadados stale com segurança.

### O que esta etapa NÃO É
- Não implementa ainda o loop completo do orquestrador (fase 2.x).
- Não implementa estratégia de merge Tier 3/4.
- Não resolve cache universal de dependências para qualquer cenário; apenas política segura e previsível.
- Não substitui locks nativos do Git por lock distribuído entre processos/máquinas.

---

## 3) Contratos e invariantes obrigatórios

1. **Mapeamento determinístico**: `agentId -> branchName -> worktreePath` é estável e reversível.
2. **Uma instância `SimpleGit` por worktree** para comandos locais (`cwd/baseDir` isolado).
3. **Toda mutação de refs compartilhadas passa por mutex** (create branch, delete branch, operações no root).
4. **`list()` sempre parseia `git worktree list --porcelain -z`** (nunca output humano).
5. **Remoção é idempotente**: repetir `remove(agentId)` não deve corromper estado.
6. **Sem falha silenciosa**: erros de lock, branch em uso, path inconsistente e estado sujo devem ser propagados com contexto.

## 4) Design de API recomendado

```ts
export interface WorktreeInfo {
  agentId: string;
  path: string;
  branch?: string;          // refs/heads/...
  head?: string;            // sha1
  detached: boolean;
  locked: boolean;
  lockReason?: string;
  prunable: boolean;
  prunableReason?: string;
  bare: boolean;
}

export interface CreateWorktreeOptions {
  lock?: boolean;           // default true
  lockReason?: string;
  nodeModulesStrategy?: 'none' | 'symlink-root' | 'copy-on-write' | 'pnpm-store';
}

export interface RemoveWorktreeOptions {
  force?: boolean;          // para WT sujo
  deleteBranch?: boolean;   // default true
  forceDeleteBranch?: boolean;
}
```

## 5) Implementação por item do checklist

### 5.1 `WorktreeManager` class wrapping simple-git `raw()` calls

**O QUE É**  
Classe de infraestrutura que encapsula comandos `git worktree` via `simple-git.raw([...])` e expõe métodos tipados de alto nível.

**O QUE NÃO É**  
Não é uma abstração completa de todos comandos Git; é focada no lifecycle de worktree e segurança de concorrência.

**COMO implementar**
- Manter duas instâncias:
  - `rootGit`: `simpleGit({ baseDir: repoRoot, maxConcurrentProcesses: N })`
  - `worktreeGitCache`: `Map<agentId, SimpleGit>`
- Todos comandos `worktree` via `raw()` em formato array (evita parsing de shell).

```ts
await this.rootGit.raw(['worktree', 'list', '--porcelain', '-z']);
await this.rootGit.raw(['worktree', 'add', '--lock', '--reason', reason, path, branch]);
```

**CRITÉRIO DE ACEITAÇÃO**
- API pública tipada compila em `strict`.
- Comandos executam sem interpolação de shell.
- Erros de git retornam com contexto (agentId/branch/path/comando).

**RISCOS (e mitigação)**
- *Risco*: misturar comandos de root e worktree em mesma instância.  
  *Mitigação*: separar claramente `rootGit` (metadados/refs) de `getGit(agentId)` (trabalho local).

### 5.2 `create(agentId, baseBranch)` — creates branch + worktree

**O QUE É**  
Cria branch de trabalho do agent e worktree correspondente, com lock opcional já na criação (`--lock`) para evitar corrida.

**O QUE NÃO É**  
Não faz merge, não executa testes, não instala dependências por padrão obrigatória (apenas estratégia configurável).

**COMO implementar**
1. Validar `agentId` e normalizar nome de branch (determinístico).
2. Resolver `worktreePath` determinístico.
3. Em seção crítica (mutex de refs):
   - garantir existência de `baseBranch`;
   - criar branch (`git branch ...`) se necessário;
   - executar `git worktree add --lock --reason ...`.
4. Aplicar estratégia de dependências (`node_modules`) conforme opção.

```ts
await this.sharedRefsMutex.runExclusive(async () => {
  await this.rootGit.raw(['branch', '--no-track', branchName, baseBranch]);
  await this.rootGit.raw([
    'worktree', 'add',
    '--lock',
    '--reason', `huu-agent:${agentId}`,
    worktreePath,
    branchName,
  ]);
});
```

**CRITÉRIO DE ACEITAÇÃO**
- Após `create`, `list()` retorna worktree com branch correta.
- Criar dois agents simultâneos não causa corrupção de refs.
- Lock reason fica visível em `worktree list --porcelain`.

**RISCOS (e mitigação)**
- *Risco*: branch já em uso em outro worktree.  
  *Mitigação*: detectar antes via `list()` e falhar com erro semântico.
- *Risco*: race entre `add` e `lock`.  
  *Mitigação*: usar `git worktree add --lock` (não `add` seguido de `lock`).

### 5.3 `remove(agentId)` — removes worktree + optionally deletes branch

**O QUE É**  
Remove worktree de forma segura e, opcionalmente, deleta branch associada.

**O QUE NÃO É**  
Não deve remover branch arbitrária sem validação; não deve mascarar erro de worktree sujo/lock.

**COMO implementar**
1. Resolver `WorktreeInfo` por `agentId`.
2. Remover worktree com `git worktree remove` (usar `--force` quando solicitado).
3. Se `deleteBranch=true`, executar exclusão da branch ainda dentro da seção crítica de refs.
4. Invalidar cache de `SimpleGit` do agent.
5. Em caso de remoção manual prévia do diretório, executar `worktree prune` controlado.

```ts
await this.sharedRefsMutex.runExclusive(async () => {
  await this.rootGit.raw(['worktree', 'remove', ...(force ? ['--force'] : []), info.path]);
  if (deleteBranch && info.branch) {
    await this.rootGit.raw(['branch', forceDeleteBranch ? '-D' : '-d', shortBranch(info.branch)]);
  }
});
```

**CRITÉRIO DE ACEITAÇÃO**
- `remove` limpa worktree e branch quando solicitado.
- Repetir `remove` no mesmo `agentId` não quebra estado (idempotência prática).
- Branch checked out em outro worktree não é removida (erro explícito preservado).

**RISCOS (e mitigação)**
- *Risco*: worktree lockada/suja bloqueia remoção.  
  *Mitigação*: respeitar flags de `force` (incluindo caso lockado que pode exigir `--force` duplo).
- *Risco*: apagar branch errada por parsing incorreto.  
  *Mitigação*: mapear branch por metadado estruturado (`refs/heads/...`) de `list()`.

### 5.4 `list()` — lists active worktrees with status

**O QUE É**  
Enumeração confiável das worktrees ativas e seus estados (detached, locked, prunable, etc).

**O QUE NÃO É**  
Não usa parsing de output formatado para humanos.

**COMO implementar**
- Executar `git worktree list --porcelain -z`.
- Parsear atributos por registro:
  - `worktree`, `HEAD`, `branch`, `detached`, `bare`, `locked`, `prunable`.
- Suportar `locked`/`prunable` com ou sem motivo.

```ts
const out = await this.rootGit.raw(['worktree', 'list', '--porcelain', '-z']);
const tokens = out.split('\0');
// token vazio separa registros
```

**CRITÉRIO DE ACEITAÇÃO**
- Paths com caracteres incomuns não quebram parser.
- `locked` e `prunable` retornam razão quando disponível.
- Saída consistente entre versões de Git (porcelain estável).

**RISCOS (e mitigação)**
- *Risco*: parser frágil por assumir ordem fixa de campos.  
  *Mitigação*: parser orientado a labels, não posição.

### 5.5 `getGit(agentId)` — returns isolated SimpleGit instance for a worktree

**O QUE É**  
Entrega uma instância `SimpleGit` isolada no diretório da worktree do agent.

**O QUE NÃO É**  
Não deve reutilizar `rootGit` para operações locais do agent.

**COMO implementar**
- Localizar `WorktreeInfo` válido por `agentId`.
- Criar (ou retornar do cache) `simpleGit({ baseDir: worktreePath, maxConcurrentProcesses })`.
- Invalidar cache ao remover worktree.

**CRITÉRIO DE ACEITAÇÃO**
- Comandos executados por `getGit(agentId)` afetam somente aquela worktree.
- Dois agents executando `status`/`add`/`commit` em paralelo não interferem entre si.

**RISCOS (e mitigação)**
- *Risco*: `cwd` apontando para path removido.  
  *Mitigação*: validar existência no `list()` antes de retornar instância cacheada.

### 5.6 Mutex/semaphore for operations on shared refs

**O QUE É**  
Coordenação in-process para serializar operações que tocam refs compartilhadas e limitar paralelismo de tarefas pesadas.

**O QUE NÃO É**  
Não substitui mecanismos internos do Git nem lock distribuído cross-process.

**COMO implementar**
- `Mutex` global para mutações de refs compartilhadas.
- `Semaphore` para throughput de operações caras (ex.: setup de dependências).
- Preferir `runExclusive`; se usar `acquire`, sempre `release` em `finally`.
- Aplicar timeout/cancelamento para evitar espera infinita.

```ts
await this.sharedRefsMutex.runExclusive(async () => {
  // create/remove branch, worktree add/remove, etc.
});

const [_, release] = await this.installSemaphore.acquire();
try {
  await this.prepareDependencies(worktreePath);
} finally {
  release();
}
```

**CRITÉRIO DE ACEITAÇÃO**
- Teste concorrente com `Promise.all` não gera corrupção de refs.
- Sem deadlock em falha de operação.
- Operações serializadas são observáveis em logs de teste.

**RISCOS (e mitigação)**
- *Risco*: deadlock por `release` não chamado.  
  *Mitigação*: regra de código obrigatória (`finally`), teste negativo e lint interno.
- *Risco*: lock in-process insuficiente se houver outro processo externo.  
  *Mitigação*: tratar erros de lock do Git como condição normal de disputa e propagar.

### 5.7 Dependency workspace strategy (`node_modules`) com fallback seguro

**O QUE É**  
Estratégia configurável para acelerar setup de worktrees sem comprometer isolamento semântico.

**O QUE NÃO É**  
Não é “um único `node_modules` para sempre” em qualquer branch; isso é frágil quando lockfiles divergem.

**COMO implementar**
- Estratégia padrão: `none` (mais segura).
- `symlink-root` somente quando lockfile hash + runtime compatíveis.
- `copy-on-write` quando filesystem suportar reflink/COW.
- `pnpm-store` recomendado quando projeto usa pnpm (hard links + symlinks gerenciados pelo pnpm).
- Sempre validar lockfile antes de compartilhar artefato.

```ts
type NodeModulesStrategy = 'none' | 'symlink-root' | 'copy-on-write' | 'pnpm-store';
// se hash divergir, não reutiliza: instala no worktree
```

**CRITÉRIO DE ACEITAÇÃO**
- Branches com lockfile igual reutilizam setup com sucesso.
- Lockfile divergente força isolamento (sem compartilhar indevidamente).
- Não há alteração de `node_modules` de uma worktree impactando outra sem intenção.

**RISCOS (e mitigação)**
- *Risco*: symlink compartilhado causar “dependency bleed” entre branches.  
  *Mitigação*: hash gate obrigatório + fallback automático para instalação isolada.
- *Risco*: inconsistência cross-filesystem.  
  *Mitigação*: detectar e degradar para `none`/instalação local.

### 5.8 Lifecycle management: stale detection + prune controlado

**O QUE É**  
Capacidade de identificar worktrees/metadados stale e limpar com segurança operacional.

**O QUE NÃO É**  
Não é GC agressivo contínuo, nem prune automático sem checagens.

**COMO implementar**
- `list()` já expõe atributo `prunable`.
- Adicionar utilitário de manutenção:
  - `detectStale(): WorktreeInfo[]` (filtra `prunable=true`)
  - `pruneStale({ dryRun = true, expire = 'now' })`
- Nunca executar prune enquanto houver operações ativas no manager.

```ts
await this.rootGit.raw(['worktree', 'prune', '--expire', expire, ...(dryRun ? ['--dry-run'] : [])]);
```

**CRITÉRIO DE ACEITAÇÃO**
- Stale detectado após remoção manual de diretório.
- `dryRun` reporta sem remover; modo real remove metadados stale.
- Sem impacto em worktrees ativas.

**RISCOS (e mitigação)**
- *Risco*: prune competir com operações em andamento.  
  *Mitigação*: guarda de “active operations” + janela de manutenção explícita.

### 5.9 Tests with real git repos (temp directories)

**O QUE É**  
Suíte de testes de integração usando repositórios Git reais em diretórios temporários.

**O QUE NÃO É**  
Não usar só mocks para fluxo principal de worktree (mocks escondem comportamento real do Git).

**COMO implementar**
- Criar helper de teste: init repo, commit base, branch principal.
- Casos mínimos:
  1. create/list/remove feliz
  2. concorrência create em múltiplos agents
  3. branch em uso não pode ser deletada
  4. parser porcelain com lock/prunable reason
  5. stale + prune dry-run/real
  6. estratégia de dependências (quando habilitada)

**CRITÉRIO DE ACEITAÇÃO**
- Testes verdes de forma determinística local e CI.
- Falhas retornam contexto suficiente para diagnóstico.

**RISCOS (e mitigação)**
- *Risco*: flakiness por concorrência/tempo.  
  *Mitigação*: usar barreiras controladas e asserts por estado observável, não por `sleep`.

## 6) Segurança de concorrência entre worktrees (padrão obrigatório)

1. **Operações em refs compartilhadas serializadas** (mutex).
2. **Uma instância `SimpleGit` por worktree** para reduzir interferência de contexto.
3. **`git worktree add --lock` com `--reason`** para evitar race entre criação e lock.
4. **Branch delete/reset respeita proteção de branch em uso por outra worktree** (não contornar erro).
5. **Parser estável (`--porcelain -z`)** para decisões de automação.
6. **Sem `gc/prune` agressivo durante atividade**; limpeza só por manutenção controlada.
7. **Sem acesso direto cego a caminhos internos de `.git`**; preferir `git rev-parse --git-path` se necessário.
