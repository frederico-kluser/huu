# 4.4 MCP Bridge

> Phase 4 — Intelligence (Act 3: Resolution)

Bridge entre servidores MCP e o runtime de tools dos agents, com lifecycle lazy-start, descoberta on-demand e estratégia token-efficient.

## Contrato de configuração (`mcp.json`)

Usar configuração em dois níveis:

1. **Global:** `~/.huu/mcp.json`
2. **Projeto:** `.huu/mcp.json` (override do global)

Precedência: **projeto > global**.

```json
{
  "settings": {
    "idleTimeoutMinutes": 10,
    "lifecycle": "lazy",
    "toolPrefix": "mcp",
    "directTools": false
  },
  "mcpServers": {
    "github": {
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      },
      "lifecycle": "lazy",
      "idleTimeoutMinutes": 10,
      "directTools": ["search_repositories", "get_file_contents"]
    },
    "context7": {
      "transport": "http",
      "url": "https://mcp.context7.com/mcp",
      "headers": {
        "CONTEXT7_API_KEY": "${CONTEXT7_API_KEY}"
      },
      "directTools": ["query-docs", "resolve-library-id"]
    }
  }
}
```

**Regras de segurança para config:**

- Nunca commitar tokens no JSON
- Resolver placeholders via env em runtime
- Validar schema de config antes de abrir conexão
- Rejeitar `command` vazio, args inválidos e `cwd` inexistente

## Checklist detalhado

### 1) MCP client programmatic setup (`@modelcontextprotocol/sdk` v1.x)

#### O QUE é

Criar a camada de cliente MCP em Node/TS usando o SDK estável v1.x (`@modelcontextprotocol/sdk`), com suporte a conexão, listagem de tools, chamada de tools, timeout e fechamento limpo.

#### O QUE NÃO é

- Não é implementação de servidor MCP
- Não é migração para SDK v2 (`@modelcontextprotocol/client` / `server`)
- Não é acoplamento direto do agent a JSON-RPC cru

#### COMO implementar

1. Criar `Client` por servidor MCP e isolar estado por `serverId`.
2. Para local server, usar `StdioClientTransport` (child process).
3. Confiar no fluxo de lifecycle do SDK: `connect()` executa `initialize`, negocia capabilities e envia `notifications/initialized`.
4. Usar helpers de alto nível (`listTools`, `callTool`) em vez de `request` cru sempre que possível.
5. Aplicar timeout por request (`RequestOptions.timeout`) e timeout total quando necessário.

```ts
import { Client } from "@modelcontextprotocol/sdk/client/index.js";
import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio.js";
import { CallToolResultSchema } from "@modelcontextprotocol/sdk/types.js";

export async function connectStdioServer(cfg: {
  command: string;
  args?: string[];
  cwd?: string;
  env?: Record<string, string>;
}) {
  const transport = new StdioClientTransport({
    command: cfg.command,
    args: cfg.args ?? [],
    cwd: cfg.cwd,
    env: cfg.env,
    stderr: "pipe"
  });

  const client = new Client(
    { name: "huu-mcp-bridge", version: "1.0.0" },
    { capabilities: {} }
  );

  await client.connect(transport);
  return { client, transport };
}
```

#### CRITÉRIO DE ACEITAÇÃO

- Conecta em pelo menos 1 server stdio local
- `listTools()` retorna tools válidas
- `callTool()` retorna conteúdo válido (ou `isError`)
- `close()` encerra sem leak de processo

#### RISCOS

- **Mismatch de versão/protocolo:** validar erro de `initialize` e sinalização explícita
- **Poluição de stdout no server:** logs devem ir para `stderr` (stdout é reservado ao protocolo)
- **Processo zumbi:** garantir `close()` em shutdown e em falha de connect

### 2) Bridge: MCP tools -> agent custom tools (correto e type-safe)

#### O QUE é

Adaptador que converte definições MCP (`ToolSchema`) para o contrato de custom tools do runtime do agent, mantendo tipagem forte no caminho de entrada/saída.

#### O QUE NÃO é

- Não é registrar todos os tools no prompt sem critério
- Não é usar `any` para simplificar parse
- Não é esconder erros de MCP com fallback silencioso

#### COMO implementar

1. Definir tipos explícitos para:
   - `McpServerConfig`
   - `McpServerRuntimeState`
   - `BridgeToolName` (`mcp/<server>/<tool>`)
   - `BridgeCallResult` (text/image/audio/resource/resource_link)
2. Implementar `listAllTools()` com paginação por `nextCursor`.
3. Ao mapear tool:
   - preservar `description` e `inputSchema`
   - normalizar nome para evitar colisão
4. Na execução:
   - `client.callTool({ name, arguments })`
   - se `isError === true`, lançar erro explícito com contexto
   - tratar `structuredContent` e `content` sem perder tipos
5. Registrar auditoria com server/tool/latência/status.

```ts
async function listAllTools(client: Client) {
  const tools = [];
  let cursor: string | undefined;

  do {
    const page = await client.listTools(cursor ? { cursor } : undefined);
    tools.push(...page.tools);
    cursor = page.nextCursor;
  } while (cursor);

  return tools;
}
```

#### CRITÉRIO DE ACEITAÇÃO

- Cada tool MCP vira uma custom tool utilizável no runtime
- Sem uso de `as any` no caminho principal de bridge
- Erros de protocolo e `isError` são diferenciados e rastreáveis
- Colisão de nomes entre servidores não quebra registro

#### RISCOS

- **Schema drift entre servidores:** validar input/output com schemas MCP
- **Colisão de nomes:** prefixar por servidor e sanitizar
- **Perda de conteúdo não-texto:** suportar todos os `ContentBlock` relevantes

### 3) Lazy-start servers (first use) + idle disconnect

#### O QUE é

Gerenciar ciclo de vida do servidor MCP com conexão na primeira chamada e desconexão após período de inatividade, sem interromper chamadas em voo.

#### O QUE NÃO é

- Não é matar processo por nome global (nada de `pkill`/`killall`)
- Não é desconectar enquanto há requisição ativa
- Não é reconectar agressivamente em loop sem backoff

#### COMO implementar

1. Modelar state machine por servidor:
   - `idle -> connecting -> ready -> closing -> idle|error`
2. Guardar:
   - `connectPromise` (single-flight para evitar corrida)
   - `inFlightCount`
   - `lastUsedAt`
   - `idleTimer`
3. Em cada `callTool`:
   - `ensureConnected()`
   - `inFlightCount++`
   - `touch()` e rearmar timer
   - `finally` decrementa contador e agenda idle close
4. `idleTimeout` default recomendado: 10 min
5. Implementar modos:
   - `lazy` (default)
   - `eager` (conecta no startup)
   - `keep-alive` (não desconecta por idle)

```ts
function scheduleIdleClose(rt: RuntimeState, idleMs: number) {
  clearTimeout(rt.idleTimer);
  rt.idleTimer = setTimeout(async () => {
    if (rt.inFlightCount > 0 || rt.state !== "ready") return;
    rt.state = "closing";
    await rt.client?.close();
    await rt.transport?.close();
    rt.state = "idle";
  }, idleMs);
}
```

#### CRITÉRIO DE ACEITAÇÃO

- Primeira chamada conecta automaticamente
- Sem chamadas, desconecta após timeout configurado
- Com chamada em voo, não desconecta
- Após desconectar, próxima chamada reconecta com sucesso

#### RISCOS

- **Race condition connect/disconnect:** usar single-flight e lock por servidor
- **Leak de timer:** limpar timers em shutdown global
- **Reconnect storm:** aplicar retry com backoff e limite

### 4) Token-efficient proxy pattern (single proxy tool, on-demand discovery)

#### O QUE é

Expor um tool único de proxy (`mcp_proxy`) para reduzir custo de contexto, com descoberta de tools por busca/descrição sob demanda e opção de promoção seletiva (`directTools`).

#### O QUE NÃO é

- Não é esconder permanentemente todos os tools (precisa haver discoverability)
- Não é registrar centenas de tools por default
- Não é substituir totalmente o bridge direto (deve coexistir com `directTools`)

#### COMO implementar

1. Criar tool única `mcp_proxy` com operações:
   - `status`, `search`, `describe`, `call`, `connect`, `disconnect`
2. Manter cache local de metadados (`tool name`, descrição, schema, server).
3. `search/describe` usa cache e não precisa conexão ativa.
4. `call` faz lazy connect e executa tool alvo.
5. Permitir `directTools` por servidor:
   - `false` (default, proxy-only)
   - `true` (expor todas)
   - `string[]` (allowlist)

```ts
type ProxyAction =
  | { action: "search"; query: string; server?: string }
  | { action: "describe"; tool: string }
  | { action: "call"; tool: string; argsJson?: string }
  | { action: "status"; server?: string }
  | { action: "connect"; server: string }
  | { action: "disconnect"; server: string };
```

#### CRITÉRIO DE ACEITAÇÃO

- Sem `directTools`, apenas `mcp_proxy` aparece no runtime de tools
- `search/describe` funciona sem conexão ativa
- `call` conecta sob demanda e executa corretamente
- Com `directTools`, apenas tools permitidas são promovidas

#### RISCOS

- **Descoberta fraca para o modelo:** melhorar ranking e descrição de busca
- **Args inválidos via `argsJson`:** parse/validação robusta com erro explícito
- **Cache stale:** invalidação por `notifications/tools/list_changed`

## Padrões transversais obrigatórios (discovery e call)

1. Sempre tratar paginação em `tools/list` (`nextCursor`).
2. Suportar `notifications/tools/list_changed` quando capability existir.
3. Separar erro de protocolo JSON-RPC de erro funcional (`isError`) de tool.
4. Definir timeout por chamada + timeout máximo.
5. Auditar todas as chamadas: `server`, `tool`, `durationMs`, `status`, `error`.

## Veredito de implementabilidade com SDK atual (v1.x)

| Requisito | Status | Evidência |
|---|---|---|
| Lazy-start | **Implementável** | `Client`/`Transport` são criados sob demanda e `connect()` é explícito |
| Idle disconnect | **Implementável** | `StdioClientTransport.close()` e `Client.close()` permitem desligamento controlado |
| Discovery incremental | **Implementável** | `tools/list` com paginação + notificação `notifications/tools/list_changed` |
| Bridge type-safe | **Implementável** | SDK expõe schemas (`ListToolsResultSchema`, `CallToolResultSchema`, etc.) e helpers tipados |
| Proxy token-efficient | **Implementável** | Padrão de proxy em camada de aplicação, sem dependência de feature nativa no SDK |

**Limitação importante:** o SDK v1.x não impõe política de idle/lazy automaticamente; isso deve ser implementado no manager do bridge.
