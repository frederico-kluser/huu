# 2.5 Context-Curator Integration

> Phase 2 — Orchestration (Act 2: Confrontation)

O "writers' assistant" do showrunner: roda após cada atividade de agente, mantém o scratchpad como memória viva e evita acumulação cega de contexto.

## Status atual e verificação arquitetural

**Status da etapa**: **não concluída** (checklist aberto).  
**Coerência com `ARCHITECTURE.md`**: a etapa está conceitualmente correta, mas estava subespecificada para implementação.

Mapeamento obrigatório com a arquitetura:

| Princípio em `ARCHITECTURE.md` | Implicação para esta etapa |
|---|---|
| Agents stateless, scratchpad stateful | Curador precisa atualizar memória persistente em SQLite após cada task |
| Curadoria > acumulação | Deve existir política explícita de keep/summarize/discard/quarantine |
| Retrieval just-in-time | Não carregar scratchpad completo; montar contexto sob demanda por tarefa/role |
| Strategic compact por checkpoint | Compactação principal deve seguir beats (Catalyst, Midpoint, All Is Lost, Break Into Three, Final Image), não apenas threshold |
| Model tiering | `context-curator` usa Haiku (tarefa mecânica, baixo custo) |

## Objetivo da etapa

Implementar uma camada determinística de curadoria de contexto que:
1. dispara automaticamente após `task_done`;
2. transforma output bruto em conhecimento útil de projeto;
3. mantém coerência de longo prazo via compactação estratégica alinhada ao beat sheet;
4. entrega contexto mínimo de alto sinal para cada agente no próximo ciclo.

## Escopo global (o que entra e o que não entra)

### O que entra nesta etapa
- Hook pós-atividade integrado ao loop do orquestrador
- Regras explícitas de entrada/saída do scratchpad
- Compactação estratégica guiada por checkpoints narrativos
- Retrieval just-in-time orientado por tarefa, role e dependências
- Persistência e rastreabilidade em SQLite (com trilha de decisão)

### O que **não** entra nesta etapa
- Interface TUI para edição manual de memória (Phase 3)
- Aprendizado de instintos/promoção avançada (Phase 4.2)
- Novo sistema de vetor externo obrigatório (deve funcionar só com SQLite)
- Reescrita do beat sheet engine (consumir checkpoints já definidos em 2.1)

## Dependencies (gates para iniciar)

- 0.2 SQLite Schema (tabelas `messages`, `entities`, `relations`, `sessions`, `beat_state`)
- 1.1 Agent Runtime (emissão consistente de `task_done` com metadados)
- 2.1 Beat Sheet Engine (checkpoint atual disponível no estado)
- 2.2 Orchestrator Loop (ponto de integração do hook pós-atividade)
- 2.3 Remaining Agents (definição operacional do `context-curator`)

## Outputs

- `src/orchestrator/curator.ts` — orchestration da curadoria pós-atividade
- `src/orchestrator/scratchpad.ts` — operações de leitura/escrita/arquivamento do scratchpad
- `src/orchestrator/retrieval-jit.ts` — montagem de contexto por role+tarefa
- `src/orchestrator/strategic-compact.ts` — compactação por checkpoint do beat sheet
- Integração no `orchestrator loop` para auto-trigger após `task_done`

## Fluxo end-to-end da tríade de contexto

1. Agente termina tarefa e publica `task_done` no SQLite.
2. Orquestrador aciona `context-curator` (idempotente por `task_id + run_id`).
3. Curador extrai delta (arquivos alterados, decisões, riscos, falhas, pendências).
4. Curador classifica itens via política de curação (keep/summarize/discard/quarantine).
5. Scratchpad é atualizado transacionalmente.
6. Se houve checkpoint do beat sheet, dispara strategic compact.
7. Próximo agente recebe **context pack JIT** com apenas o subconjunto relevante.

## Checklist detalhado

### - [ ] 1) Post-activity hook: curator runs after every agent completes

#### O QUE é
Mecanismo automático no loop do orquestrador que, ao receber `task_done`, executa curadoria de memória antes da próxima delegação.

#### O QUE NÃO é
- Não é hook opcional/manual.
- Não é processamento "best effort" silencioso.
- Não é execução sem idempotência (não pode duplicar fatos em retries).

#### COMO implementar
1. Ponto de integração: handler de conclusão no orquestrador (mesmo ponto que inicia merge/completion handling).
2. Criar chave de idempotência (`task_id:agent_id:run_id`).
3. Executar curadoria em transação SQLite:
   - carregar mensagens/fatos da tarefa recém-finalizada;
   - classificar delta;
   - aplicar updates no scratchpad;
   - registrar auditoria.
4. Se falhar, marcar erro explícito e enviar `escalation` (sem swallow).

Snippet de referência:

```ts
export async function onTaskDone(evt: TaskDoneEvent): Promise<void> {
  const key = `${evt.taskId}:${evt.agentId}:${evt.runId}`;
  if (await curatorStore.alreadyProcessed(key)) return;

  await db.transaction(async (tx) => {
    const delta = await buildTaskDelta(tx, evt);
    const decisions = await curator.classify(delta); // keep/summarize/discard/quarantine
    await scratchpad.apply(tx, decisions, evt);
    await curatorStore.markProcessed(tx, key);
  });
}
```

#### CRITÉRIO DE ACEITAÇÃO
- Cada `task_done` dispara exatamente uma curadoria efetiva (idempotência validada).
- Falhas de curadoria são visíveis no log/audit e geram fluxo de erro explícito.
- Nenhuma próxima delegação ocorre com scratchpad inconsistente parcialmente escrito.

#### RISCOS e mitigação
- **Race condition** com múltiplos `task_done` simultâneos -> usar transações + lock por `task_id`.
- **Duplicação de memória** em retry -> chave idempotente persistida.
- **Latência excessiva** -> curadoria baseada em delta (não reprocessar histórico inteiro).

### - [ ] 2) Scratchpad update logic: what changed, what to add/remove from knowledge base

#### O QUE é
Política formal para decidir, de forma reproduzível, o que entra, é resumido, sai ou é quarantinado na memória central.

#### O QUE NÃO é
- Não é "append-only log infinito".
- Não é apagar fatos contraditórios sem rastreabilidade.
- Não é guardar output bruto de tool quando já existe abstração útil.

#### COMO implementar
Adotar classificação por sinal com score explícito:

```ts
type Signal = {
  relevance: number;    // 0..1 para objetivo/beat atual
  durability: number;   // 0..1 (efêmero -> durável)
  confidence: number;   // 0..1 (evidência observável)
  actionability: number;// 0..1 (impacta próximos passos?)
  novelty: number;      // 0..1 (novo vs redundante)
  contradiction: boolean;
  impact: 'low' | 'medium' | 'high';
};

const keepScore =
  0.30 * s.relevance +
  0.20 * s.durability +
  0.20 * s.confidence +
  0.20 * s.actionability +
  0.10 * s.novelty;
```

Política de decisão (obrigatória):
- `quarantine`: `contradiction=true` e `impact=high` (não some; vai para revisão)
- `keep`: `keepScore >= 0.70` e `confidence >= 0.60`
- `summarize`: `0.45 <= keepScore < 0.70`
- `discard`: `keepScore < 0.45` e sem dependências abertas

Heurísticas de knowledge base (o que manter vs descartar):

| Tipo de informação | Política |
|---|---|
| Decisão arquitetural, contrato, invariantes | **KEEP** |
| Risco aberto, blocker, dívida ativa | **KEEP** |
| Tentativa falha com aprendizado reutilizável | **SUMMARIZE** |
| Log bruto repetitivo de tool, output transitório | **DISCARD** |
| Fato contraditório de alto impacto | **QUARANTINE** |
| Fato antigo supersedido | Arquivar com `superseded_by`, não apagar hard |

Persistência recomendada (exemplo SQL com rastreabilidade):

```sql
-- upsert de entidade curada
INSERT INTO entities (id, type, value_json, confidence, updated_at)
VALUES (:id, :type, :json, :confidence, CURRENT_TIMESTAMP)
ON CONFLICT(id) DO UPDATE SET
  value_json = excluded.value_json,
  confidence = excluded.confidence,
  updated_at = CURRENT_TIMESTAMP;

-- vínculo de contradição/supersessão
INSERT INTO relations (from_id, to_id, relation_type)
VALUES (:new_id, :old_id, 'supersedes');
```

#### CRITÉRIO DE ACEITAÇÃO
- Existe função determinística de classificação documentada e testada.
- Para qualquer item curado, o sistema consegue explicar por que entrou/saiu.
- Contradições de alto impacto nunca são descartadas silenciosamente.
- Scratchpad mantém alto sinal (sem crescimento linear de ruído por tarefa).

#### RISCOS e mitigação
- **Overfitting de score** -> calibrar com traces reais e testes de regressão.
- **Perda de contexto sutil** -> rota `summarize` antes de `discard` em casos limítrofes.
- **Acúmulo de lixo** -> política de arquivamento e decay por tipo de entidade.

### - [ ] 3) Strategic compact at beat sheet checkpoints

#### O QUE é
Compactação deliberada da memória em marcos narrativos do beat sheet para manter coerência de longo horizonte.

#### O QUE NÃO é
- Não é compactação apenas por pressão de token.
- Não é sumarização genérica sem vínculo ao estado do beat.
- Não é apagar evidências auditáveis.

#### COMO implementar
Disparar compactação estratégica somente em checkpoints de 2.1:
- Catalyst (~10%)
- Midpoint (~50%)
- All Is Lost (~75%)
- Break Into Three (~77%)
- Final Image (100%)

Matriz de compactação por checkpoint:

| Checkpoint | Foco da compactação | Saída mínima obrigatória |
|---|---|---|
| Catalyst | Clarificar objetivo e restrições | objetivo operacional + riscos iniciais |
| Midpoint | Replanejar com base no progresso | decisões confirmadas + gaps restantes |
| All Is Lost | Tornar explícito o maior risco | risco raiz + plano de contenção |
| Break Into Three | Consolidar nova estratégia | mudança de abordagem + novos critérios |
| Final Image | Encerrar e preparar handoff | resumo final + lições + pendências residuais |

Pipeline:
1. Snapshot do estado ativo do scratchpad.
2. Compact prompt estruturado (decisions, blockers, open tasks, evidence, next actions).
3. Geração de sumário compacto de alta fidelidade.
4. Marcação de itens supersedidos/arquivados (sem hard delete).
5. Registro em `sessions`/`beat_state` com `checkpoint_id`.

Fallback operacional (não estratégico):
- Se contexto atingir limite crítico, permitir compactação emergencial com flag `trigger=window_pressure`.
- Essa compactação **não substitui** a compactação estratégica por checkpoint.

#### CRITÉRIO DE ACEITAÇÃO
- Toda transição de checkpoint gera snapshot compacto versionado.
- Sumário compacto preserva decisões arquiteturais, bugs abertos e próximos passos.
- É possível reconstruir "como chegamos aqui" via trilha de snapshots + relações.

#### RISCOS e mitigação
- **Compactação agressiva demais** -> priorizar recall primeiro, precisão depois.
- **Resumo desalinhado do beat atual** -> prompt de compactação recebe `beat_state` explícito.
- **Custo extra** -> usar Haiku e compactar somente em checkpoints (não em todo turno).

### - [ ] 4) Retrieval just-in-time: load relevant context per agent, not everything

#### O QUE é
Montagem de contexto sob demanda para cada agente, baseada em tarefa, role e dependências, evitando dump completo do scratchpad.

#### O QUE NÃO é
- Não é enviar histórico total para todos os agentes.
- Não é retrieval one-shot fixo sem iteração.
- Não é dependência obrigatória de índice vetorial externo para funcionar.

#### COMO implementar
Pipeline de retrieval por role/tarefa:
1. Construir consulta a partir de `task`, arquivos-alvo, dependências e checkpoint atual.
2. Buscar candidatos em `entities`, `relations`, `messages` recentes.
3. Expandir 1-hop por relações (`depends_on`, `supersedes`, `affects_file`).
4. Ranquear por relevância, confiança, recência e custo de recuperação.
5. Cortar por orçamento de tokens da role.
6. Entregar `context pack` estruturado.

Exemplo de contrato do `context pack`:

```ts
type ContextPack = {
  objective: string;
  currentBeat: string;
  decisions: CuratedItem[];
  risks: CuratedItem[];
  fileFacts: CuratedItem[];
  openQuestions: CuratedItem[];
  references: { kind: 'file' | 'url' | 'query'; pointer: string }[];
};
```

Padrão iterativo recomendado (max 3 ciclos):
- ciclo 1: contexto mínimo;
- ciclo 2: carregar evidências faltantes após avaliação do agente;
- ciclo 3: fechamento com evidências específicas (se necessário).

Integração com Context7 (quando houver dependência de docs externas):
- Resolver biblioteca (`resolve-library-id`) e buscar docs (`get-library-docs`) sob demanda;
- Persistir no scratchpad apenas resumo/decisão/citação relevante, não o dump completo;
- Guardar ponteiros (`library_id`, versão, URL) para reconsulta futura.

#### CRITÉRIO DE ACEITAÇÃO
- Cada agente recebe contexto diferente conforme role+tarefa.
- Queda mensurável de tokens por turno sem perda de qualidade percebida.
- Retrieval recupera fatos críticos (decisões + riscos + pendências) com alta consistência.

#### RISCOS e mitigação
- **Under-retrieval** (faltou contexto crítico) -> loop iterativo com autoavaliação de lacunas.
- **Over-retrieval** (bloat) -> budgets por role e ranking por sinal.
- **Dados externos desatualizados** -> preferir ponteiros versionados e reconsulta JIT.

## Critérios de aceite da etapa 2.5 (Definition of Done)

- [ ] Hook pós-atividade executa de forma idempotente para todo `task_done`
- [ ] Política keep/summarize/discard/quarantine está implementada, testada e auditável
- [ ] Strategic compact ocorre nos checkpoints do beat sheet com snapshots versionados
- [ ] Retrieval JIT entrega context pack mínimo por role/tarefa
- [ ] Contradições de alto impacto entram em quarantine (não são descartadas)
- [ ] Logs e auditoria permitem rastrear qualquer decisão de curadoria

## Testes mínimos obrigatórios

1. **Unitário — classificação de curação**  
   Dado um conjunto de sinais, decisão final é determinística.
2. **Unitário — idempotência do hook**  
   Mesmo `run_id` processado 2x não duplica memória.
3. **Integração — checkpoint compact**  
   Avanço de beat dispara compactação e grava snapshot.
4. **Integração — retrieval JIT por role**  
   `builder` e `reviewer` recebem pacotes diferentes e focados.
5. **Regressão — contradição crítica**  
   Fato conflitante de alto impacto é quarantinado.

## Notas técnicas finais

- `context-curator` deve usar **Haiku** por custo/latência.
- A memória real é o SQLite; agentes continuam stateless workers.
- Curadoria deve privilegiar **sinal alto e rastreabilidade**, não volume.
- Simplicidade primeiro: começar com regras explícitas e calibrar com dados reais.
