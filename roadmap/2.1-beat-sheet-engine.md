## Modelo de dados (4 niveis obrigatorios)

## O QUE e

Estrutura hierarquica deterministica com quatro niveis:

1. **Objetivo (root)**: intencao global + criterio de sucesso.
2. **Ato**: macro-fase narrativa (`setup`, `confrontation`, `resolution`).
3. **Sequencia**: agrupamento de tarefas correlatas por subobjetivo.
4. **Tarefa atomica**: menor unidade executavel com mudanca de estado verificavel.

## O QUE NAO e

- Nao e backlog infinito de ideias.
- Nao e plano apenas textual sem IDs, status e dependencias.
- Nao e sistema de execucao (execucao com agentes fica principalmente em 2.2).

## COMO implementar

```ts
export type BeatStatus =
  | 'pending'
  | 'ready'
  | 'running'
  | 'blocked'
  | 'done'
  | 'failed';

export type ActType = 'setup' | 'confrontation' | 'resolution';

export interface CheckpointState {
  catalyst: 'pending' | 'passed' | 'failed';
  midpoint: 'pending' | 'passed' | 'failed';
  allIsLost: 'pending' | 'passed' | 'failed';
  breakIntoThree: 'pending' | 'passed' | 'failed';
  finalImage: 'pending' | 'passed' | 'failed';
}

export interface AtomicTask {
  id: string;
  actId: string;
  sequenceId: string;
  title: string;
  precondition: string;
  action: string;
  postcondition: string;
  verification: string;
  dependencies: string[]; // task ids
  critical: boolean;
  estimatedEffort: 'small' | 'medium' | 'large';
  status: BeatStatus;
}

export interface SequenceNode {
  id: string;
  actId: string;
  name: string;
  objective: string;
  tasks: AtomicTask[];
}

export interface ActNode {
  id: string;
  type: ActType;
  name: string;
  objective: string;
  sequences: SequenceNode[];
}

export interface BeatSheet {
  id: string;
  objective: string;
  successCriteria: string[];
  constraints: string[];
  acts: ActNode[];
  checkpoints: CheckpointState;
  version: number;
  createdAt: string;
  updatedAt: string;
}
```

**Invariantes obrigatorios**

- Toda tarefa atomica deve ter `precondition`, `action`, `postcondition` e `verification`.
- `dependencies` so podem apontar para tarefas do mesmo beat sheet.
- Grafo de tarefas atomicas deve ser aciclico.
- Toda sequencia deve possuir ao menos 1 tarefa atomica.
- Todo ato deve possuir ao menos 1 sequencia.

## CRITERIO DE ACEITACAO

- Validacao estrutural rejeita beat sheets com niveis ausentes.
- Validacao semantica rejeita tarefa sem mudanca de estado verificavel.
- IDs unicos em toda a arvore.

## RISCOS

- **Risco:** granularidade inconsistente entre sequencias/tarefas.  
  **Mitigacao:** regra de tamanho (`small/medium/large`) + verificador de tarefa atomica.
- **Risco:** dependencias cruzadas incorretas.  
  **Mitigacao:** validador de referencias e ciclo antes de persistir.

### 2) Decomposition prompt for planner agent

#### O QUE e

Prompt de planejamento em estilo **Plan-and-Execute + HTN**, com saida estruturada para gerar `BeatSheet`.

#### O QUE NAO e

- Nao e prompt de execucao de tarefas.
- Nao e prompt de review/QA.

#### COMO implementar

Boas praticas usadas:
- Anthropic: instrucoes claras, XML tags, formato deterministico.
- Plan-and-Execute: separar planejamento da execucao.
- HTN: decomposicao top-down com refinamento recursivo ate tarefa atomica.

Template recomendado:

```xml
<planner_task>
  <role>Voce e o planner do showrunner.</role>
  <objective>{{USER_OBJECTIVE}}</objective>
  <constraints>{{PROJECT_CONSTRAINTS}}</constraints>
  <instructions>
    1) Decomponha em 3 atos: setup, confrontation, resolution.
    2) Para cada ato, crie sequencias.
    3) Para cada sequencia, crie tarefas atomicas com:
       precondition, action, postcondition, verification.
    4) Crie dependencies entre tarefas atomicas em formato DAG.
    5) Nao gere tarefas sem mudanca de estado verificavel.
  </instructions>
  <output_format>
    Retorne APENAS JSON valido no schema BeatSheet.
  </output_format>
</planner_task>
```

#### CRITERIO DE ACEITACAO

- Planner retorna JSON parseavel em >= 95% das execucoes de teste.
- Todas as tarefas possuem os 4 campos fractais obrigatorios.
- Dependencias podem ser carregadas pelo validador DAG sem ciclos.

#### RISCOS

- Saida textual fora do schema JSON.  
  Mitigar com prompt de formato estrito + parser defensivo + retry limitado.

### 3) Checkpoint definitions

#### O QUE e

Definir checkpoints com criterios verificaveis para transicao de estado.

#### O QUE NAO e

- Nao e checkpoint subjetivo ("parece bom").
- Nao e milestone apenas temporal sem evidencia.

#### COMO implementar

Criar `evaluateCheckpoint(sheet, telemetry)` com regras abaixo:

| Checkpoint | Gatilho | Criterios de aprovacao |
|---|---|---|
| **Catalyst (~10%)** | Planejamento inicial concluido | Objetivo claro, constraints listadas, >=1 ato e >=1 sequencia por ato, backlog inicial de tarefas atomicas valido |
| **Midpoint (~50%)** | Metade do trabalho planejado atingida | >=50% tarefas `done`; nenhum bloqueio critico sem owner; risco principal atualizado |
| **All Is Lost (~75%)** | Aproximacao de conclusao com risco maximo exposto | Maior risco identificado e documentado; plano de contingencia definido; tarefas de mitigacao criadas |
| **Break Into Three (~77%)** | Saida de crise/replanejamento | DAG revisado (version +1), estrategia ajustada, dependencias recalculadas sem ciclo |
| **Final Image (100%)** | Encerramento | 100% tarefas concluidas ou justificadas como descartadas; checkpoints anteriores `passed`; sucesso validado contra requisitos originais |

#### CRITERIO DE ACEITACAO

- Cada checkpoint produz `passed/failed` + lista de evidencias.
- Falha em criterio bloqueia avancar checkpoint seguinte.

#### RISCOS

- Threshold rigido demais para tarefas pequenas.  
  Mitigar com tolerancias configuraveis por projeto.

### 4) Dependency graph between atomic tasks

#### O QUE e

DAG de tarefas atomicas para determinar ordem valida e paralelismo seguro.

#### O QUE NAO e

- Nao e grafo entre atos/sequencias (o grafo operacional e em tarefas atomicas).
- Nao e scheduler distribuido completo (isso evolui no 2.2).

#### COMO implementar

**Algoritmos recomendados**
- **Kahn (topsort)** para ordenar e detectar ciclo em `O(V+E)`.
- **Tarjan / SCC** para diagnostico detalhado em caso de ciclo.
- **Fila de prioridade** para desempate deterministico (`critical desc`, `estimatedEffort asc`, `id asc`).

```ts
export function computeReadySet(
  tasks: AtomicTask[],
  doneTaskIds: Set<string>
): AtomicTask[] {
  return tasks.filter((t) =>
    t.status === 'pending' &&
    t.dependencies.every((dep) => doneTaskIds.has(dep))
  );
}
```

**Implementacao em TypeScript**
- Opcao simples: implementacao propria de adjacency list.
- Opcao de biblioteca: `@dagrejs/graphlib` (`alg.isAcyclic`, `alg.topsort`).

#### CRITERIO DE ACEITACAO

- Ciclo retorna erro explicito com trilha de IDs envolvidos.
- Para DAG valido, retorna ordem topologica e "waves" paralelas.
- Ordem e deterministica para mesma entrada.

#### RISCOS

- Deadlock logico por dependencia ausente/invalida.  
  Mitigar com validacao de referencias antes da execucao.

### 5) Beat state persistence in SQLite

#### O QUE e

Persistir estado do beat sheet, checkpoint e progresso para retomada e auditoria.

#### O QUE NAO e

- Nao e event sourcing completo.
- Nao e historico detalhado de cada tool call (isso e do `audit_log`).

#### COMO implementar

Usar `beat_state` (previsto em 0.2) como fonte oficial:

```sql
-- Estrutura sugerida para 2.1 (ajustar a migration real de 0.2)
CREATE TABLE IF NOT EXISTS beat_state (
  id TEXT PRIMARY KEY,
  objective TEXT NOT NULL,
  version INTEGER NOT NULL DEFAULT 1,
  status TEXT NOT NULL,
  current_act_id TEXT,
  current_sequence_id TEXT,
  checkpoints_json TEXT NOT NULL,
  sheet_json TEXT NOT NULL,
  updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

Regras de persistencia:
1. Sempre salvar em transacao (`BEGIN IMMEDIATE ... COMMIT`).
2. Incrementar `version` a cada replanejamento estrutural.
3. Nunca sobrescrever `sheet_json` sem atualizar `updated_at`.

#### CRITERIO DE ACEITACAO

- Estado pode ser salvo e restaurado sem perda de IDs/dependencias.
- Replanejamento incrementa versao e preserva tarefas concluidas.

#### RISCOS

- Corrupcao logica por escrita concorrente.  
  Mitigar com transacao + lock de escrita no orchestrator.

#### O QUE e

Renderizacao textual hierarquica para CLI, com foco em progresso e bloqueios.

- Mesmo beat sheet sempre gera mesma ordem de exibicao.

#### RISCOS

- Ruido visual em projetos grandes.  
  Mitigar com modo resumido e paginacao por ato/sequencia.

## Recomendacoes de melhores praticas incorporadas

1. **Plan-and-Execute / HTN:** separar fase de plano e fase de execucao; replanejar apenas quando necessario.
2. **DAG em TypeScript:** priorizar `O(V+E)` para ordenacao/deteccao de ciclo; manter scheduler deterministico.
3. **Dependency resolution:** combinar Kahn (ordem) + SCC/Tarjan (diagnostico) + prioridade para desempate.
4. **Multi-agent hierarchical planning:** planner central, workers especializados, contexto minimamente necessario por tarefa.
5. **Prompt engineering para Claude:** instrucoes claras, XML tags, formato de saida estrito, exemplos canonicos e poucos.
