# 5.2 Resilience

> Phase 5 — Polish (Final Image)

Resiliência operacional do orquestrador: recuperação após crash, limpeza segura de worktrees/processos stale, controle de timeout/retry de agents e shutdown gracioso com preservação de progresso.

---

## Diagnóstico de completude e coerência (auditoria desta etapa)

**Status atual da etapa 5.2 antes desta revisão: incompleta.**

Principais lacunas identificadas:

1. O item de crash recovery era genérico e não definia **como retomar do ponto exato** (granularidade de checkpoint, idempotência, replay determinístico).
2. Graceful shutdown não definia ordem de drenagem, nem garantia explícita de **preservação de trabalho já concluído**.
3. Timeout/retry não tinha política técnica (tipos de timeout, backoff, jitter, limite de tentativas, erros retryable).
4. Stale cleanup não tinha critérios seguros para não apagar trabalho local não mergeado.
5. Faltava o padrão explícito de **state machine recovery** para manter o orquestrador reentrante e determinístico.

Coerência com `ARCHITECTURE.md`:

- Alinhado com **SQLite WAL unificado** (Decisão 3, 7 e 12).
- Alinhado com **worktrees isoladas** + merge sequencial no `main` (Decisão 2 e 4).
- Alinhado com **agents stateless e scratchpad stateful** (Decisão 9).
- Alinhado com runtime **Node.js + TypeScript + better-sqlite3 + simple-git** (Decisão 11).

---

## Objetivo da etapa 5.2

Implementar uma camada de resiliência que garanta:

- retomada determinística após falha de processo/máquina;
- preservação de progresso já persistido;
- recuperação operacional sem intervenção manual na maioria dos cenários;
- limites seguros para retries e execução concorrente;
- desligamento previsível em SIGINT/SIGTERM.

## 5.2.1 Crash recovery: resume from last SQLite checkpoint

### O QUE é

Sistema de recuperação que reconstrói o estado do orquestrador usando dados duráveis no SQLite (WAL), retomando execução no **último ponto consistente** sem repetir efeitos colaterais já confirmados.

### O QUE NÃO é

- Não é snapshot “best effort” sem semântica de consistência.
- Não é “reiniciar tudo do zero”.
- Não é reexecutar merges/ações externas cegamente sem deduplicação.
- Não é depender de estado em memória para decisões de recuperação.

### COMO implementar

1. **Persistência transacional do progresso**
   - Cada transição de tarefa (ex.: `assigned -> running -> merge_ready -> done`) deve ser gravada em transação atômica.
   - Persistir `run_id`, `task_id`, `attempt`, `state`, `updated_at`, `idempotency_key`.

2. **Checkpoint + event log (modelo híbrido)**
   - Manter snapshot materializado do estado atual (rápido para carregar).
   - Manter log append-only de eventos (fonte de verdade para replay e auditoria).

3. **High-watermark de processamento**
   - Salvar `last_applied_event_id` por `run_id`.
   - No boot: carregar snapshot + replay somente eventos `id > last_applied_event_id`.

4. **Idempotência obrigatória**
   - Toda operação com efeito externo (merge, commit, dispatch) recebe `idempotency_key`.
   - Se a chave já foi confirmada, não repetir efeito.

5. **Política WAL/checkpoint**
   - WAL ativo por padrão.
   - Checkpoint periódico (`wal_checkpoint(PASSIVE|FULL)` conforme carga) e checkpoint final no shutdown.
   - `synchronous=FULL` no caminho crítico para durabilidade.

6. **Integridade na inicialização**
   - Startup roda validação rápida (`PRAGMA quick_check`) e falha de forma explícita se banco inconsistente.

Exemplo de esquema mínimo (referência):

```sql
CREATE TABLE IF NOT EXISTS orchestrator_runs (
  run_id TEXT PRIMARY KEY,
  status TEXT NOT NULL, -- running | draining | recovered | done | failed
  state_version INTEGER NOT NULL DEFAULT 0,
  last_applied_event_id INTEGER NOT NULL DEFAULT 0,
  shutdown_requested_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS orchestrator_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  run_id TEXT NOT NULL,
  task_id TEXT,
  event_type TEXT NOT NULL,
  payload_json TEXT NOT NULL,
  idempotency_key TEXT,
  created_at TEXT NOT NULL,
  UNIQUE(run_id, idempotency_key)
);

CREATE TABLE IF NOT EXISTS task_attempts (
  run_id TEXT NOT NULL,
  task_id TEXT NOT NULL,
  attempt INTEGER NOT NULL,
  state TEXT NOT NULL, -- assigned|running|done|failed|timeout
  heartbeat_at TEXT,
  updated_at TEXT NOT NULL,
  PRIMARY KEY (run_id, task_id, attempt)
);
```

Exemplo de ponto de persistência transacional:

```ts
db.transaction((input: {
  runId: string;
  taskId: string;
  attempt: number;
  nextState: string;
  eventType: string;
  payloadJson: string;
  idempotencyKey: string;
}) => {
  updateTaskAttempt.run(input);
  insertEvent.run(input);
  bumpRunVersion.run({ runId: input.runId });
})(input);
```

### CRITÉRIO DE ACEITAÇÃO

- Teste forçando `SIGKILL` durante execução: após restart, o run retorna ao mesmo `task_id + attempt + state` esperado.
- Nenhum evento confirmado é perdido após crash.
- Nenhum efeito externo idempotente é duplicado após recuperação.
- Tempo de recuperação observável e logado (`recovery_duration_ms`).

### RISCOS (e mitigação)

- **WAL crescer demais** → checkpoint periódico + alerta por tamanho.
- **Duplicação de side effects** → `idempotency_key` + UNIQUE.
- **Estado divergente entre tabelas** → transações únicas para mudança de estado + evento.
- **Recuperação parcial silenciosa** → falhar explicitamente quando integridade não passa.

### Verificação explícita: retomada no ponto exato

Para considerar “retoma no ponto exato”, os quatro invariantes abaixo devem ser verdadeiros:

1. Mesmo `run_id` e `state_version` monotônico.
2. Mesmo `task_id` e `attempt` ativo antes do crash (ou promovido para retry se lease expirada).
3. Mesmo `last_applied_event_id` (ou maior apenas por eventos de recovery).
4. Nenhum `idempotency_key` confirmado executado duas vezes.

## 5.2.2 Stale process/worktree detection and cleanup

### O QUE é

Mecanismo de detecção de workers órfãos (sem heartbeat/PID inválido) e remoção segura de worktrees abandonadas, preservando mudanças não integradas.

### O QUE NÃO é

- Não é apagar worktree apenas por idade.
- Não é remover branch com commits não mergeados sem confirmação.
- Não é rodar `git gc`/`git prune` indiscriminadamente com worktrees ativas.

### COMO implementar

1. **Heartbeat + lease**
   - Cada agent atualiza heartbeat (ex.: a cada 5s).
   - Considerar stale se `now - heartbeat_at > stale_threshold_ms` e PID não existe.

2. **Validação de segurança antes de remover worktree**
   - `git status --porcelain` vazio.
   - Branch já mergeada (ou explicitamente marcada descartável).
   - Worktree não está lockada.
   - Sem processo ativo associado.

3. **Fluxo de cleanup em duas fases**
   - Fase A (quarentena): marcar `cleanup_candidate`.
   - Fase B (remoção): apenas após revalidação.

4. **Operações Git recomendadas**
   - `git worktree list --porcelain`
   - `git worktree remove <path>`
   - `git worktree prune` para metadados órfãos
   - lock opcional para worktrees long-lived (`git worktree lock`)

Exemplo de interface:

```ts
type CleanupDecision =
  | { action: 'skip'; reason: string }
  | { action: 'quarantine'; reason: string }
  | { action: 'remove'; reason: string };

function decideCleanup(input: {
  hasLivePid: boolean;
  hasUncommittedChanges: boolean;
  hasUnmergedCommits: boolean;
  isLocked: boolean;
  staleMs: number;
}): CleanupDecision {
  if (input.hasLivePid || input.isLocked) return { action: 'skip', reason: 'active_or_locked' };
  if (input.hasUncommittedChanges || input.hasUnmergedCommits) {
    return { action: 'quarantine', reason: 'preserve_local_work' };
  }
  return { action: 'remove', reason: 'safe_to_remove' };
}
```

### CRITÉRIO DE ACEITAÇÃO

- Worktree stale limpa sem tocar em branches/worktrees ativas.
- Worktree com mudanças locais não mergeadas **não é removida automaticamente**.
- `cleanup_report` registra decisões (`skip/quarantine/remove`) com motivo.

### RISCOS (e mitigação)

- **Perda de trabalho local** → quarentena obrigatória para dirty/unmerged.
- **Falso positivo de stale** → validar heartbeat + PID + lock status.
- **Corrida com merge em andamento** → mutex de merge + janela de exclusão para cleanup.

## 5.2.3 Agent timeout with automatic retry

### O QUE é

Política de timeout por fase (spawn, execução, merge handoff) com retries controlados para falhas transitórias, evitando loops infinitos e tempestade de retries.

### O QUE NÃO é

- Não é retry ilimitado.
- Não é retry para erro permanente (ex.: prompt inválido, permissão negada, bug determinístico).
- Não é timeout único global sem granularidade por etapa.

### COMO implementar

1. **Dois níveis de timeout**
   - `soft_timeout_ms`: sinaliza degradação e solicita heartbeat imediato.
   - `hard_timeout_ms`: aborta tentativa, marca `timeout`, avalia retry.

2. **Classificação de erro**
   - Retryable: timeout, 429, 5xx transitório, lock temporário.
   - Não retryable: 4xx permanente, falha de validação, erro lógico determinístico repetido.

3. **Retry com backoff exponencial + full jitter**
   - Fórmula: `delay = random(0, min(cap, base * 2^attempt))`.
   - `max_attempts` baixo (tipicamente 3).
   - Budget total por tarefa (`max_retry_window_ms`).

4. **Idempotência por tentativa**
   - `attempt_id` + `idempotency_key` para evitar duplicar efeitos de tentativa anterior.

Exemplo:

```ts
function nextDelayMs(attempt: number, baseMs: number, capMs: number): number {
  const max = Math.min(capMs, baseMs * 2 ** attempt);
  return Math.floor(Math.random() * (max + 1)); // full jitter
}
```

### CRITÉRIO DE ACEITAÇÃO

- Falhas transitórias recuperam sem intervenção humana até `max_attempts`.
- Erro permanente não entra em loop de retry.
- Logs mostram `attempt`, `reason`, `delay_ms`, `retryable`.

### RISCOS (e mitigação)

- **Retry storm** → jitter + limite de concorrência + max attempts.
- **Esconder bug real com retries** → classificador de erro + escalonamento após limite.
- **Timeout agressivo demais** → calibrar por tipo de tarefa e baseline real.

## 5.2.4 Graceful shutdown (SIGINT, SIGTERM)

### O QUE é

Procedimento de encerramento controlado para sinais de término: parar novas admissões, drenar tarefas em execução, consolidar resultados já concluídos e sair com estado consistente.

### O QUE NÃO é

- Não é `process.exit()` imediato ao receber sinal.
- Não é depender de `process.on('exit')` para cleanup assíncrono.
- Não é abandonar merge queue ou progresso já confirmado.

### COMO implementar

1. **Signal handlers one-shot**
   - Registrar `SIGINT` e `SIGTERM` para a mesma rotina.
   - Segunda interrupção força saída (fail-safe).

2. **Fases de shutdown**
   - `QUIESCE`: `accept_new_tasks = false`.
   - `DRAIN`: aguardar agents em progresso até `grace_period_ms`.
   - `FLUSH`: persistir checkpoint final + enfileirar merges prontos.
   - `CLEANUP`: liberar recursos (DB pools, watchers, timers).
   - `EXIT`: código 0 (graceful) ou 1 (forçado).

3. **Integração com merge**
   - Se tarefa já está `merge_ready`, processar merge antes de sair (até limite de tempo).
   - Se não couber no budget de shutdown, persistir como pendente para recovery.

Exemplo mínimo:

```ts
let shuttingDown = false;

async function shutdown(signal: NodeJS.Signals): Promise<void> {
  if (shuttingDown) {
    process.exit(1); // segunda interrupção
  }
  shuttingDown = true;
  await orchestrator.requestShutdown(signal); // quiesce
  await orchestrator.drainInFlight();         // drain
  await orchestrator.flushCheckpoint();       // flush
  await orchestrator.closeResources();        // cleanup
  process.exit(0);
}

process.on('SIGINT', () => void shutdown('SIGINT'));
process.on('SIGTERM', () => void shutdown('SIGTERM'));
```

### CRITÉRIO DE ACEITAÇÃO

- Em SIGINT/SIGTERM, nenhuma tarefa nova inicia após `QUIESCE`.
- Tarefas já concluídas antes do sinal permanecem visíveis como concluídas após restart.
- Merge queue não perde itens `merge_ready` já persistidos.
- Segunda interrupção força encerramento com log explícito de saída não graciosa.

### RISCOS (e mitigação)

- **Shutdown infinito** → `grace_period_ms` + saída forçada controlada.
- **Perda de progresso em memória** → persistir antes de qualquer `process.exit`.
- **Deadlock em drain** → timeout por agent + cancelamento cooperativo.

### Verificação explícita: preserva todo trabalho concluído?

Só considerar “preserva trabalho concluído” quando:

1. Todo item `task_done` persistido antes do sinal reaparece como `done` após reboot.
2. Todo item `merge_ready` persistido antes do sinal reaparece na merge queue.
3. Nenhum item `done` volta para `running` sem novo `attempt`.

## 5.2.5 State machine recovery patterns para o orquestrador

### O QUE é

Formalização da máquina de estados do run para permitir recuperação determinística, evitando transições inválidas durante retries, shutdown e reboot.

### O QUE NÃO é

- Não é estado implícito espalhado em flags soltas.
- Não é permitir qualquer transição em qualquer ordem.
- Não é tratar recovery como fluxo separado do runtime normal.

### COMO implementar

1. **Estados explícitos do run**
   - `running`, `draining`, `recovering`, `paused_for_human`, `done`, `failed`.

2. **Transições válidas**
   - `running -> draining` (sinal de shutdown)
   - `running -> recovering` (startup pós-crash com pendências)
   - `recovering -> running` (reconciliação concluída)
   - `* -> failed` (erro não recuperável)
   - `running|draining -> done` (todas tarefas finalizadas)

3. **Guardas de transição**
   - Transição só acontece se pré-condições forem verdadeiras (ex.: não sair de `draining` sem flush).
   - Persistir transição + evento na mesma transação.

Exemplo:

```ts
const allowed: Record<string, Set<string>> = {
  running: new Set(['draining', 'recovering', 'paused_for_human', 'done', 'failed']),
  recovering: new Set(['running', 'paused_for_human', 'failed']),
  draining: new Set(['done', 'failed']),
  paused_for_human: new Set(['running', 'failed']),
  done: new Set([]),
  failed: new Set([])
};
```

### CRITÉRIO DE ACEITAÇÃO

- Nenhuma transição inválida passa sem erro explícito.
- Após crash no meio de transição, replay reconstrói estado final consistente.
- Recovery reutiliza a mesma máquina de estados (sem caminho paralelo ad hoc).

### RISCOS (e mitigação)

- **Complexidade excessiva** → manter poucos estados com semântica clara.
- **Estado zumbi** → timeout de estado + watchdog de reconciliação.
- **Divergência entre código e banco** → testes de transição com tabela de verdade.
