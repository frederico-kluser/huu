### Práticas técnicas importantes (resumo)

1. **TypeScript com Node 22+:** Node já suporta type-stripping, mas para suporte completo de TS (tsconfig, .tsx, ergonomia de dev), manter `tsx` como runtime principal de desenvolvimento.
2. **ESM-first:** usar `"type": "module"` e `module/moduleResolution: "NodeNext"`.
3. **Ink CLI:** separar entrada CLI, árvore TUI e camadas de infraestrutura/domínio.
4. **Vitest:** usar `environment: "node"` e cobertura com provider `v8`.
5. **better-sqlite3:** inicializar com `journal_mode = WAL`, `foreign_keys = ON`, statements preparados e transações explícitas.
6. **simple-git:** uma instância por diretório/worktree; usar `.raw()` para `git worktree`.

## Checklist detalhado (expandido para implementação)

### 1) `npm init` + TypeScript config (tsx, strict mode)

### O QUE é
Inicializar o projeto Node.js/TypeScript com ESM, configuração estrita de tipos e execução de scripts TS com `tsx`.

### O QUE NÃO é
- Não é implementação de regras de negócio.
- Não é setup de bundler para produção (ex.: esbuild/tsup) nesta etapa.
- Não é migração para monorepo.

### COMO implementar
1. Inicializar projeto:
   ```bash
   npm init -y
   npm pkg set private=true type=module
   npm pkg set engines.node=">=22"
   ```
2. Instalar base TS runtime/dev:
   ```bash
   npm i -D typescript tsx @types/node
   ```
3. Criar `tsconfig.json` estrito (base recomendada):
   ```json
   {
     "compilerOptions": {
       "target": "ES2023",
       "module": "NodeNext",
       "moduleResolution": "NodeNext",
       "strict": true,
       "noUncheckedIndexedAccess": true,
       "exactOptionalPropertyTypes": true,
       "verbatimModuleSyntax": true,
       "resolveJsonModule": true,
       "skipLibCheck": true,
       "rootDir": "src",
       "outDir": "dist"
     },
     "include": ["src", "test", "vitest.config.ts"]
   }
   ```
4. Padrão de imports para ESM Node: usar extensão `.js` nos imports relativos em código TS (ex.: `import { x } from './x.js'`).

### CRITÉRIO DE ACEITAÇÃO
- `npm run typecheck` executa sem erros com arquivos stub mínimos.
- Projeto está em ESM (`"type": "module"`).
- `tsx` executa um entrypoint TS sem transpilar manualmente.

### RISCOS
- **Risco:** quebra por resolução ESM/CJS.
  **Mitigação:** padronizar `NodeNext` + imports relativos com `.js`.
- **Risco:** confiar apenas no type-stripping nativo do Node e perder recursos de `tsconfig`.
  **Mitigação:** manter `tsx` como runtime de desenvolvimento.

### 4) Instalar dependências de desenvolvimento

### O QUE é
Configurar toolchain de desenvolvimento/testes para TypeScript.

### O QUE NÃO é
- Não é setup completo de lint/format corporativo.
- Não é pipeline CI/CD nesta etapa.

### COMO implementar
Instalar:
```bash
npm i -D vitest @vitest/coverage-v8 tsx typescript @types/node @types/react
```

Criar `vitest.config.ts`:
```ts
import { defineConfig, configDefaults } from 'vitest/config';

export default defineConfig({
  test: {
    environment: 'node',
    include: ['src/**/*.test.ts', 'test/**/*.test.ts'],
    exclude: [...configDefaults.exclude, 'dist/**'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html'],
      include: ['src/**/*.ts'],
      exclude: ['src/**/*.test.ts']
    }
  }
});
```

Criar teste mínimo (`test/smoke.test.ts`) para provar execução do runner.

### CRITÉRIO DE ACEITAÇÃO
- `npm test` passa.
- `npm run test:coverage` gera relatório sem erro.
- Vitest roda em ambiente Node (não browser/jsdom por padrão).

### RISCOS
- **Risco:** configuração de include/exclude esconder arquivos relevantes.
  **Mitigação:** definir globs explícitas para `src` e revisar no primeiro relatório de cobertura.
