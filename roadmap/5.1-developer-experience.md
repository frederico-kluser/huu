## 1) `huu init` — initialize HUU in existing project

### O QUE É (What it is)

An idempotent bootstrap command that prepares the minimum runtime contract required by the orchestrator in the current repository:

1. `.huu/` directory.
2. SQLite database in WAL mode.
3. Valid default config file.

It must work both interactively (human-friendly) and non-interactively (CI/scripts).

### O QUE NÃO É (What it is not)

- Not a project scaffolder for application source code.
- Not a command that runs agents/orchestration.
- Not a command that installs Node dependencies.
- Not a destructive reset unless user explicitly confirms/forces.

### COMO implementar (Recommended technical approach)

#### Required artifact contract

`huu init` must guarantee all artifacts below after successful execution:

| Artifact | Required | Purpose |
|---|---:|---|
| `.huu/` | Yes | Runtime home directory |
| `.huu/huu.db` | Yes | Unified SQLite store (messages, memory, audit, beat state) |
| `.huu/config.json` | Yes | Runtime configuration consumed by CLI/orchestrator |
| SQLite `WAL` mode enabled | Yes | Concurrency + durability requirements from architecture |

#### Init lifecycle

1. **Preflight**
   - Validate current directory is writable.
   - Detect existing `.huu/` and determine if operation is first-init or re-init.
   - Resolve mode: interactive vs non-interactive (`--yes` / `--non-interactive`).
2. **Plan**
   - Build an in-memory operation plan (create directory, create/open DB, run migrations, write config).
   - If `--dry-run` is passed, print the plan and exit `0` with no mutations.
3. **Execute**
   - Create `.huu/` with `mkdir(..., { recursive: true })`.
   - Create/open `.huu/huu.db`.
   - Set PRAGMAs (`journal_mode=WAL`, recommended: `foreign_keys=ON`).
   - Run migrations from `0.2 SQLite Schema`.
   - Write `.huu/config.json` atomically (temp file + rename).
4. **Report**
   - Print summary of created/updated/skipped artifacts.
   - On re-run, clearly state idempotent behavior (e.g., “already exists, validated”).

#### Suggested CLI flags

- `--yes`: accept defaults, no confirmation prompts.
- `--non-interactive`: fail instead of prompting when input is missing.
- `--force`: allow overwrite of config after explicit warning.
- `--dry-run`: show plan only, no side effects.

#### Example config skeleton (written by init)

```json
{
  "version": 1,
  "projectRoot": ".",
  "database": {
    "path": ".huu/huu.db",
    "journalMode": "WAL"
  },
  "orchestrator": {
    "maxConcurrency": 5,
    "defaultAgentModel": {
      "orchestrator": "opus",
      "worker": "sonnet",
      "support": "haiku"
    }
  },
  "logging": {
    "level": "notice"
  }
}
```

#### Example implementation snippet

```ts
const plan = await buildInitPlan({ cwd, flags });
if (flags.dryRun) return printInitPlan(plan);

await fs.mkdir(path.join(cwd, ".huu"), { recursive: true });
const db = openDb(path.join(cwd, ".huu", "huu.db"));
db.pragma("journal_mode = WAL");
db.pragma("foreign_keys = ON");
runMigrations(db);
await writeJsonAtomic(path.join(cwd, ".huu", "config.json"), plan.config);
```

### CRITÉRIO DE ACEITAÇÃO (Definition of done)

- Running `huu init` in a clean repository creates `.huu/`, `.huu/huu.db`, `.huu/config.json`.
- Database opens and reports `journal_mode = wal`.
- Migrations complete and required tables from `0.2` exist.
- Re-running `huu init` is safe and does not corrupt existing state.
- `huu init --dry-run` produces a complete plan with zero filesystem/database mutations.

### RISCOS (Risks + mitigations)

- **Risk:** partial init if process crashes mid-way.  
  **Mitigation:** atomic config writes + transactional migrations + explicit rollback/diagnostics.
- **Risk:** accidental overwrite of existing config.  
  **Mitigation:** prompt/confirm by default, require `--force` in non-interactive mode.
- **Risk:** WAL not enabled due environment constraints.  
  **Mitigation:** verify PRAGMA result and fail with actionable error.

## 2) `huu config` — interactive configuration

### O QUE É (What it is)

A command that reads, validates, edits, and persists `.huu/config.json` through guided prompts and optional non-interactive overrides.

### O QUE NÃO É (What it is not)

- Not a replacement for `huu init`.
- Not a raw text editor that bypasses validation.
- Not a runtime command that executes orchestration.

### COMO implementar (Recommended technical approach)

#### Prompt library strategy (Node.js)

Use `@clack/prompts` as default interactive layer for this stage:

- Built-in cancellation guard (`isCancel`) and friendly cancellation output.
- Opinionated UX suitable for setup flows.
- Supports grouped prompts and progress feedback.

Keep adapter boundaries so `prompts` or `enquirer` can be swapped later if needed.

#### Minimum interactive flow

1. Load existing config (fail with clear guidance if missing: “run `huu init` first”).
2. Prompt for high-value fields:
   - model tier defaults,
   - max concurrency,
   - log level,
   - optional paths (if customizable).
3. Validate each field in-prompt and cross-field before save.
4. Show final summary diff and ask confirmation.
5. Write atomically and print success summary.

#### Cancellation behavior

- `Ctrl+C` exits cleanly with exit code `130`.
- No partial writes on cancellation.
- Partial answers may be shown in memory summary, but never persisted without explicit confirmation.

#### Suggested command options

- `--set key=value` (repeatable) for scripted updates.
- `--json` for machine-readable output.
- `--reset` to return selected keys to defaults.
- `--non-interactive` to disallow prompts and require explicit values.

### CRITÉRIO DE ACEITAÇÃO (Definition of done)

- `huu config` updates valid keys and rejects invalid values with clear suggestions.
- Interactive flow handles cancel gracefully without config corruption.
- Non-interactive mode supports automation safely.
- All writes are atomic and schema-validated.

### RISCOS (Risks + mitigations)

- **Risk:** schema drift as new options are added.  
  **Mitigation:** central JSON schema + versioned migrations for config shape.
- **Risk:** inconsistent validation between prompt and runtime.  
  **Mitigation:** single shared validation module reused by both.
- **Risk:** poor UX in long prompt chains.  
  **Mitigation:** group prompts by concern and keep defaults pre-filled.

## 3) Error messages with actionable suggestions

### O QUE É (What it is)

A standardized error system where every user-facing error includes:

1. what failed,
2. why it failed (if known),
3. what to do next (actionable suggestion),
4. proper exit code.

### O QUE NÃO É (What it is not)

- Not raw stack trace dumps in normal mode.
- Not generic “something went wrong” messages.
- Not silent failures.

### COMO implementar (Recommended technical approach)

#### Error contract

Adopt a typed error structure:

```ts
type CliError = {
  code: string;           // e.g., HUU_DB_WAL_UNAVAILABLE
  message: string;        // short human summary
  details?: string;       // optional technical context
  suggestion?: string;    // explicit next command/action
  docsUrl?: string;       // optional reference
  exitCode: number;       // stable process code
};
```

#### Message format pattern

```
Error [HUU_CONFIG_NOT_FOUND]: Configuration file was not found at .huu/config.json.
Suggestion: Run `huu init` to create the default runtime files.
```

#### Exit code policy

- `0`: success.
- `1`: generic runtime failure.
- `2`: usage/argument error.
- `130`: user cancelled (SIGINT-style behavior).

#### Verbose integration

- Default: show concise actionable message.
- `--verbose`: include stack/cause chain and internal context.
- `--quiet`: only essential error line.

### CRITÉRIO DE ACEITAÇÃO (Definition of done)

- Top recurring failure modes (`missing config`, `db open failure`, `invalid option`, `migration failure`) have clear suggestions.
- Error output is consistent across commands.
- Exit codes are stable and documented.
- Verbosity flags influence error detail as designed.

### RISCOS (Risks + mitigations)

- **Risk:** over-verbose messages reduce clarity.  
  **Mitigation:** short default + optional expanded detail.
- **Risk:** stale suggestions after refactors.  
  **Mitigation:** central error catalog with tests.
- **Risk:** undocumented exit-code changes break automation.  
  **Mitigation:** lock exit-code mapping and treat changes as breaking.

## 4) `--verbose` and `--quiet` modes

### O QUE É (What it is)

Global output controls to adjust signal density without changing core command behavior.

### O QUE NÃO É (What it is not)

- Not business-logic flags.
- Not replacements for explicit `--json` machine output.
- Not permission to suppress critical fatal errors.

### COMO implementar (Recommended technical approach)

#### Recommended level mapping

| Flags | Effective level | Behavior |
|---|---|---|
| `--quiet` / `-q` | `warn` (or `error`) | Hide informational progress output |
| default | `notice` | Human-friendly normal output |
| `-v` | `info` | More operational detail |
| `-vv` | `debug` | Deep troubleshooting detail |
| `-vvv` | `trace` | Full low-level trace |

#### Rules

- Treat `--quiet` and `--verbose` as conflicting flags (explicit parse error).
- Send logs/progress to `stderr`; keep machine-readable command output on `stdout`.
- Allow config default (`logging.level`) but let CLI flags override per execution.

#### Implementation note

Adopt a single logging facade (`logger`) with level gating and context metadata (`command`, `requestId`, `durationMs`).

### CRITÉRIO DE ACEITAÇÃO (Definition of done)

- All commands honor global verbosity flags consistently.
- `-vv` actually reveals more than `-v`; `-q` suppresses non-essential output.
- Conflict between `--quiet` and `-v` yields clear argument error.
- stdout/stderr separation works for shell pipelines.

### RISCOS (Risks + mitigations)

- **Risk:** leaking sensitive data in debug logs.  
  **Mitigation:** redact secrets before logging and document safe defaults.
- **Risk:** inconsistent levels across modules.  
  **Mitigation:** forbid ad-hoc `console.log`; enforce shared logger.
- **Risk:** automation break from noisy stdout.  
  **Mitigation:** reserve stdout for command results, not diagnostics.

## 5) `--dry-run` for beat sheet preview without execution

### O QUE É (What it is)

A preview mode that performs planning/decomposition and displays what would be executed, without spawning agents, creating worktrees, writing DB state, or mutating files.

### O QUE NÃO É (What it is not)

- Not a partial execution mode.
- Not a hidden write mode.
- Not a guarantee that runtime conditions will be identical at execution time.

### COMO implementar (Recommended technical approach)

#### Behavior contract

For `huu run "task" --dry-run`:

1. Run decomposition pipeline (Beat Sheet Engine `2.1`).
2. Compute assignment/scheduling preview in memory.
3. Render structured preview:
   - objective,
   - acts/sequences/atomic tasks,
   - dependency graph,
   - candidate agent per task,
   - estimated parallelism.
4. Explicitly print: “No side effects executed.”

#### Output modes

- Human-readable table/tree (default).
- `--json` for automation and UI integration.

#### Safety guarantee

Dry-run path must skip all side-effectful operations:

- no DB writes,
- no worktree creation,
- no file writes,
- no network mutations.

### CRITÉRIO DE ACEITAÇÃO (Definition of done)

- `--dry-run` returns full preview artifact for a valid task.
- Running with and without `--dry-run` yields same plan shape pre-execution.
- Instrumentation confirms zero side effects in dry-run path.
- Output clearly communicates simulation scope and limits.

### RISCOS (Risks + mitigations)

- **Risk:** drift between preview and real execution logic.  
  **Mitigation:** share planning code path; gate only side-effect executors.
- **Risk:** hidden side effects (e.g., implicit DB writes).  
  **Mitigation:** side-effect audit tests and strict interface boundaries.
- **Risk:** false user confidence.  
  **Mitigation:** display limitations explicitly (state may change before real run).
