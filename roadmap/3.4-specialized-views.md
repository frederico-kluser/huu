# 3.4 Specialized Views

> Phase 3 — TUI (Act 2 → Act 3: Resolution)

Especificação completa das abas especializadas da TUI: Logs, Merge Queue, Cost e Beat Sheet.

---

## Objetivo da etapa

Transformar as abas `[L]`, `[M]`, `[C]` e `[B]` em visões operacionais completas, sem duplicar responsabilidades da Kanban (3.1) e da Detail View (3.2).

## Alinhamento com `ARCHITECTURE.md`

Esta etapa está **coerente** com a arquitetura definida e deve implementar explicitamente:

- **Decisão 10 (Interface: Kanban + tabs K/L/M/C/B)**: as quatro views especializadas vivem dentro do shell da 3.1.
- **Decisão 4 (Merge workflow FIFO + 4 tiers)**: a aba de Merge Queue deve exibir FIFO real e progressão T1→T4.
- **Decisão 5 (Model tiering)**: a aba Cost precisa quebrar custo por agent/model/phase.
- **Decisão 6 (Beat Sheet hierárquico + checkpoints)**: a aba Beat Sheet deve representar os 4 níveis e marcos (Catalyst, Midpoint, All Is Lost, Break Into Three, Final Image).
- **Decisões 3, 7 e 12 (SQLite + auditabilidade)**: todas as views devem consumir dados do banco unificado com semântica de observabilidade.

---

## Fronteiras e não redundância entre views

| View | Responsabilidade principal | Não deve cobrir (já coberto em...) |
|---|---|---|
| Logs | Linha do tempo agregada cross-agent | Investigação profunda de um único card (`3.2 Detail View`) |
| Merge Queue | Estado operacional da fila de merge (FIFO + tiers) | Diff de conflito e resolução semântica detalhada (`2.4` / `3.2`) |
| Cost | Custo consolidado e tendências por dimensão | Faturamento externo/contábil ou otimização automática de roteamento |
| Beat Sheet | Progresso hierárquico macro da execução | Edição de plano/decomposição em si (`2.1`/`2.2`) |

---

## Padrões transversais (obrigatórios para as 4 abas)

1. **Navegação**
   - Atalhos globais permanecem consistentes: `k/l/m/c/b` para troca direta de tab.
   - Dentro de cada view: `↑/↓` navegação, `PgUp/PgDn` paginação, `Home/End` salto topo/fim, `Esc` limpa seleção/foco local.
   - Seguir convenção de widget composto: `Tab` troca região; setas navegam dentro da região.

2. **Estados de interface**
   - Toda aba deve ter estados explícitos: `loading`, `ready`, `empty`, `error`.
   - Estado `empty` precisa dizer qual dependência/dado está faltando (ex.: "nenhuma entrada na merge queue").

3. **Polling/refresh**
   - Polling padrão: `500ms` (ajustável por config).
   - Para view com volume alto (Logs), renderizar janela visível + buffer circular, não lista ilimitada.

4. **Acessibilidade em terminal**
   - Não depender só de cor para codificação de estado; usar também ícones/símbolos (`E`, `W`, `I`, `T1..T4`, `✔`, `✖`).
   - Destaque visual diferente para **foco** versus **seleção**.

5. **Performance**
   - Nunca recalcular agregações pesadas em cada keystroke sem memoização.
   - Usar `useMemo` para filtros/sumários e `useCallback` para handlers.

## Checklist detalhado

### - [ ] `[L]ogs` tab — aggregated log view from all agents

#### O QUE é
Uma visão unificada de eventos de execução (orchestrator + agents), ordenada no tempo, com filtros combináveis (agent, nível, fase) e busca textual/regex.

#### O QUE NÃO é
- Não substitui o **log aprofundado por card** da Detail View.
- Não é editor de logs.
- Não é armazenamento definitivo; é camada de apresentação sobre dados persistidos no SQLite.

#### COMO implementar
1. **Contrato de dados (normalização)**
   ```ts
   type LogLevel = 'debug' | 'info' | 'warn' | 'error';

   interface AggregatedLogEntry {
     id: string;           // estável e único
     ts: number;           // epoch ms
     agentId: string;      // orchestrator, builder, tester...
     phase?: string;       // setup, confrontation, resolution...
     level: LogLevel;
     taskId?: string;
     message: string;
     source: 'messages' | 'audit_log';
   }
   ```
2. **Pipeline**
   - Ler eventos de `messages` + `audit_log`.
   - Mapear para `AggregatedLogEntry`.
   - Ordenar por `(ts, id)` para estabilidade quando timestamps empatarem.
3. **UI**
   - Header: contadores (`total`, `errors`, `warns`, agents ativos), filtros ativos.
   - Corpo: lista scrollável com timestamp, badge de agent, nível, mensagem.
   - Footer: atalhos (`/` buscar, `f` filtros, `r` regex, `e` só erros, `g` fim da lista).
4. **Color-coding**
   - Nível: `error=red`, `warn=yellow`, `info=blue`, `debug=gray`.
   - Agent: paleta estável por hash de `agentId`.
   - Sempre imprimir símbolo textual junto da cor (`[ERR]`, `[WRN]`...).
5. **Busca e filtros**
   - Filtro combinável por `agentId[]`, `level[]`, `phase[]`.
   - Busca textual case-insensitive.
   - Regex opcional com fallback seguro se expressão inválida.
6. **Escalabilidade**
   - Buffer circular (ex.: `max=5000` linhas em memória).
   - Renderização de janela visível (ex.: últimas `200` linhas).
7. **Snippet base**
   ```tsx
   <Static items={visibleLogs}>
     {entry => (
       <Text key={entry.id}>
         [{fmtTime(entry.ts)}] [{entry.agentId}] [{entry.level.toUpperCase()}] {entry.message}
       </Text>
     )}
   </Static>
   ```

#### CRITÉRIO DE ACEITAÇÃO
- Usuário consegue filtrar por agent + nível simultaneamente.
- Busca destaca ocorrências sem travar render.
- Em carga de 1k+ linhas recentes, navegação permanece responsiva.
- Cada linha mostra contexto mínimo: tempo, agent, nível, mensagem.

#### RISCOS (e mitigação)
- **Explosão de memória** com stream longo → buffer circular.
- **Cor insuficiente para distinguir agents** → legenda fixa + símbolo textual.
- **Desordem temporal por relógios diferentes** → ordenação estável por `(ts, id)`.
- **Regex custoso** → limite de tamanho e opção de fallback para substring.

### - [ ] `[M]erge Queue` tab — FIFO queue with tier indicators

#### O QUE é
Painel operacional da fila de merge mostrando posição FIFO, status atual e tier de resolução (T1..T4) de cada item.

#### O QUE NÃO é
- Não reordena fila manualmente.
- Não implementa prioridade fora do modelo FIFO definido.
- Não exibe patch/diff completo de conflito (isso fica em detalhe específico).

#### COMO implementar
1. **Contrato de dados**
   ```ts
   type MergeStatus = 'queued' | 'running' | 'blocked' | 'merged' | 'failed';
   type MergeTier = 1 | 2 | 3 | 4;

   interface MergeQueueItemView {
     id: string;
     position: number;
     taskId: string;
     branch: string;
     enqueuedAt: number;
     waitMs: number;
     currentTier?: MergeTier;
     status: MergeStatus;
     retries: number;
     lastError?: string;
   }
   ```
2. **Regras de ordenação**
   - Itens `queued/running/blocked`: ordem FIFO por `enqueuedAt`.
   - Itens finalizados (`merged/failed`) em seção histórica separada (janela curta).
3. **UI**
   - Resumo superior: `queue length`, `running`, `blocked`, `avg wait`.
   - Tabela principal: `Pos | Task | Branch | Tier | Status | Wait | Retries`.
   - Painel de detalhe (linha selecionada): histórico de transição de tier + último erro.
4. **Visual de tier**
   - Badge textual obrigatório (`T1`, `T2`, `T3`, `T4`).
   - Cor sugerida: `T1 green`, `T2 cyan`, `T3 yellow`, `T4 magenta`.
5. **Atalhos**
   - `↑/↓`: mover seleção.
   - `Enter`: expandir detalhe.
   - `f`: filtrar por status.
   - `r`: refresh manual (além do polling).

#### CRITÉRIO DE ACEITAÇÃO
- A posição 1 é sempre o item mais antigo não finalizado.
- Trocas de tier aparecem em tempo quase real.
- Estado `blocked` sempre apresenta motivo legível.
- Histórico recente de `merged/failed` visível sem poluir fila ativa.

#### RISCOS (e mitigação)
- **Item travado bloqueando visibilidade operacional** → indicar SLA de espera e motivo explícito.
- **Duplicatas por retry/requeue** → chave única por merge request + deduplicação no adapter.
- **Perda de contexto de falha** → persistir `lastError` + timestamp de falha.

### - [ ] `[C]ost` tab — breakdown by agent, model, phase

#### O QUE é
Dashboard de custo e uso de tokens com visão consolidada e comparativa por agent, modelo e fase do beat sheet.

#### O QUE NÃO é
- Não é fatura oficial de provedor.
- Não faz chargeback contábil.
- Não decide automaticamente roteamento de modelos (apenas observabilidade).

#### COMO implementar
1. **Modelo de cálculo**
   ```ts
   interface UsageEvent {
     id: string;
     ts: number;
     agentId: string;
     model: string;
     phase?: string;
     promptTokens: number;
     completionTokens: number;
   }

   interface ModelRate {
     inputUsdPerMTok: number;
     outputUsdPerMTok: number;
   }
   ```
2. **Rate card inicial (arquitetura)**
   - Opus: `$15/MTok` (input, ajustar output por config).
   - Sonnet: `$3/MTok`.
   - Haiku: `$0.80/MTok`.
   - Guardar em config versionada, não hardcode espalhado.
3. **Fórmula**
   - `cost = (promptTokens/1_000_000)*inputRate + (completionTokens/1_000_000)*outputRate`.
   - Evitar drift: armazenar centavos/microdólares em inteiro internamente.
4. **UI recomendada**
   - Linha 1: KPIs (`Total USD`, `Total tokens`, `avg/task`, `runway` opcional).
   - Linha 2: tabela de breakdown por `agent/model/phase`.
   - Linha 3: tendência temporal (sparkline/asciichart) + top consumidores.
5. **Ferramentas visuais**
   - Tabela: `@inkjs/ui Table` (ou fallback monoespaçado).
   - Mini-gráfico: sparkline Unicode ou `asciichart` para série temporal.
6. **Filtros**
   - Janela temporal (últimos N min / sessão atual).
   - Agrupamento (`agent`, `model`, `phase`) alternável por atalho.

#### CRITÉRIO DE ACEITAÇÃO
- Soma por agrupamento bate com total (diferença apenas de arredondamento de exibição).
- Falta de metadado de tokens é tratada como `unknown` (não somar como zero silenciosamente).
- É possível alternar visão por dimensão sem perder contexto total.
- Tendência temporal mostra direção de gasto (subindo/estável/caindo).

#### RISCOS (e mitigação)
- **Rate card desatualizado** → versionar e exibir versão/fonte da tarifa.
- **Dados incompletos de usage** → bucket `unknown` + alerta visual.
- **Cálculo inconsistente entre views** → extrair utilitário único `calculateCost()`.

### - [ ] `[B]eat Sheet` tab — hierarchical progress view with checkpoints

#### O QUE é
Visualização hierárquica do progresso narrativo/técnico em 4 níveis:
`Objetivo → Ato → Sequência → Tarefa Atômica`, com checkpoints obrigatórios.

#### O QUE NÃO é
- Não edita decomposição/plano.
- Não substitui Kanban de execução.
- Não executa ação de intervenção (isso fica em fluxos próprios).

#### COMO implementar
1. **Contrato de árvore**
   ```ts
   type BeatNodeType = 'objective' | 'act' | 'sequence' | 'task';
   type BeatStatus = 'pending' | 'running' | 'done' | 'blocked';

   interface BeatNode {
     id: string;
     parentId?: string;
     type: BeatNodeType;
     title: string;
     status: BeatStatus;
     progressPct: number; // 0..100
     checkpoint?: 'catalyst' | 'midpoint' | 'all_is_lost' | 'break_into_three' | 'final_image';
   }
   ```
2. **Render hierárquico**
   - Construir árvore + lista achatada para render (`flattenVisibleNodes`).
   - Manter `expandedNodeIds` e `selectedNodeId`.
   - Indentação por nível + glifo de expansão (`▸`/`▾`).
3. **Navegação**
   - `↑/↓`: item anterior/próximo visível.
   - `→`: expandir nó (ou ir para primeiro filho).
   - `←`: colapsar nó (ou ir para pai).
   - `Home/End`: primeiro/último visível.
4. **Checkpoint lane**
   - Painel resumido com estado dos marcos obrigatórios:
     `Catalyst`, `Midpoint`, `All Is Lost`, `Break Into Three`, `Final Image`.
   - Cada checkpoint exibe `status + evidência` (id da sequência/tarefa associada).
5. **Painel de detalhe lateral**
   - Para nó selecionado: precondição, pós-condição, bloqueios, owner agent, timestamps.

#### CRITÉRIO DE ACEITAÇÃO
- Estrutura de 4 níveis renderiza sem quebrar alinhamento.
- Usuário navega e expande/colapsa apenas com teclado.
- Checkpoints obrigatórios são visíveis e atualizam com o estado real.
- Estados de nós (`pending/running/done/blocked`) ficam distinguíveis por símbolo + cor.

#### RISCOS (e mitigação)
- **Árvore profunda demais para viewport** → colapsar por padrão níveis profundos e permitir paginação.
- **Confusão entre foco e seleção** → estilos distintos e legenda.
- **Inconsistência de hierarquia no dado** → validação no adapter (órfãos/loops).
