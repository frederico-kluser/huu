# 3.1 Kanban Board

> Phase 3 — TUI (Act 2 -> Act 3: Resolution)

Visao principal da TUI: um board Kanban com 5 estados fixos para dar visibilidade operacional imediata do sistema multi-agent.

## Revisao de coerencia com a arquitetura (ARCHITECTURE.md)

Status da revisao: **coerente com ajustes obrigatorios de detalhamento**.

- **Alinhado** com a Decisao 10: Kanban de 5 colunas (`Backlog -> Running -> Review -> Done -> Failed`), header com ato/beat/custo e navegacao por teclado.
- **Alinhado** com a Decisao 3: dados vindos de SQLite WAL (fonte unica para estado e mensagens).
- **Alinhado** com o modelo human-in-the-loop: board prioriza observabilidade e selecao de tarefa para drill-down.
- **Gap identificado e corrigido nesta especificacao**: dependencia explicita de `2.1 Beat Sheet Engine` para preencher `act/beat` no header (nao apenas `0.2` e `2.2`).
- **Escopo preservado**: 3.1 entrega shell + board + navegacao + refresh; detalhes profundos ficam em 3.2/3.3/3.4.

## Checklist (detalhado para implementacao)

- [ ] Ink app shell com navegacao de tabs (K/L/M/C/B)
- [ ] Kanban de 5 colunas: Backlog, Running, Review, Done, Failed
- [ ] Card: task ID, name, agent icon, model, elapsed time, cost
- [ ] Header: current act, current beat, total cost
- [ ] Navegacao por teclado (setas para selecionar cards)
- [ ] Auto-refresh via polling eficiente no SQLite

## Padroes AGTX aplicados (terminal Kanban)

- **Overview first, drill-down depois:** board sempre abre com panorama global; `Enter` delega detalhe para 3.2.
- **Tabs previsiveis da esquerda para direita:** padrao operacional `K -> L -> M -> C -> B`, com saltos diretos por tecla.
- **Estado unico e observavel:** view sempre deriva de snapshot SQLite + selectors puros (sem estado paralelo oculto).
- **Interacao deterministica:** mesma tecla sempre produz mesma acao no mesmo contexto de foco.
- **Escalabilidade visual por densidade:** o mesmo board se adapta por breakpoints sem trocar o modelo mental.

### 1) Ink app shell com tab navigation (K/L/M/C/B)

**O QUE E**  
Criar o shell principal da TUI (`App.tsx`) com roteamento por tabs: `[K]anban`, `[L]ogs`, `[M]erge Queue`, `[C]ost`, `[B]eat Sheet`, mantendo Kanban como view default.

**O QUE NAO E**  
Nao implementa o conteudo completo de `L/M/C/B` nesta etapa. Estas tabs podem exibir placeholders estaveis ate 3.4 ficar pronta.

**COMO implementar**  
1. Definir um enum/tipo para tabs e um mapa de keybindings.  
2. Capturar teclado com `useInput`, priorizando atalhos diretos (`k/l/m/c/b`) e fallback com `leftArrow/rightArrow`.  
3. Usar `useInput(..., {isActive})` para evitar conflito com handlers internos de cards.  
4. Manter estado de tab em nivel de app e passar contexto de foco para subviews.

```tsx
type AppTab = 'kanban' | 'logs' | 'merge' | 'cost' | 'beat';

const TAB_BY_KEY: Record<string, AppTab> = {
  k: 'kanban',
  l: 'logs',
  m: 'merge',
  c: 'cost',
  b: 'beat',
};
```

**CRITERIO DE ACEITACAO**  
- Pressionar `k/l/m/c/b` muda de tab sem flicker.  
- Kanban abre por padrao.  
- Tabs nao-Kanban renderizam placeholder claro e estavel.  
- `ESC`/`q` sai via `useApp().exit()` sem deixar terminal quebrado.

**RISCOS**  
- **Conflito de teclas** entre shell e board interno.  
  - **Mitigacao:** `isActive` por camada + ordem de prioridade explicita de handlers.

### 2) Kanban 5-colunas (Backlog, Running, Review, Done, Failed)

**O QUE E**  
Renderizar o board com as 5 colunas fixas da arquitetura, agrupando tarefas por status e mostrando distribuicao de trabalho em tempo real.

**O QUE NAO E**  
Nao inclui drag-and-drop, edicao manual de status, nem reordenacao persistida no banco.

**COMO implementar**  
1. Mapear status do dominio para colunas fixas (`backlog`, `running`, `review`, `done`, `failed`).  
2. Agrupar tarefas em memoria (selector puro) antes de renderizar.  
3. Renderizar colunas com `Box` + Flexbox (`flexDirection="row"`, `flexGrow`, `minWidth`).  
4. Usar `useWindowSize()` para responsividade por breakpoint (detalhado em "Validacao de layout").  
5. Em telas estreitas, usar modo paginado horizontal de colunas (mantendo as 5 colunas logicas).

```tsx
const STATUSES = ['backlog', 'running', 'review', 'done', 'failed'] as const;
const grouped = groupTasksByStatus(tasks, STATUSES);
```

**CRITERIO DE ACEITACAO**  
- Toda tarefa aparece em exatamente uma coluna valida.  
- Ordem de colunas permanece fixa (Backlog -> Failed).  
- Coluna vazia continua visivel com contagem `0`.  
- Sem quebra visual em 80, 120 e 200+ colunas.

**RISCOS**  
- **Overflow/truncamento agressivo** em terminais pequenos.  
  - **Mitigacao:** breakpoints com densidade visual e truncamento previsivel (`wrap="truncate"`).  
- **Status invalido vindo do backend.**  
  - **Mitigacao:** fallback explicito para `backlog` + log de anomalia.

### 3) Card component (task ID, name, agent icon, model, elapsed time, cost)

**O QUE E**  
Componente visual unico de tarefa, reutilizavel em todas as colunas, com destaque de selecao/foco.

**O QUE NAO E**  
Nao e detail view completo (logs, diff, metricas profundas), nao exibe historico da tarefa.

**COMO implementar**  
1. Criar `Card.tsx` com props tipadas (`task`, `isSelected`, `density`).  
2. Mostrar campos obrigatorios nesta prioridade: `id -> name -> agent/model -> elapsed -> cost`.  
3. Aplicar fallback textual (`unknown`, `n/a`) quando dado faltar.  
4. Adaptar densidade por largura:
   - `compact` (80-119): linha curta com campos abreviados.
   - `normal` (120-199): 2 linhas.
   - `wide` (200+): 2-3 linhas com nome completo.
5. Usar borda/cor para estado selecionado.

```tsx
interface KanbanCardTask {
  id: string;
  name: string;
  agent: string;
  model: string;
  elapsedMs: number;
  costUsd: number;
}
```

**CRITERIO DE ACEITACAO**  
- Card sempre mostra ID e nome (mesmo truncado).  
- Estado selecionado e visualmente obvio.  
- Custo e tempo tem formato consistente (ex.: `03:12`, `$0.42`).  
- Nao ocorre quebra de layout por string longa.

**RISCOS**  
- **Ruido visual por excesso de metadados.**  
  - **Mitigacao:** densidade por breakpoint + prioridade de campos.  
- **Dados faltantes** em estados intermediarios.  
  - **Mitigacao:** placeholders explicitos e nao silenciosos.

### 4) Header (current act, current beat, total cost)

**O QUE E**  
Barra superior do board com status global narrativo/financeiro: `Act`, `Beat`, `Total Cost`, e opcionalmente `Last Refresh`.

**O QUE NAO E**  
Nao substitui painel completo de metricas (3.2) nem analytics por agente/fase (3.4 Cost tab).

**COMO implementar**  
1. Ler `act/beat` do estado de beat sheet (`beat_state`, dependencia 2.1).  
2. Ler custo acumulado da fonte canonicamente definida (mensagens/eventos do orquestrador).  
3. Renderizar em `Header.tsx` com `Box` + `Spacer` para alinhamento estavel.  
4. Em 80 colunas, reduzir verbosidade (labels curtos).

```tsx
<Box borderStyle="single" paddingX={1}>
  <Text>Act {act}</Text>
  <Spacer />
  <Text>Beat {beat}</Text>
  <Spacer />
  <Text>Total ${totalCost.toFixed(2)}</Text>
</Box>
```

**CRITERIO DE ACEITACAO**  
- Header atualiza quando beat muda.  
- Custo total nunca fica negativo/NaN na UI.  
- Em 80 colunas continua legivel (sem overlap).

**RISCOS**  
- **Inconsistencia de custo** por fonte de dados ambigua.  
  - **Mitigacao:** definir contrato unico de agregacao e testar contra snapshots de DB.

### 5) Keyboard navigation (arrow keys para selecionar cards)

**O QUE E**  
Navegacao keyboard-first no board: mover coluna/cartao, selecionar tarefa e preparar entrada no detail view.

**O QUE NAO E**  
Nao inclui comandos de intervencao humana (`S/F/A/P`) - estes pertencem a 3.3.

**COMO implementar**  
1. Modelar selecao como par `{columnIndex, rowIndex}`.  
2. `left/right` troca coluna; `up/down` muda card dentro da coluna.  
3. Opcional recomendado: aliases `h/j/k/l` para users avancados, sem remover setas.  
4. `enter` emite evento `onOpenTask(taskId)` para integrar com 3.2 depois.  
5. Garantir highlight consistente no card selecionado.

```tsx
type Selection = {columnIndex: number; rowIndex: number};
```

**CRITERIO DE ACEITACAO**  
- Nao existe estado "selecionado invalido" (fora dos limites).  
- Ao trocar de coluna vazia, selecao vai para placeholder de coluna e nao quebra.  
- `Enter` em card valido dispara callback unico.

**RISCOS**  
- **Perda de foco** ao alternar tabs e voltar ao Kanban.  
  - **Mitigacao:** persistir selecao por tab e restaurar ao reentrar.

### 6) Auto-refresh from SQLite polling

**O QUE E**  
Atualizacao quase em tempo real do board via polling no SQLite WAL, com custo de CPU baixo e sem consultas sobrepostas.

**O QUE NAO E**  
Nao usa websocket/pub-sub nesta etapa; nao implementa stream de log detalhado (3.2).

**COMO implementar**  
1. Criar hook dedicado (`useKanbanData`) em vez de polling inline no componente.  
2. Usar loop com `setTimeout` recursivo (nao `setInterval`) para impedir overlap de query.  
3. Executar **change detection em 2 fases**:
   - Fase A: query barata de watermark (ex.: max id/timestamp relevante).
   - Fase B: snapshot completo apenas se watermark mudar.
4. Fazer `setState` somente quando snapshot realmente mudou.  
5. Intervalo adaptativo recomendado:
   - `500ms` com tarefas `running`
   - `1500ms` sem tarefas ativas
   - polling pausado quando tab Kanban nao estiver ativa
6. Cleanup obrigatorio no unmount.

```tsx
// Pseudo-codigo
// 1) readWatermark() -> string
// 2) se mudou: readBoardSnapshot()
// 3) shallow/deep compare e setState apenas quando necessario
```

**CRITERIO DE ACEITACAO**  
- Nao ha queries simultaneas para o mesmo polling loop.  
- Rerender nao acontece quando nao existe mudanca real de dados.  
- Mudancas no SQLite aparecem na UI no SLA esperado (<= 1.5s em idle, <= 0.5s em running).

**RISCOS**  
- **Polling agressivo** degradando CPU/IO.  
  - **Mitigacao:** intervalo adaptativo + change detection + prepared statements.  
- **`SQLITE_BUSY` em picos de escrita.**  
  - **Mitigacao:** leitura curta, retry com backoff curto, sem loop apertado.

## Validacao de layout por tamanho de terminal (obrigatorio)

| Largura | Modo | Comportamento esperado |
|---|---|---|
| 80-119 cols | `compact` | Header reduzido, cards compactos, colunas paginadas horizontalmente (2 por vez), truncamento controlado |
| 120-199 cols | `normal` | 5 colunas visiveis, cards em 2 linhas, metadados principais visiveis |
| 200+ cols | `wide` | 5 colunas confortaveis, nome completo e metadados adicionais sem truncamento agressivo |

### Como verificar tecnicamente

1. **Teste de render virtual** com `renderToString(..., {columns})` para 80, 120 e 200.  
2. **Teste interativo manual** redimensionando terminal real e validando:
   - navegacao nao quebra
   - selecao permanece visivel
   - header continua legivel
3. **Teste de estabilidade** com polling ativo por alguns minutos (sem memory leak perceptivel).
