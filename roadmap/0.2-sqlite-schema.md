## Melhores práticas usadas nesta especificação (web + context7 + docs internos)

- SQLite WAL oficial: concorrência (muitos leitores + 1 escritor), checkpoints, limites em network FS.
- `better-sqlite3` performance guide: `journal_mode=WAL`, checkpoint manual em caso de starvation, `synchronous` em WAL.
- PRAGMA patterns para produção: `foreign_keys=ON`, `busy_timeout`, `wal_autocheckpoint`.
- Migrações em TypeScript para SQLite: versionamento por `schema_migrations` + opcionalmente `PRAGMA user_version`, transações `BEGIN IMMEDIATE`.
- Knowledge graph em SQLite: `entities` + `relations` com `UNIQUE` composto, FKs e índices por direção de aresta.
- Queue patterns em SQLite: status (`pending/processing/acked/dead_letter`), `available_at`, timeout de lock, retries com `attempt_count`.
- Audit log: append-only, sanitização de params/result, trigger anti-update/delete, hash chain opcional para tamper-evidence.
- Context7: usado como referência de documentação viva (version-aware) para reduzir risco de decisões com API desatualizada.

## 1) Database initialization with WAL mode

### O QUE é

Inicializar o banco SQLite único do projeto com pragmas de produção compatíveis com o modelo arquitetural local-first.

### O QUE NÃO é

- Não é sharding, réplica, cluster ou banco remoto.
- Não é otimização prematura com dezenas de pragmas sem benchmark.
- Não é suporte a WAL em filesystem de rede.

### COMO implementar

```ts
import Database from 'better-sqlite3';

export function openDatabase(filePath: string): Database.Database {
  const db = new Database(filePath, { timeout: 5000 });
  db.pragma('journal_mode = WAL');
  db.pragma('synchronous = NORMAL');
  db.pragma('foreign_keys = ON');
  db.pragma('busy_timeout = 5000');
  db.pragma('temp_store = MEMORY');
  db.pragma('wal_autocheckpoint = 1000');
  return db;
}
```

Recomendação operacional:

- Executar `PRAGMA wal_checkpoint(RESTART)` quando `*.db-wal` passar de limite definido (ex.: 128MB).
- Manter transações de escrita curtas.
- Nunca usar DB em volume de rede (NFS/SMB) com WAL.

### CRITÉRIO DE ACEITAÇÃO

- `PRAGMA journal_mode` retorna `wal`.
- `PRAGMA foreign_keys` retorna `1`.
- Teste concorrente com múltiplos leitores + 1 escritor sem `SQLITE_BUSY` recorrente.
- Checkpoint manual reduz arquivo `-wal` quando crescimento excede limite.

### RISCOS

- **Checkpoint starvation** por leituras longas -> mitigar com checkpoints periódicos.
- **Durabilidade insuficiente** em `synchronous=NORMAL` -> usar `FULL` se requisito exigir.
- **Contenção de escrita** -> limitar sessões escritoras concorrentes e reduzir duração de transação.

## 2) `messages` table (mail system tipado + fila)

### O QUE é

Fila persistente tipada para comunicação entre orquestrador, agentes, merger e intervenções humanas (TUI).

### O QUE NÃO é

- Não é broker distribuído (Kafka/RabbitMQ).
- Não é stream infinito sem política de retenção.
- Não é broadcast mágico com uma linha para múltiplos ACKs (broadcast deve ser fan-out em N mensagens unicast).

### COMO implementar

Tipos obrigatórios (núcleo arquitetural):

- `task_assigned`
- `task_progress`
- `task_done`
- `merge_ready`
- `merge_result`
- `escalation`
- `health_check`
- `broadcast`

Tipos estendidos (cobertura completa dos fluxos do orquestrador e intervenção humana):

- `steer` (TUI/orchestrator -> agent)
- `follow_up` (TUI/orchestrator -> agent)
- `abort_requested` (TUI/orchestrator -> agent)
- `abort_ack` (agent -> orchestrator)
- `promote_instinct` (TUI/orchestrator -> context-curator)

Cobertura de fluxo:

| Fluxo | Tipo de mensagem |
|---|---|
| Atribuição de tarefa | `task_assigned` |
| Heartbeat/progresso | `task_progress` |
| Conclusão de tarefa | `task_done` |
| Entrada na fila de merge | `merge_ready` |
| Resultado do merge | `merge_result` |
| Escalação de bloqueio | `escalation` |
| Ping de saúde | `health_check` |
| Comunicação geral | `broadcast` (fan-out unicast) |
| Steering em runtime | `steer` |
| Instrução pós-turno | `follow_up` |
| Cancelamento | `abort_requested` + `abort_ack` |
| Promoção de aprendizado | `promote_instinct` |

Semântica de fila recomendada:

- `status`: `pending`, `processing`, `acked`, `dead_letter`
- `available_at`: atraso/agenda (inclui follow-up)
- `lock_expires_at`: visibility timeout
- `attempt_count` + `max_attempts`: retry seguro

Exemplo de dequeue atômico:

```sql
BEGIN IMMEDIATE;

UPDATE messages
SET status = 'processing',
    locked_at = strftime('%Y-%m-%dT%H:%M:%fZ','now'),
    lock_expires_at = strftime('%Y-%m-%dT%H:%M:%fZ','now','+30 seconds'),
    attempt_count = attempt_count + 1,
    updated_at = strftime('%Y-%m-%dT%H:%M:%fZ','now')
WHERE id = (
  SELECT id
  FROM messages
  WHERE recipient_agent = :recipient
    AND status = 'pending'
    AND available_at <= strftime('%Y-%m-%dT%H:%M:%fZ','now')
  ORDER BY priority ASC, id ASC
  LIMIT 1
)
RETURNING *;

COMMIT;
```

### CRITÉRIO DE ACEITAÇÃO

- Mensagens de todos os tipos acima são inseríveis e consultáveis.
- Dequeue concorrente não entrega a mesma mensagem para dois consumidores.
- Mensagens expiradas em `processing` retornam para retry ou vão para DLQ ao atingir `max_attempts`.
- Índices suportam polling sem full scan.

### RISCOS

- **Duplicidade de processamento** por retry -> usar `correlation_id` e handlers idempotentes.
- **Backlog infinito** -> política de retenção/arquivamento e DLQ.
- **Broadcast inconsistente** -> sempre fan-out unicast por destinatário ativo.

## 3) `entities` table (knowledge graph nodes)

### O QUE é

Nós do grafo de conhecimento do projeto (arquivos, decisões, padrões, artefatos, constraints, etc.).

### O QUE NÃO é

- Não é repositório de embeddings vetoriais.
- Não é tabela de logs de tool call.
- Não é substituto de documentação humana completa.

### COMO implementar

Modelar com chave canônica estável por projeto:

- `project_id`
- `entity_type`
- `canonical_key` (única por projeto)
- `display_name`
- `summary`
- `metadata_json`
- `first_seen_at`, `last_seen_at`
- `source_message_id`

`UNIQUE(project_id, canonical_key)` evita duplicação semântica.

### CRITÉRIO DE ACEITAÇÃO

- Inserção idempotente por `canonical_key`.
- Atualização de `last_seen_at` ao reencontrar entidade.
- Consulta por tipo e recência com índice.

### RISCOS

- **Explosão de cardinalidade** por chaves mal normalizadas -> padronizar canonicalização.
- **Entidades ambíguas** -> exigir `entity_type` e fonte (`source_message_id`).

## 4) `relations` table (knowledge graph edges)

### O QUE é

Arestas direcionais entre entidades, com tipo de relação e confiança.

### O QUE NÃO é

- Não é dedução automática de causalidade sem evidência.
- Não é grafo não-direcional implícito.

### COMO implementar

Campos essenciais:

- `from_entity_id`, `to_entity_id` (FK para `entities`)
- `relation_type` (ex.: `depends_on`, `implements`, `contradicts`, `references`)
- `confidence` (`0.0..1.0`)
- `metadata_json`
- `source_message_id`

Constraints:

- `CHECK(from_entity_id <> to_entity_id)`
- `UNIQUE(project_id, from_entity_id, relation_type, to_entity_id)`

Índices:

- `(project_id, from_entity_id, relation_type)`
- `(project_id, to_entity_id, relation_type)`

### CRITÉRIO DE ACEITAÇÃO

- Não permite autorrelação trivial.
- Não permite duplicata da mesma aresta semântica.
- Permite travessia eficiente por origem e destino.

### RISCOS

- **Grafo ruidoso** por arestas fracas -> usar limiar mínimo de confiança.
- **Relações órfãs** -> FKs com `ON DELETE CASCADE`.

## 5) `observations` table (eventos de ferramentas, decay 30 dias)

### O QUE é

Registro determinístico de observações de execução (hooks/tool calls) para alimentar aprendizado.

### O QUE NÃO é

- Não é audit trail legal completo (isso é função de `audit_log`).
- Não é armazenamento eterno.

### COMO implementar

Campos:

- `session_id`, `agent_id`, `tool_name`, `tool_phase` (`pre`/`post`)
- `input_summary`, `output_summary`, `success`
- `latency_ms`, `tokens_input`, `tokens_output`, `cost_usd`
- `occurred_at`, `expires_at` (`+30 dias`)
- `metadata_json`

Política de decay:

- Job periódico remove ou arquiva `observations` expiradas.
- Nunca confiar em trigger de tempo para limpeza automática no SQLite.

### CRITÉRIO DE ACEITAÇÃO

- Toda execução relevante de tool gera observação (cobertura 100% dos hooks definidos).
- Query de observações recentes por agente/tool é indexada.
- Cleanup de expiradas é testado.

### RISCOS

- **Crescimento de volume** -> TTL e rotina de compactação.
- **Dados sensíveis em resumo** -> sanitização antes do insert.

## 6) `sessions` table (resumo + janela de 7 dias)

### O QUE é

Estado macro de sessões do orquestrador para recuperação de contexto recente.

### O QUE NÃO é

- Não é histórico eterno detalhado de cada tool call.
- Não é substituto de logs de execução completos.

### COMO implementar

Campos:

- `id` (session id), `project_id`, `status`
- `started_at`, `ended_at`
- `summary_markdown`, `summary_json`
- `total_messages`, `total_tool_calls`, `total_cost_usd`

Recuperação de contexto:

```sql
SELECT *
FROM sessions
WHERE project_id = :project_id
  AND ended_at >= strftime('%Y-%m-%dT%H:%M:%fZ','now','-7 days')
ORDER BY ended_at DESC;
```

### CRITÉRIO DE ACEITAÇÃO

- Sessão inicia como `running` e termina com status final consistente.
- Resumo pode ser recuperado por janela de 7 dias.
- Métricas agregadas batem com contagens reais (amostragem em teste).

### RISCOS

- **Resumo pobre/ruidoso** -> padronizar template de resumo.
- **Deriva de métricas** -> atualizar contadores em transações atômicas.

## 9) `audit_log` table (append-only)

### O QUE é

Rastro imutável de chamadas de ferramentas e ações críticas para verificação pós-sessão.

### O QUE NÃO é

- Não é dump integral bruto de stdout/stderr sem filtro.
- Não é local para segredo/token.

### COMO implementar

Campos:

- `session_id`, `agent_id`, `tool_name`
- `params_json` (sanitizado), `result_json` (resumo)
- `result_status` (`success`, `error`)
- `duration_ms`, `message_id`
- `prev_hash`, `entry_hash` (tamper-evidence opcional)
- `created_at`

Triggers de imutabilidade:

```sql
CREATE TRIGGER audit_log_no_update
BEFORE UPDATE ON audit_log
BEGIN
  SELECT RAISE(ABORT, 'audit_log is append-only');
END;

CREATE TRIGGER audit_log_no_delete
BEFORE DELETE ON audit_log
BEGIN
  SELECT RAISE(ABORT, 'audit_log is append-only');
END;
```

### CRITÉRIO DE ACEITAÇÃO

- Toda tool call relevante gera uma linha de auditoria.
- Tentativas de `UPDATE/DELETE` falham por trigger.
- Campos sensíveis são redigidos/sanitizados antes da persistência.

### RISCOS

- **Vazamento de segredo** -> política explícita de redaction + testes.
- **Alto volume** -> retenção por política e export de longo prazo.

## 10) Migration system (versionamento)

### O QUE é

Sistema determinístico de evolução de schema com histórico verificável.

### O QUE NÃO é

- Não é `syncSchema` implícito em runtime sem controle.
- Não é migration automática sem revisão de SQL.

### COMO implementar

Estrutura:

- `src/db/migrations/0001_init.sql`
- `src/db/migrations/0002_*.sql`, ...
- tabela `schema_migrations(version, name, checksum, applied_at)`

Regras:

1. Carregar migrations por ordem numérica.
2. Para cada migration:
   - `BEGIN IMMEDIATE`
   - validar checksum (se já aplicada, checksum precisa bater)
   - aplicar SQL
   - inserir em `schema_migrations`
   - atualizar `PRAGMA user_version`
   - `COMMIT`
3. Em erro: `ROLLBACK` e falha explícita do startup.

### CRITÉRIO DE ACEITAÇÃO

- Banco vazio sobe do zero até versão atual sem intervenção manual.
- Banco parcialmente migrado retoma do ponto correto.
- Alteração de migration já aplicada é detectada por checksum.

### RISCOS

- **Drift de schema** entre ambientes -> checksum + CI.
- **Migração concorrente** em dois processos -> lock via `BEGIN IMMEDIATE`.

### O QUE é

Suite de testes para garantir corretude do contrato de dados.

### O QUE NÃO é

- Não é benchmark de performance extrema.

### RISCOS

- **Falso positivo** por testar apenas happy-path -> incluir testes de falha e concorrência.
- **Flakiness temporal** -> controlar clock (helper) nos testes.

## DDL de referência (migration `0001_init.sql`)

```sql
PRAGMA foreign_keys = ON;

CREATE TABLE IF NOT EXISTS schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  checksum TEXT NOT NULL,
  applied_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE TABLE IF NOT EXISTS messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id TEXT NOT NULL,
  run_id TEXT,
  correlation_id TEXT,
  causation_id INTEGER REFERENCES messages(id),
  message_type TEXT NOT NULL CHECK (message_type IN (
    'task_assigned','task_progress','task_done',
    'merge_ready','merge_result','escalation',
    'health_check','broadcast',
    'steer','follow_up','abort_requested','abort_ack','promote_instinct'
  )),
  sender_agent TEXT NOT NULL,
  recipient_agent TEXT NOT NULL,
  priority INTEGER NOT NULL DEFAULT 100,
  payload_json TEXT NOT NULL CHECK (json_valid(payload_json)),
  status TEXT NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending','processing','acked','dead_letter')),
  attempt_count INTEGER NOT NULL DEFAULT 0,
  max_attempts INTEGER NOT NULL DEFAULT 5 CHECK (max_attempts >= 1),
  available_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  locked_at TEXT,
  lock_expires_at TEXT,
  acked_at TEXT,
  error_text TEXT,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE INDEX IF NOT EXISTS idx_messages_dequeue
  ON messages (recipient_agent, status, available_at, priority, id);
CREATE INDEX IF NOT EXISTS idx_messages_timeout
  ON messages (status, lock_expires_at);
CREATE INDEX IF NOT EXISTS idx_messages_type_created
  ON messages (message_type, created_at);
CREATE INDEX IF NOT EXISTS idx_messages_correlation
  ON messages (correlation_id);

CREATE TABLE IF NOT EXISTS sessions (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('running','completed','failed','aborted')),
  started_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  ended_at TEXT,
  summary_markdown TEXT,
  summary_json TEXT NOT NULL DEFAULT '{}' CHECK (json_valid(summary_json)),
  total_messages INTEGER NOT NULL DEFAULT 0,
  total_tool_calls INTEGER NOT NULL DEFAULT 0,
  total_cost_usd REAL NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE INDEX IF NOT EXISTS idx_sessions_recent
  ON sessions (project_id, ended_at DESC);

CREATE TABLE IF NOT EXISTS entities (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id TEXT NOT NULL,
  entity_type TEXT NOT NULL,
  canonical_key TEXT NOT NULL,
  display_name TEXT NOT NULL,
  summary TEXT,
  metadata_json TEXT NOT NULL DEFAULT '{}' CHECK (json_valid(metadata_json)),
  confidence REAL NOT NULL DEFAULT 0.7 CHECK (confidence >= 0.0 AND confidence <= 1.0),
  source_message_id INTEGER REFERENCES messages(id) ON DELETE SET NULL,
  first_seen_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  last_seen_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  UNIQUE(project_id, canonical_key)
);

CREATE INDEX IF NOT EXISTS idx_entities_type
  ON entities (project_id, entity_type, last_seen_at DESC);

CREATE TABLE IF NOT EXISTS relations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id TEXT NOT NULL,
  from_entity_id INTEGER NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
  to_entity_id INTEGER NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
  relation_type TEXT NOT NULL,
  confidence REAL NOT NULL DEFAULT 0.7 CHECK (confidence >= 0.0 AND confidence <= 1.0),
  metadata_json TEXT NOT NULL DEFAULT '{}' CHECK (json_valid(metadata_json)),
  source_message_id INTEGER REFERENCES messages(id) ON DELETE SET NULL,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  last_seen_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  CHECK (from_entity_id <> to_entity_id),
  UNIQUE(project_id, from_entity_id, relation_type, to_entity_id)
);

CREATE INDEX IF NOT EXISTS idx_relations_from
  ON relations (project_id, from_entity_id, relation_type);
CREATE INDEX IF NOT EXISTS idx_relations_to
  ON relations (project_id, to_entity_id, relation_type);

CREATE TABLE IF NOT EXISTS observations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id TEXT NOT NULL,
  session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  agent_id TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  tool_phase TEXT NOT NULL CHECK (tool_phase IN ('pre','post')),
  input_summary TEXT,
  output_summary TEXT,
  success INTEGER NOT NULL CHECK (success IN (0,1)),
  latency_ms INTEGER,
  tokens_input INTEGER,
  tokens_output INTEGER,
  cost_usd REAL,
  metadata_json TEXT NOT NULL DEFAULT '{}' CHECK (json_valid(metadata_json)),
  occurred_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  expires_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now','+30 days'))
);

CREATE INDEX IF NOT EXISTS idx_observations_recent
  ON observations (project_id, occurred_at DESC);
CREATE INDEX IF NOT EXISTS idx_observations_expiry
  ON observations (expires_at);
CREATE INDEX IF NOT EXISTS idx_observations_agent_tool
  ON observations (project_id, agent_id, tool_name, occurred_at DESC);

CREATE TABLE IF NOT EXISTS instincts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id TEXT NOT NULL,
  title TEXT NOT NULL,
  instinct_text TEXT NOT NULL,
  confidence REAL NOT NULL CHECK (confidence >= 0.30 AND confidence <= 0.85),
  state TEXT NOT NULL CHECK (state IN ('candidate','active','deprecated','promoted')),
  evidence_count INTEGER NOT NULL DEFAULT 0,
  contradiction_count INTEGER NOT NULL DEFAULT 0,
  source_observation_id INTEGER REFERENCES observations(id) ON DELETE SET NULL,
  metadata_json TEXT NOT NULL DEFAULT '{}' CHECK (json_valid(metadata_json)),
  last_validated_at TEXT,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  UNIQUE(project_id, title)
);

CREATE INDEX IF NOT EXISTS idx_instincts_state_conf
  ON instincts (project_id, state, confidence DESC);

CREATE TABLE IF NOT EXISTS beat_state (
  project_id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  current_act INTEGER NOT NULL CHECK (current_act BETWEEN 1 AND 3),
  current_sequence TEXT,
  current_beat TEXT,
  checkpoint_name TEXT,
  progress_pct REAL NOT NULL DEFAULT 0 CHECK (progress_pct >= 0 AND progress_pct <= 100),
  status TEXT NOT NULL CHECK (status IN ('running','blocked','completed')),
  blocked_reason TEXT,
  snapshot_json TEXT NOT NULL DEFAULT '{}' CHECK (json_valid(snapshot_json)),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE TABLE IF NOT EXISTS audit_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id TEXT NOT NULL,
  session_id TEXT REFERENCES sessions(id) ON DELETE SET NULL,
  agent_id TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  params_json TEXT NOT NULL CHECK (json_valid(params_json)),
  result_json TEXT NOT NULL CHECK (json_valid(result_json)),
  result_status TEXT NOT NULL CHECK (result_status IN ('success','error')),
  duration_ms INTEGER,
  message_id INTEGER REFERENCES messages(id) ON DELETE SET NULL,
  error_text TEXT,
  prev_hash TEXT,
  entry_hash TEXT,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE INDEX IF NOT EXISTS idx_audit_time
  ON audit_log (project_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_audit_agent_tool
  ON audit_log (project_id, agent_id, tool_name, created_at DESC);

CREATE TRIGGER IF NOT EXISTS audit_log_no_update
BEFORE UPDATE ON audit_log
BEGIN
  SELECT RAISE(ABORT, 'audit_log is append-only');
END;

CREATE TRIGGER IF NOT EXISTS audit_log_no_delete
BEFORE DELETE ON audit_log
BEGIN
  SELECT RAISE(ABORT, 'audit_log is append-only');
END;
```
