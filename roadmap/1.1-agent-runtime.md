# 1.1 Agent Runtime

> Phase 1 — Single Agent Loop (Act 1 → Act 2)

Runtime de execução de um agent especializado, com ciclo completo e auditável:
**spawn → inject context → execute → collect → cleanup**.

## 2) Escopo da etapa 1.1

### Objetivo desta etapa

Entregar um runtime de agent único, confiável e previsível, que:
1. cria um ambiente isolado de execução;
2. prepara contexto de alta sinalização;
3. executa o loop de tool use via Claude API/SDK;
4. coleta e persiste resultados no SQLite;
5. finaliza com limpeza determinística (inclusive em erro/abort).

### Fora de escopo (evitar scope creep)

- Orquestração multi-agent completa (2.2).
- Beat Sheet decomposition engine (2.1).
- Merge avançado Tier 3-4 (2.4).
- TUI Kanban (Phase 3).
- Sistema de “instincts”/aprendizado contínuo completo (Phase 4).

### Dependências de entrada

- **0.2 SQLite Schema** (mínimo: `messages`; ideal: `audit_log`).
- **0.3 WorktreeManager** (create/remove/list/getGit).
- Recomendado: 0.1 já ter baseline TS/scripts para rodar testes.

### Saídas mínimas obrigatórias

- `src/agents/types.ts`
- `src/agents/runtime.ts`
- `src/agents/context.ts`
- `src/agents/tools.ts`
- `src/agents/abort.ts` (ou equivalente dentro de `runtime.ts`)

## 3) Melhores práticas consolidadas (Web + Context7 + docs internas)

### 3.1 `@anthropic-ai/sdk` + tool use

- Preferir **tool runner** quando possível para reduzir boilerplate de loop.
- Se fizer loop manual, seguir contrato estrito:
  - `stop_reason = "tool_use"` → executar tool e devolver `tool_result`;
  - `tool_result` deve vir **imediatamente** após `tool_use`;
  - em mensagem com `tool_result`, blocos `tool_result` devem vir **antes** de texto livre;
  - tratar `pause_turn` (server tools) com continuação da conversa;
  - tratar `max_tokens` com retry controlado.
- Usar `strict: true` em tools críticas para conformidade de schema.
- Descrições de tools devem ser detalhadas (o que faz, quando usar, limites).

### 3.2 Agent SDK patterns para spawn de agents especializados

- Definir subagents via `agents` (programático) com `description` clara.
- Contexto do subagent é **isolado**; só entra o que for enviado no prompt do Agent tool.
- Restringir `tools` por papel (least privilege).
- O parent recebe apenas mensagem final do subagent (evitar poluição de contexto).
- Evitar árvore de delegação profunda (subagent não deve virar orquestrador recursivo).

### 3.3 Cancelamento com `AbortController` em Node.js

- Criar controller por execução de agent.
- Compor sinais com `AbortSignal.any([userCancel, timeoutSignal, parentSignal])`.
- Propagar `signal` para toda operação suportada:
  - chamadas HTTP/fetch para LLM/tool externa;
  - `child_process` (com opção `signal`);
  - timers promissificados e operações assíncronas internas.
- Usar listeners com `{ once: true }` para evitar leak.
- Encerrar sempre com `finally` + cleanup idempotente.

### 3.4 Context preparation + prompt construction

- Arquitetura em camadas (sistema → projeto → memória/scratchpad → tarefa).
- Princípio: **menor conjunto de tokens de alto sinal**.
- Organizar prompt em seções claras (`<background_information>`, `<instructions>`, `<tool_guidance>`, `<output_description>` ou headers markdown).
- Preferir retrieval just-in-time ao invés de despejar contexto inteiro.
- Incluir exemplos canônicos (não coleções infinitas de edge case).

### 3.5 Sandboxing de execução de tools

- Aplicar least privilege por agent/tool.
- Allowlist por tool + path + operação.
- Para shell: políticas de comando, negação explícita de padrões perigosos.
- Manter auditoria completa de requests/responses de tools.
- Se usar sandbox programático (Agent SDK):
  - `sandbox.enabled = true`;
  - restringir `network.allowedDomains`;
  - restringir `filesystem.allowWrite/denyRead`;
  - evitar `allowUnsandboxedCommands` e `bypassPermissions` fora de ambiente controlado.

### 3.6 Context7 (documentação atualizada em tempo de execução)

- Em fluxos de pesquisa técnica, usar `resolveLibraryId` + `queryDocs` para evitar docs obsoletas.
- Para integração rápida: `Context7Agent` com limite de steps.
- Em produção: combinar Context7 com cache e trilha de versão da fonte consultada.
- **Observação de escopo:** em 1.1, Context7 entra como padrão recomendado para o futuro `researcher`; não é obrigatório acoplar agora no runtime base.

## 4) Ciclo de vida completo do agent (obrigatório)

```text
Spawn
  -> create runId / validate agent config / create worktree
Inject Context
  -> load project rules + scratchpad snapshot + task payload
Execute
  -> call Claude + handle tool_use loop + publish progress
Collect
  -> persist final output + files changed + metrics + status
Cleanup
  -> stop processes + close resources + remove worktree (policy-based)
```

### Invariantes de ciclo

1. Toda execução tem `runId` único.
2. Toda transição de estado é persistida (observabilidade).
3. `cleanup` roda em `success`, `error` e `abort`.
4. Nenhum erro é silenciosamente ignorado.

### Estados sugeridos

`created -> spawning -> context_ready -> running -> collecting -> completed|failed|aborted -> cleaned`

## 5) Checklist expandido (implementação detalhada)

### 5.1 Agent definition format (TypeScript interface matching YAML frontmatter)

### O QUE é

Contrato tipado único para definir papéis de agent, modelo, tools permitidas, limites de execução e instruções base.

### O QUE NÃO é

- Não é catálogo final de todos os agents da fase 2.
- Não é sistema de skills completo.
- Não é parser de markdown genérico sem validação de domínio.

### COMO implementar

```ts
export type AgentModel = "opus" | "sonnet" | "haiku" | "inherit";

export interface AgentDefinition {
  name: string;
  role: string;
  description: string;
  model: AgentModel;
  tools: string[];
  disallowedTools?: string[];
  maxTurns?: number;
  systemPrompt: string;
}

export interface AgentRunInput {
  agent: AgentDefinition;
  taskId: string;
  taskPrompt: string;
  parentRunId?: string;
}
```

Implementar função de validação:
- nome não vazio e estável;
- tools sem duplicidade;
- `disallowedTools` prevalece sobre `tools`;
- `maxTurns` > 0;
- `systemPrompt` obrigatório.

Se houver suporte a frontmatter YAML, converter para `AgentDefinition` e validar antes de executar.

### CRITÉRIO DE ACEITAÇÃO

- Parse de definição válida produz objeto tipado.
- Definição inválida falha com erro explícito (mensagem acionável).
- Testes cobrindo conflitos de tool allow/deny.

### RISCOS

- **Drift** entre frontmatter e interface TS.  
  *Mitigação:* parser único + validação central.
- **Superpermissão de tool.**  
  *Mitigação:* denylist prioritária e testes de segurança.

### 5.2 Agent spawner: creates worktree, injects context, runs Claude API call

### O QUE é

Componente que inicia a execução de um agent com isolamento de workspace, contexto inicial e runtime controlado.

### O QUE NÃO é

- Não é orquestrador de múltiplos agents.
- Não é scheduler global.
- Não é merge manager.

### COMO implementar

Responsabilidades do spawner:
1. gerar `runId`;
2. criar worktree via `WorktreeManager.create(agentId, baseBranch)`;
3. montar `cwd` isolado;
4. preparar `AbortController` composto;
5. chamar `runAgentLifecycle()`.

```ts
export async function spawnAgent(input: AgentRunInput): Promise<AgentRunResult> {
  const runId = crypto.randomUUID();
  const worktree = await worktreeManager.create(runId, "main");
  const controller = createRunAbortController(runId);

  try {
    return await runAgentLifecycle({
      ...input,
      runId,
      cwd: worktree.path,
      signal: controller.signal,
    });
  } finally {
    await safeCleanupRun({ runId, worktreePath: worktree.path });
  }
}
```

### CRITÉRIO DE ACEITAÇÃO

- Spawner cria worktree isolado e retorna resultado tipado.
- Falha no meio da execução ainda dispara cleanup.
- Eventos de início/fim persistidos no SQLite.

### RISCOS

- **Worktree órfã em crash.**  
  *Mitigação:* cleanup em `finally` + job de varredura de órfãs.
- **Corrida em refs git compartilhadas.**  
  *Mitigação:* merge/refs protegidos por mutex no manager.

### 5.3 Context preparation: read from scratchpad, build focused prompt

### O QUE é

Pipeline determinístico para montar contexto de alta relevância (baixo ruído) antes da primeira chamada ao modelo.

### O QUE NÃO é

- Não é injetar histórico completo da sessão.
- Não é RAG genérico de todo repositório.
- Não é compactação global da conversa (isso é estratégia separada).

### COMO implementar

Camadas mínimas:
1. **System base** (instruções de comportamento/safety);
2. **Project rules** (CLAUDE.md/AGENTS.md quando habilitado);
3. **Task payload** (objetivo e critérios da tarefa);
4. **Scratchpad snapshot** (estado relevante da atividade);
5. **Current diff/contexto local** (somente se necessário).

```ts
interface PreparedContext {
  system: string;
  messages: Array<{ role: "user" | "assistant"; content: string }>;
  metadata: { tokensEstimate: number; sources: string[] };
}
```

Boas práticas:
- usar seções claras no prompt;
- limitar tamanho por orçamento de tokens;
- carregar contexto just-in-time quando possível.

### CRITÉRIO DE ACEITAÇÃO

- Contexto preparado inclui somente itens relevantes à tarefa.
- Existe estimativa de tokens antes de executar.
- Requisições equivalentes geram contexto estável/reprodutível.

### RISCOS

- **Contexto excessivo (context rot).**  
  *Mitigação:* orçamento por camada + pruning.
- **Contexto insuficiente.**  
  *Mitigação:* fallback controlado para retrieval adicional.

### 5.4 Tool execution: map agent tools to actual implementations

### O QUE é

Camada de roteamento e execução de tools permitidas para o agent, com validação de schema, permissão e auditoria.

### O QUE NÃO é

- Não é execução irrestrita de shell.
- Não é acoplamento direto tool-call -> comando sem validação.
- Não é ignorar `stop_reason`/contrato de tool_result do Claude.

### COMO implementar

1. `ToolRegistry` com handlers tipados.
2. Filtro por allowlist/denylist do agent.
3. Execução de tool via SDK runner (preferencial) ou loop manual.
4. Para loop manual:
   - detectar `tool_use`;
   - executar handler;
   - devolver `tool_result` imediatamente e no formato correto.

```ts
type ToolHandler = (input: unknown, ctx: ToolExecutionContext) => Promise<ToolResult>;

interface ToolExecutionContext {
  runId: string;
  cwd: string;
  signal: AbortSignal;
  agent: AgentDefinition;
}
```

Com `@anthropic-ai/sdk`, preferir:
- helpers `betaZodTool` / `betaTool`;
- `toolRunner` para loop automático;
- `ToolError` para erro estruturado em tool result.

### CRITÉRIO DE ACEITAÇÃO

- Tool fora da allowlist é negada com erro explícito.
- Tool válida executa e retorna resultado no formato esperado.
- Loop encerra corretamente quando não há mais `tool_use`.

### RISCOS

- **Injeção/comando perigoso.**  
  *Mitigação:* validação de input + sandbox + denylist.
- **Quebra de contrato do tool_result (400 da API).**  
  *Mitigação:* testes de serialização e ordem de blocos.

### 5.5 Result collection: capture output, write to SQLite messages

### O QUE é

Persistência estruturada do resultado da execução do agent, com status, payload e metadados de execução.

### O QUE NÃO é

- Não é logging ad-hoc em arquivo solto.
- Não é apenas armazenar texto final sem estado.
- Não é depender de memória em processo para status.

### COMO implementar

Publicar mensagens tipadas no SQLite (`messages`), por exemplo:
- `task_progress` (heartbeat/etapas);
- `task_done` (resultado final);
- `escalation` (bloqueio/erro sem recuperação);
- opcional: `task_failed`, `task_aborted`.

Payload mínimo:
- `runId`, `taskId`, `agentName`;
- `status`, `summary`, `artifacts`;
- `usage` (tokens/custo), `durationMs`;
- erro estruturado quando aplicável.

### CRITÉRIO DE ACEITAÇÃO

- Toda execução gera pelo menos 1 mensagem de progresso + 1 final.
- Em erro, grava `escalation` ou `task_failed` com causa raiz.
- Dados suficientes para montar timeline de execução.

### RISCOS

- **Perda de observabilidade em falha parcial.**  
  *Mitigação:* persistir evento antes/depois de pontos críticos.
- **Schema inconsistente entre agentes.**  
  *Mitigação:* payload versionado e tipado.

### 5.6 Cleanup: remove worktree on completion or abort

### O QUE é

Desmontagem segura de recursos da execução: processos, handles, worktree, objetos em memória e locks.

### O QUE NÃO é

- Não é cleanup “best effort” sem garantia.
- Não é remover branch/artefato sem política definida.
- Não é ignorar cleanup em abort.

### COMO implementar

`safeCleanupRun()` idempotente:
1. cancelar subprocessos vivos ligados ao run;
2. flush final de mensagens;
3. remover worktree;
4. liberar locks e referências em memória;
5. registrar evento de cleanup concluído.

Política recomendada:
- sucesso: remover worktree;
- erro/abort: remover worktree, mantendo evidência (logs/sqlite);
- debug mode opcional: preservar worktree mediante flag explícita.

### CRITÉRIO DE ACEITAÇÃO

- Cleanup executa em sucesso/erro/abort.
- Repetir cleanup não gera falha (idempotência).
- Não restam worktrees órfãs após execução normal.

### RISCOS

- **Falha de remoção de worktree** por lock/processo ativo.  
  *Mitigação:* retry com backoff + diagnóstico explícito.
- **Perda de evidência de falha.**  
  *Mitigação:* persistir resumo antes de remover workspace.

### 5.7 Abort support via AbortController

### O QUE é

Mecanismo de cancelamento cooperativo e propagado ponta a ponta, com encerramento previsível e seguro.

### O QUE NÃO é

- Não é `throw` isolado sem propagar sinal.
- Não é matar processo sem cleanup.
- Não é silêncio em cancelamento.

### COMO implementar

1. mapear controllers por `runId`;
2. compor sinais (`AbortSignal.any`) para:
   - cancelamento do usuário;
   - timeout de execução;
   - cancelamento de pai/orquestrador;
3. checar `signal.aborted` em pontos longos e chamar `throwIfAborted()`;
4. garantir listeners com `{ once: true }`.

```ts
function composeRunSignal(input: {
  userSignal?: AbortSignal;
  timeoutMs: number;
  parentSignal?: AbortSignal;
}): AbortSignal {
  const timeoutSignal = AbortSignal.timeout(input.timeoutMs);
  return AbortSignal.any(
    [input.userSignal, input.parentSignal, timeoutSignal].filter(Boolean) as AbortSignal[],
  );
}
```

### CRITÉRIO DE ACEITAÇÃO

- Cancelamento do usuário interrompe execução em tempo aceitável.
- Timeout encerra run com status `aborted` e reason explícita.
- Nenhum processo filho fica órfão após abort.

### RISCOS

- **Memory leak por listeners não removidos.**  
  *Mitigação:* listeners `once` e limpeza explícita.
- **Abort parcial (só LLM cancela, tool segue rodando).**  
  *Mitigação:* propagar `signal` a todo I/O e subprocesso.

## 6) Definition of Done (global da etapa 1.1)

A etapa 1.1 está pronta quando:

1. Existe execução end-to-end de um agent único com ciclo completo.
2. O ciclo suporta sucesso, erro e cancelamento com cleanup consistente.
3. Tool use funciona com contrato correto (`tool_use`/`tool_result`) e segurança mínima por allowlist.
4. Contexto é preparado por camadas, com orçamento de tokens e sem dump irrestrito.
5. Resultado da execução é persistido no SQLite com rastreabilidade suficiente.
6. Testes cobrem contratos de tipos, fluxo principal, falhas e abort.
