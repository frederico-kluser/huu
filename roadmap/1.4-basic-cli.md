# 1.4 Basic CLI

> Phase 1 — Single Agent Loop (Act 1 → Act 2)

Interface de linha de comando mínima antes da TUI existir. Deve ser suficiente para disparar e monitorar o loop completo de **um único agent** (task → worktree → implementação → merge).

## Resultado da revisão (1.4 vs ARCHITECTURE.md)

### Estado atual

A versão anterior desta etapa estava **incompleta** para implementação prática (apenas checklist curto). Faltavam:

- decisão explícita de framework CLI;
- padrão de configuração de `bin` no `package.json` para TypeScript;
- contrato de argumentos, saída, códigos de erro e UX de help;
- definição clara de quais estados o `status` deve mostrar para validar o pipeline.

### Coerência arquitetural exigida

Esta etapa precisa respeitar as decisões do `ARCHITECTURE.md`:

- **Decisão 11 (Node.js + TypeScript)**: CLI deve ser nativa do runtime Node+TS.
- **Decisão 3 (SQLite WAL / mail system)**: `status` deve refletir estado derivado de mensagens tipadas (ex.: `task_progress`, `task_done`, `merge_result`).
- **Decisão 2 + 4 (worktrees + merge por tiers)**: `run` precisa acionar runtime+merge do single-agent loop, sem pular etapas.
- **Decisão 10 (TUI vem depois)**: output estruturado em terminal, sem componentes Ink nesta fase.

## Pesquisa aplicada (web search + Context7)

### Frameworks CLI (Node.js/TS)

- **Commander**: maduro, documentação ampla, help/erro consistentes, bom para subcomandos simples; recomendado para 1.4 pela estabilidade.
- **Yargs**: parser muito poderoso para CLIs grandes e altamente configuráveis; maior complexidade e superfície de API.
- **Citty**: leve/zero-dependency, API moderna e async-friendly; ecossistema menor que Commander/Yargs.
- **Cleye**: tipagem forte e API enxuta; ótimo para ergonomia TS, mas menos consolidado para CLIs maiores.

**Decisão recomendada para 1.4:** usar **Commander** como default por maturidade e previsibilidade.  
**Alternativa válida:** Citty se a prioridade for footprint mínimo desde o início.

### `bin` entry point em package.json (TypeScript)

Boas práticas aplicáveis:

- configurar `bin` apontando para executável real (ex.: `bin/huu`);
- garantir shebang `#!/usr/bin/env node` no arquivo executável;
- compilar TS para `dist/` e manter o bin como ponte para `dist/cli.js`;
- garantir inclusão de `bin` e `dist` no pacote (`files`) para publicação.

Exemplo recomendado:

```json
{
  "bin": {
    "huu": "bin/huu"
  },
  "files": ["dist", "bin/huu"],
  "scripts": {
    "build": "tsc -p tsconfig.json",
    "dev:cli": "tsx src/cli.ts"
  }
}
```

Exemplo de `bin/huu`:

```bash
#!/usr/bin/env node
node "$(dirname "$0")/../dist/cli.js" "$@"
```

### Structured output com Chalk/Picocolors

- Preferir **picocolors** nesta etapa por simplicidade, baixo overhead e suporte fácil a `NO_COLOR`.
- Usar semântica fixa de cores + ícones/símbolos (não depender só de cor).
- Respeitar `NO_COLOR`, `--no-color`, e saída sem ANSI em ambientes sem suporte.

### Padrões de parsing de argumentos

- comando/subcomando explícito (`huu run`, `huu status`);
- help automático (`-h`, `--help`) em comando raiz e subcomandos;
- erro claro para opção desconhecida e uso inválido;
- códigos de saída previsíveis (0 sucesso, 1 falha operacional, 2 erro de uso);
- suporte ao separador `--` quando necessário para repasse futuro de argumentos.

## Detalhamento implementável por item do checklist

### 1) `huu run "task description"`

**O QUE é**

- Comando principal para validar o loop fase 1:
  1. receber tarefa textual;
  2. acionar Agent Runtime (1.1) com Builder Agent (1.2);
  3. acompanhar execução;
  4. acionar Merge Workflow Tier 1-2 (1.3);
  5. reportar sucesso/falha no terminal.

**O QUE NÃO é**

- Não é orquestração multi-agent (fase 2).
- Não é TUI/Kanban (fase 3).
- Não é decomposição em Beat Sheet (2.1).
- Não é command set final do produto (`init`, `config`, `dry-run`, etc.).

**COMO implementar**

- Definir subcomando com 1 argumento posicional obrigatório: `<taskDescription>`.
- Validar entrada não vazia e tamanho mínimo útil (ex.: `trim().length >= 5`).
- Chamar um serviço de alto nível (ex.: `runSingleAgentTask`) que encapsula runtime + merge.
- Stream de eventos para console usando logger estruturado (tipo de evento + mensagem + metadados resumidos).
- Persistir e exibir um `runId` para correlação com `status`.
- Retornar exit code:
  - `0`: execução e merge concluídos;
  - `1`: erro operacional (runtime/merge/sqlite);
  - `2`: erro de uso (argumentos inválidos).

Snippet de referência:

```ts
program
  .command("run")
  .argument("<taskDescription>")
  .description("Execute one builder agent end-to-end")
  .action(async (taskDescription) => {
    const result = await runSingleAgentTask({ taskDescription });
    process.exitCode = result.ok ? 0 : 1;
  });
```

**CRITÉRIO DE ACEITAÇÃO**

- `huu run "implementar X"` executa sem intervenção manual.
- Durante execução, há feedback de progresso legível.
- Ao final, o comando informa claramente success/failure e `runId`.
- Falhas reais retornam código diferente de zero.

**RISCOS**

- *Risco:* comando dispara runtime mas não mergeia resultado.  
  *Mitigação:* exigir estado final explícito de merge no resumo de saída.
- *Risco:* exceções assíncronas silenciosas.  
  *Mitigação:* handler central de erro + `process.exitCode = 1`.
- *Risco:* tarefa vazia causa comportamento imprevisível.  
  *Mitigação:* validação rígida de argumento na borda do CLI.

### 2) `huu status`

**O QUE é**

- Comando de leitura que mostra um snapshot do estado atual/mais recente do loop single-agent com base no banco SQLite.

**O QUE NÃO é**

- Não é dashboard em tempo real com navegação (isso é TUI).
- Não é histórico analítico completo de sessões.
- Não substitui logs detalhados por arquivo.

**COMO implementar**

- Consultar mensagens mais recentes por tipo (`task_assigned`, `task_progress`, `task_done`, `merge_result`, `escalation`).
- Resolver estado agregado amigável, por exemplo:
  - `idle`, `running`, `merge_pending`, `merged`, `failed`, `escalated`.
- Exibir no mínimo:
  - `runId`;
  - status agregado;
  - agent atual;
  - timestamp do último evento;
  - resumo de merge (quando houver).
- Se não houver execução prévia, mostrar mensagem explícita de "nenhuma execução registrada".

Snippet de referência:

```ts
program
  .command("status")
  .description("Show current single-agent execution status")
  .action(async () => {
    const snapshot = await readSingleAgentStatus();
    renderStatus(snapshot);
  });
```

**CRITÉRIO DE ACEITAÇÃO**

- `huu status` funciona antes e depois de `huu run`.
- Durante execução ativa, mostra estado `running` e último progresso.
- Após merge bem-sucedido, mostra estado final `merged`.
- Em falha/escalation, mostra estado final correspondente e mensagem resumida.

**RISCOS**

- *Risco:* estado inconsistente por múltiplos eventos concorrentes.  
  *Mitigação:* definir regra determinística de precedência de estados.
- *Risco:* status sem contexto útil (apenas "running").  
  *Mitigação:* sempre mostrar timestamp + último evento + runId.
- *Risco:* erro de DB derruba UX sem explicação.  
  *Mitigação:* mensagem de erro acionável (ex.: caminho do DB, migração pendente).

### 3) Structured console output

**O QUE é**

- Camada única de apresentação textual para eventos da CLI com formato consistente, cores semânticas e fallback para ambientes sem cor.

**O QUE NÃO é**

- Não é UI interativa com componentes React Ink.
- Não é sistema completo de logging distribuído.
- Não é personalização avançada de tema nesta etapa.

**COMO implementar**

- Criar módulo dedicado (ex.: `src/cli/output.ts`) com funções como:
  - `printInfo`, `printSuccess`, `printWarn`, `printError`, `printEvent`.
- Mapear tipo de evento -> símbolo + cor + prefixo textual.
- Adotar `picocolors` (ou `chalk` se já padrão do projeto), garantindo:
  - respeito a `NO_COLOR`;
  - legibilidade em CI;
  - informação redundante semântica (ex.: `[ERROR]`, não só cor).
- Estruturar linhas para fácil leitura e troubleshooting:
  - `[timestamp] [runId] [event] mensagem`.

**CRITÉRIO DE ACEITAÇÃO**

- Saída de `run` e `status` usa o mesmo padrão visual.
- Em ambiente sem cor, mensagens continuam legíveis e completas.
- Erros e sucesso são diferenciáveis por texto (mesmo sem ANSI).

**RISCOS**

- *Risco:* excesso de cor/poluição visual.  
  *Mitigação:* limitar paleta e manter uma linha por evento relevante.
- *Risco:* dependência de ANSI quebra logs em CI.  
  *Mitigação:* fallback plain text + respeito a `NO_COLOR`.
- *Risco:* formato instável entre comandos.  
  *Mitigação:* centralizar renderização em um único módulo.

### 4) Entry point e `bin` configuration

**O QUE é**

- Estrutura de inicialização da CLI para execução local, `npm link`, e publicação futura.

**O QUE NÃO é**

- Não é empacotamento standalone (pkg, nexe, etc.).
- Não é estratégia de distribuição multi-plataforma avançada.

**COMO implementar**

- `src/cli.ts`: define parser, comandos e bootstrap.
- `bin/huu`: wrapper executável com shebang e chamada ao build em `dist/`.
- `package.json#bin`: mapear `huu` para `bin/huu`.
- Garantir permissões executáveis em `bin/huu` e inclusão no publish (`files`).
- Manter caminhos relativos portáveis (sem caminhos absolutos locais).

**CRITÉRIO DE ACEITAÇÃO**

- Após `npm run build && npm link`, comando `huu --help` funciona globalmente.
- `huu run ...` e `huu status` funcionam via bin real (não só `tsx src/cli.ts`).
- Script funciona em macOS/Linux e tem comportamento equivalente em Windows via shim do npm.

**RISCOS**

- *Risco:* bin aponta para arquivo TS não compilado.  
  *Mitigação:* apontar sempre para wrapper JS executável.
- *Risco:* ausência de shebang impede execução direta.  
  *Mitigação:* teste automatizado simples de execução do bin.
- *Risco:* build gera ESM/CJS incompatível com wrapper.  
  *Mitigação:* padronizar módulo alvo e validar no `--help` pós-build.

## Os comandos mínimos (`run`, `status`) são suficientes?

**Sim, para validação da fase 1, desde que:**

- `run` execute o pipeline completo (runtime + merge), não apenas spawner;
- `status` leia estado real persistido no SQLite e reflita progresso/finalização;
- ambos usem códigos de saída previsíveis e mensagens acionáveis.

Com isso, é possível validar o **single agent loop** sem TUI e sem comandos extras.  
Comandos adicionais (`logs`, `init`, `config`, `dry-run`) ficam explicitamente para fases posteriores.
