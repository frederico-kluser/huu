# 1.2 Builder Agent (first agent)

> Phase 1 — Single Agent Loop (Act 1 → Act 2)

Primeiro agente de implementação real do sistema. Esta etapa deve provar, de ponta a ponta, que o runtime consegue: receber uma subtarefa, abrir worktree isolada, gerar/editar código, validar, commitar e sinalizar conclusão no mail system (SQLite).

## Status de completude e coerência com `ARCHITECTURE.md`

**Situação atual:** etapa **ainda não concluída**.

### Verificação de coerência arquitetural

| Requisito arquitetural | Estado esperado | Situação atual | Ação nesta etapa |
|---|---|---|---|
| Builder usa **Sonnet** | Builder no tier de implementação com custo/qualidade balanceado | Não implementado | Definir `modelTier: "sonnet"` na definição do agente |
| Builder usa ferramentas **Read/Write/Edit/Bash** | Implementação de código com edição + execução de comandos | Não implementado | Definir allowlist explícita e enforcement no runtime |
| Isolamento por **git worktree** | 1 worktree por agent, branch própria | Não implementado | E2E deve validar criação e limpeza de worktree |
| Comunicação por **SQLite messages** | `task_progress` e `task_done` com payload estruturado | Não implementado | E2E deve validar escrita de mensagens |
| Merge progressivo (fases separadas) | Builder só entrega commit/merge_ready, não resolve conflitos avançados | Não implementado | Manter escopo restrito ao commit + sinalização |

**Conclusão:** o checklist original estava correto no direcionamento, mas insuficiente em detalhe operacional. Este documento expande a etapa para implementação executável sem ambiguidades.

## Checklist detalhado (escopo implementável)

### 1) Builder agent definition (Sonnet, Read/Write/Edit/Bash)

#### O QUE é

A especificação formal do agente `builder`, incluindo identidade, modelo, ferramentas permitidas, prompt de sistema orientado à implementação e critérios de término. É o contrato entre o Orchestrator/Runtime e o agente executor de código.

#### O QUE NÃO é

- Não é o loop completo do orquestrador multi-agent (isso é 2.2).
- Não é sistema de merge Tier 3-4 (isso é 2.4).
- Não é agente de planejamento/review (papéis de `planner` e `reviewer`).
- Não é abertura indiscriminada de ferramentas além do mínimo necessário.

#### COMO implementar

1. Criar definição tipada em `src/agents/definitions/builder.ts`.
2. Fixar tier de modelo como Sonnet (resolução de modelo concreto fica no runtime/config).
3. Definir allowlist **estrita** de tools: `Read`, `Write`, `Edit`, `Bash`.
4. Usar prompt de sistema com foco em execução (não apenas sugestão), mudanças mínimas e verificáveis.
5. Definir critério explícito de done: commit criado + `task_done` emitido + resumo de mudanças.

Exemplo de estrutura:

```ts
export const builderAgent: AgentDefinition = {
  id: 'builder',
  role: 'implementation',
  modelTier: 'sonnet',
  tools: ['Read', 'Write', 'Edit', 'Bash'],
  systemPrompt: BUILDER_SYSTEM_PROMPT,
  stopConditions: {
    requireCommit: true,
    requireTaskDoneMessage: true
  }
};
```

Template de prompt recomendado (implementation-focused):

```xml
<role>
Você é o Builder Agent. Sua função é implementar código funcional e validado.
</role>

<goal>
Entregar a subtarefa com mudanças mínimas, corretas e commitáveis.
</goal>

<constraints>
- Use apenas as ferramentas permitidas.
- Não faça refactors amplos fora do escopo.
- Não "descreva o que faria": implemente de fato, salvo bloqueio real.
- Antes de concluir, valide build/testes relevantes da área alterada.
</constraints>

<execution_flow>
1) Ler contexto e arquivos alvo.
2) Planejar mudanças curtas e objetivas.
3) Editar arquivos necessários.
4) Rodar validações relevantes.
5) Gerar resumo de arquivos alterados.
6) Criar commit e sinalizar task_done.
</execution_flow>

<done_contract>
Só concluir quando houver commit válido e payload de conclusão com changed_files.
</done_contract>
```

#### CRITÉRIO DE ACEITAÇÃO

- Definição `builder` existe com `modelTier: sonnet`.
- Allowlist contém exatamente `Read/Write/Edit/Bash` nesta fase.
- Prompt inclui: papel, restrições, fluxo de execução e contrato de término.
- Runtime rejeita tentativa de tool fora da allowlist.

#### RISCOS

- **Risco:** prompt genérico demais e agente só “aconselha”.  
  **Mitigação:** instrução explícita de execução + critério de done baseado em commit.
- **Risco:** over-engineering em tarefas simples.  
  **Mitigação:** regra de mudanças mínimas e estritamente no escopo.
- **Risco:** toolset insuficiente para descoberta de contexto.  
  **Mitigação:** runtime deve fornecer contexto inicial (arquivos-alvo) vindo do planner/runtime.

### 2) End-to-end test: receive subtask → create worktree → implement → commit → signal done

#### O QUE é

Teste de integração real (não mock superficial) que valida o pipeline completo do builder dentro de uma worktree isolada, incluindo artefatos git e mensagens no SQLite.

#### O QUE NÃO é

- Não é teste do loop completo com múltiplos agentes.
- Não é teste de UI/TUI.
- Não é benchmark de custo/token.

#### COMO implementar

1. Subir repositório temporário em teste (fixture git real).
2. Criar subtarefa de entrada e iniciar runtime com agente `builder`.
3. Validar criação de branch/worktree isolada.
4. Simular execução do agente com edição real de arquivo de fixture.
5. Validar commit criado na branch da worktree.
6. Validar mensagens `task_progress` e `task_done` no SQLite.
7. Validar cleanup (worktree removida ao final, conforme política da etapa).

Exemplo de cenário de teste (vitest):

```ts
it('builder completes task end-to-end in isolated worktree', async () => {
  const repo = await createTempGitRepo();
  const runtime = createAgentRuntime({ repoPath: repo.path });

  const result = await runtime.run({
    agentId: 'builder',
    task: 'Add function sum(a,b) and tests'
  });

  expect(result.status).toBe('done');
  expect(result.commitSha).toMatch(/^[0-9a-f]{7,40}$/);
  expect(result.changedFiles.added.length + result.changedFiles.modified.length).toBeGreaterThan(0);

  const doneMessage = await db.getLastMessage('task_done', result.taskId);
  expect(doneMessage).toBeTruthy();
});
```

#### CRITÉRIO DE ACEITAÇÃO

- Teste E2E passa com repositório git real (não fake de filesystem).
- O teste prova os 5 passos: subtarefa, worktree, implementação, commit, sinalização.
- Há validação explícita de `task_done` e `commitSha`.
- Não há mudanças no branch principal durante execução do builder.

#### RISCOS

- **Risco:** falso positivo por mocks excessivos.  
  **Mitigação:** usar fixture de git real e assertions em objetos git reais.
- **Risco:** flakiness por dependência de ambiente.  
  **Mitigação:** fixtures determinísticas, diretórios temporários isolados e teardown robusto.
- **Risco:** cleanup incompleto de worktree.  
  **Mitigação:** `afterEach` com remoção forçada + verificação de `worktree list`.

### 3) File change tracking (created/modified/deleted)

#### O QUE é

Mecanismo de rastreamento estruturado das mudanças produzidas pelo builder, com classificação por tipo (added/modified/deleted/renamed) e persistência junto do resultado da tarefa.

#### O QUE NÃO é

- Não é sistema de auditoria completo multi-sessão (detalhado em 4.3).
- Não é diff semântico por IA.
- Não substitui revisão humana em mudanças críticas.

#### COMO implementar

1. Coletar estado do git no final da execução (pré-commit e/ou pós-commit).
2. Usar comandos parseáveis para automação:
   - `git status --porcelain=v2 -z` (estado do working tree/index)
   - `git diff --name-status --find-renames -z` (mudanças não commitadas)
   - `git diff-tree --no-commit-id --name-status -r <commitSha>` (mudanças do commit final)
3. Normalizar resultado para um payload estável:

```ts
type FileChangeSummary = {
  added: string[];
  modified: string[];
  deleted: string[];
  renamed: Array<{ from: string; to: string }>;
};
```

4. Persistir `changed_files` no payload de `task_done` e, quando aplicável, em `audit_log`.

Exemplo de payload final:

```json
{
  "taskId": "task-123",
  "agentId": "builder",
  "commitSha": "abc1234",
  "changed_files": {
    "added": ["src/math/sum.ts"],
    "modified": ["src/math/index.ts"],
    "deleted": [],
    "renamed": []
  }
}
```

#### CRITÉRIO DE ACEITAÇÃO

- `task_done` inclui `commitSha` e `changed_files`.
- Mudanças `A/M/D/R` são classificadas corretamente em testes.
- Renames não aparecem como delete+add quando detectáveis (`--find-renames`).
- Estrutura de payload é estável (consumível por CLI/TUI futuras).

#### RISCOS

- **Risco:** perda de precisão em rename detection.  
  **Mitigação:** habilitar `--find-renames` e cobrir com teste específico.
- **Risco:** parsing frágil por espaços/UTF-8 em caminho.  
  **Mitigação:** formato `-z` e parser por NUL.
- **Risco:** divergência entre estado em memória e estado git real.  
  **Mitigação:** derivar resumo do git ao final, não de estado interno.

## Sequência recomendada de implementação

1. Implementar definição do builder + prompt.
2. Implementar rastreamento de mudanças baseado em git.
3. Implementar integração com `task_done`.
4. Criar teste E2E completo e estabilizar.

## Definition of Done (global da etapa 1.2)

- Builder executa subtarefa real em worktree isolada.
- Builder gera commit válido.
- Builder emite `task_done` com `commitSha` e `changed_files`.
- Teste E2E passa de forma determinística.
