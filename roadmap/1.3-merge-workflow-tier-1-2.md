## Fluxo técnico de referência (end-to-end)

1. Agent envia evento `merge_ready` com `source_branch`, `source_head_sha`, `target_branch`.
2. Orquestrador faz `enqueue` em `merge_queue` com ordem FIFO (`enqueued_at`, `id`).
3. Merge worker adquire próximo item com transação `BEGIN IMMEDIATE` (claim atômico + lease).
4. Worker valida invariantes:
   - branch origem ainda existe;
   - `source_head_sha` atual = `source_head_sha` enfileirado;
   - branch alvo existe (normalmente `main`).
5. Worker roda pré-checagem com `git merge-tree --write-tree`.
6. Worker decide tier:
   - fast-forward possível -> Tier 1;
   - sem fast-forward + sem conflito -> Tier 2;
   - conflito detectado -> falha controlada (sem tentar Tier 3/4 nesta etapa).
7. Worker executa merge real, registra resultado em `merge_results`, atualiza status da fila.
8. Em erro: registra falha com contexto e libera item (requeue ou failed, conforme política de retry).

## Item 1 — FIFO merge queue (SQLite-backed)

### O QUE é

Uma fila persistente, ordenada por chegada, para serializar merges em refs compartilhadas (`main`), garantindo:

- ordem determinística;
- recuperação após crash;
- prevenção de corrida entre múltiplos pedidos concorrentes.

### O QUE NÃO é

- Não é um broker distribuído (Kafka/Rabbit).
- Não é paralelização de merge em `main`.
- Não é scheduler de prioridade nesta fase (somente FIFO).

### COMO implementar

1. **Schema mínimo recomendado** (ajustar para convenções de 0.2):

```sql
CREATE TABLE IF NOT EXISTS merge_queue (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  request_id TEXT NOT NULL UNIQUE,
  source_branch TEXT NOT NULL,
  source_head_sha TEXT NOT NULL,
  target_branch TEXT NOT NULL DEFAULT 'main',
  status TEXT NOT NULL CHECK (status IN ('queued','in_progress','merged','conflict','failed','retry_wait')),
  attempts INTEGER NOT NULL DEFAULT 0,
  lease_owner TEXT,
  lease_expires_at TEXT,
  available_at TEXT NOT NULL DEFAULT (datetime('now')),
  started_at TEXT,
  finished_at TEXT,
  last_error TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_merge_queue_ready
  ON merge_queue(status, available_at, created_at, id);
```

2. **Claim atômico FIFO** com `BEGIN IMMEDIATE`:

```sql
BEGIN IMMEDIATE;

WITH next_item AS (
  SELECT id
  FROM merge_queue
  WHERE status IN ('queued','retry_wait')
    AND available_at <= datetime('now')
  ORDER BY created_at ASC, id ASC
  LIMIT 1
)
UPDATE merge_queue
SET status = 'in_progress',
    lease_owner = :worker_id,
    lease_expires_at = datetime('now', '+120 seconds'),
    attempts = attempts + 1,
    started_at = COALESCE(started_at, datetime('now')),
    updated_at = datetime('now')
WHERE id = (SELECT id FROM next_item);

COMMIT;
```

3. **Lease/heartbeat**:
   - enquanto merge roda, atualizar `lease_expires_at` periodicamente;
   - no startup do worker, reprocessar leases vencidos (`status='in_progress' AND lease_expires_at < now`).

4. **Pragmas de concorrência** na conexão:
   - `PRAGMA journal_mode=WAL;`
   - `PRAGMA busy_timeout=5000;`

5. **Idempotência**:
   - nunca confiar apenas no `source_branch`;
   - validar `source_head_sha` enfileirado antes de merge real.

### CRITÉRIO DE ACEITAÇÃO

- Dois requests simultâneos entram como `queued` e são processados estritamente na ordem de `created_at,id`.
- Com dois workers ativos, apenas um item fica `in_progress` por vez para o mesmo repositório principal.
- Se o worker morrer no meio, item é recuperado por lease vencido sem duplicar merge.

### RISCOS

- **Risco:** starvation por item com falha infinita.  
  **Mitigação:** `attempts` + `retry_wait` com backoff + limite de tentativas.
- **Risco:** lock contention (`SQLITE_BUSY`).  
  **Mitigação:** `busy_timeout`, transações curtas, heartbeat simples.
- **Risco:** branch mutado após enqueue.  
  **Mitigação:** validar SHA original e marcar `failed` com razão explícita.

## Item 2 — Tier 1: fast-forward detection and execution

### O QUE é

Caminho mais barato: atualizar ponteiro da branch alvo quando ela é ancestral da branch origem (sem merge commit).

### O QUE NÃO é

- Não cria merge commit.
- Não faz squash.
- Não tenta resolver conflito.

### COMO implementar

1. Ler SHAs atuais:

```ts
const targetSha = (await git.revparse([targetBranch])).trim();
const sourceSha = (await git.revparse([sourceBranch])).trim();
```

2. Detectar fast-forward sem depender de exit-code:

```ts
const baseSha = (await git.raw(['merge-base', targetSha, sourceSha])).trim();
const canFastForward = baseSha === targetSha;
const alreadyMerged = baseSha === sourceSha;
```

3. Executar Tier 1 apenas quando `canFastForward`:

```ts
await git.merge(['--ff-only', sourceBranch]);
```

4. Se `alreadyMerged`, tratar como **no-op idempotente** (status `merged` com `merge_mode='noop_already_merged'`).

### CRITÉRIO DE ACEITAÇÃO

- Quando `target` é ancestral de `source`, merge termina com `--ff-only` sem merge commit.
- Quando `source` já está contido em `target`, pipeline não falha; registra no-op.
- Quando não é fast-forward, fluxo passa para decisão de Tier 2.

### RISCOS

- **Risco:** falsa decisão por SHA desatualizado.  
  **Mitigação:** recalcular `targetSha/sourceSha` imediatamente antes do merge real.
- **Risco:** branch avançar entre detecção e execução (TOCTOU).  
  **Mitigação:** detectar erro de `--ff-only`, revalidar precondições e requeue controlado.

## Item 3 — Tier 2: recursive merge com detecção automática de conflito

### O QUE é

Merge de três vias quando fast-forward não é possível, aceitando apenas merge mecânico limpo (sem intervenção humana).

### O QUE NÃO é

- Não faz heurística `ours/theirs` nesta etapa.
- Não executa AI resolver.
- Não deixa repositório em estado de conflito pendente.

### COMO implementar

1. Pré-condição: pre-merge indica possibilidade de merge sem conflito.
2. Executar merge explícito:

```ts
await git.merge(['--no-ff', '--no-edit', '-s', 'ort', sourceBranch]);
```

3. Em falha de merge (corrida ou conflito real), capturar `GitResponseError` e sempre limpar:

```ts
try {
  await git.merge(['--no-ff', '--no-edit', '-s', 'ort', sourceBranch]);
} catch (err) {
  const abortError = await git.merge(['--abort']).then(() => null).catch((abortErr) => abortErr);
  if (abortError) {
    throw new Error(`Merge falhou e abort também falhou: ${String(abortError)}`);
  }
  throw err;
}
```

4. Se conflito ocorrer, marcar fila como `conflict` e registrar arquivos conflitados disponíveis em `err.git.conflicts`.
5. Parsing de resultado (`simple-git`):
   - sucesso: usar `mergeResult.merges`, `mergeResult.summary`, `mergeResult.result`;
   - conflito: ler `GitResponseError<MergeResult>.git.conflicts` e `git.result` para log estruturado.

### CRITÉRIO DE ACEITAÇÃO

- Em divergência sem conflito, merge commit é criado e registrado.
- Em conflito, merge é abortado automaticamente e workspace volta a estado limpo.
- Nenhum item em conflito fica bloqueando a fila indefinidamente.

### RISCOS

- **Risco:** pre-check diz “limpo”, merge real conflita por mudança concorrente.  
  **Mitigação:** rodar pre-check imediatamente antes do merge; em mismatch, abort + retry.
- **Risco:** `merge --abort` falhar por estado inconsistente.  
  **Mitigação:** registrar erro com detalhe máximo, marcar item como `failed` e exigir recuperação explícita do worker de integração.

## Item 4 — Pre-merge detection via `git merge-tree --write-tree`

### O QUE é

Simulação de merge sem tocar index/working tree para prever conflito antes de `git merge`.

### O QUE NÃO é

- Não substitui totalmente o merge real (há janela de corrida).
- Não deve depender de parsing de mensagens humanas como única fonte de verdade.
- Não deve varrer árvore inteira para detectar conflito.

### COMO implementar

1. Pré-requisito: garantir **Git >= 2.38** no ambiente de execução.
2. Comando base:

```bash
git merge-tree --write-tree --name-only --no-messages <target> <source>
```

3. Regras de interpretação (best practices):
   - `exit 0`: merge limpo;
   - `exit 1`: merge com conflito;
   - `exit != 0/1`: erro operacional.

4. Importante do manual Git:
   - preferir status de saída para clean vs conflict;
   - não inferir conflito só por ausência/presença de arquivos listados;
   - não parsear mensagens livres para lógica crítica.

5. Integração com Node:
   - usar helper que preserve `exitCode` (ex.: `spawn/execFile`) para classificação robusta;
   - manter `simple-git` para merge/pull/parsers estruturados.

### CRITÉRIO DE ACEITAÇÃO

- Pré-checagem não altera working tree nem index.
- Pipeline distingue corretamente `clean`, `conflict` e `fatal`.
- Em `conflict`, Tier 1/Tier 2 **não** executam merge real.

### RISCOS

- **Risco:** wrapper ocultar exit code (dependendo da API usada).  
  **Mitigação:** encapsular chamada em helper que exponha `code/stdout/stderr`.
- **Risco:** confiar em output textual instável entre versões de Git.  
  **Mitigação:** lógica principal baseada em exit status.

## Item 5 — Merge result logging to SQLite

### O QUE é

Persistir auditoria completa da decisão e execução do merge para observabilidade, debug e retomada segura.

### O QUE NÃO é

- Não é log textual solto em arquivo.
- Não é armazenamento de diff completo do repositório.
- Não deve omitir falhas (incluindo falhas de infraestrutura).

### COMO implementar

Campos mínimos em `merge_results` (ou equivalente):

- `request_id`, `queue_id`
- `source_branch`, `source_head_sha`
- `target_branch`, `target_head_before`, `target_head_after`
- `premerge_status` (`clean|conflict|fatal`)
- `tier_selected` (`tier1|tier2|none`)
- `merge_mode` (`ff-only|no-ff-ort|noop_already_merged`)
- `outcome` (`merged|conflict|failed`)
- `conflicts_json` (lista estruturada quando existir)
- `error_code`, `error_message`
- `started_at`, `finished_at`, `duration_ms`, `attempt`

Exemplo de insert:

```sql
INSERT INTO merge_results (
  request_id, queue_id, source_branch, source_head_sha, target_branch,
  target_head_before, target_head_after, premerge_status, tier_selected,
  merge_mode, outcome, conflicts_json, error_code, error_message,
  started_at, finished_at, duration_ms, attempt
) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?);
```

### CRITÉRIO DE ACEITAÇÃO

- 100% dos merges tentados (sucesso e falha) deixam rastro em SQLite.
- É possível reconstruir cronologia de um merge só consultando banco.
- `merge_results` correlaciona 1:1 com item de `merge_queue`.

### RISCOS

- **Risco:** merge aplicado, mas log não gravado (crash entre operações).  
  **Mitigação:** reconciliação no startup comparando SHAs e estados `in_progress`.
- **Risco:** schema insuficiente para debug posterior.  
  **Mitigação:** armazenar SHAs antes/depois + erro bruto + tier escolhido.

## Item 6 — Tests para clean merges e conflict detection

### O QUE é

Suite de testes com repositórios Git temporários reais validando comportamento funcional e de resiliência da etapa 1.3.

### O QUE NÃO é

- Não é mock-only de comandos Git.
- Não é teste de Tier 3/Tier 4.
- Não é benchmark de performance nesta fase.

### COMO implementar

Matriz mínima de cenários:

1. **FIFO order**: 3 requests enfileirados -> processados na ordem.
2. **Tier 1 success**: fast-forward limpo -> sem merge commit.
3. **Tier 2 success**: divergência sem conflito -> merge commit criado.
4. **Pre-merge conflict**: `merge-tree` detecta conflito -> merge real não executa.
5. **Conflict cleanup**: conflito no merge real -> `merge --abort` executado.
6. **Crash recovery**: item `in_progress` com lease vencido -> volta para fila.
7. **Concurrent enqueue**: múltiplos writers -> sem duplicidade/perda.
8. **Stale source SHA**: branch alterada após enqueue -> item falha com erro explícito.

### CRITÉRIO DE ACEITAÇÃO

- Todos os cenários acima reproduzíveis em CI.
- Nenhum teste deixa `.git/MERGE_HEAD` pendente.
- Após execução da suite, banco e estado Git permanecem consistentes.

### RISCOS

- **Risco:** testes flaky por timing de concorrência.  
  **Mitigação:** relógio controlado para lease e asserts por polling com timeout explícito.
- **Risco:** falso positivo por usar repo não real.  
  **Mitigação:** criar repos temporários e commits reais por cenário.

## Edge cases mandatórios (validados nesta especificação)

### 1) Agent/worker crash no meio do merge

- Coberto por `lease_owner + lease_expires_at + recovery pass`.
- Item não fica preso em `in_progress` para sempre.
- Reconciliação de SHAs evita duplicar merge após restart.

### 2) Merge requests concorrentes

- Coberto por fila FIFO + claim atômico em transação + serialização no branch alvo.
- Evita race em refs compartilhadas.
- Mantém ordem determinística de processamento.
