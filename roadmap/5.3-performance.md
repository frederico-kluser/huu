## 1) Agent concurrency cap (configurável, default 5)

### O QUE é

Implementar limite global de agents executando em paralelo no scheduler, com valor padrão **5**, configurável por arquivo de config, CLI e variável de ambiente.

Além do limite, incluir sinais de **backpressure** para impedir crescimento descontrolado de fila.

### O QUE NÃO é

- Não é “rodar todos os 11 agents sempre em paralelo”.
- Não é auto-scaling complexo por padrão.
- Não é substituir merge serializado no `main` (continua obrigatório).

### COMO implementar

1. Criar configuração única:
   - `orchestrator.maxConcurrentAgents` (default `5`).
   - Overrides (ordem): CLI flag > ENV (`HUU_MAX_CONCURRENT_AGENTS`) > config file > default.
   - Faixa validada: `1..20` (com aviso explícito acima de 10 e hard cap em 20).
2. Implementar scheduler com fila observável:
   - **Recomendado:** `p-queue` para concorrência + rate limit + introspecção.
   - `p-limit` pode ser usado em fluxos simples/lotes sem prioridade.
   - `Semaphore`/`Mutex` deve proteger recursos compartilhados (ex.: refs/merge/worktree pool), não substituir fila principal.
3. Backpressure obrigatório:
   - Definir limite de backlog (ex.: `maxPendingTasks`).
   - Ao saturar (`isSaturated`/size threshold), bloquear admissão de novas tarefas não críticas e registrar evento.
4. Cancelamento/shutdown:
   - Suportar `AbortSignal` por task.
   - Em shutdown: parar admissão, drenar em execução, marcar pendências canceladas.

```ts
import PQueue from 'p-queue';

const queue = new PQueue({
  concurrency: config.orchestrator.maxConcurrentAgents ?? 5,
  intervalCap: config.orchestrator.intervalCap ?? 30,
  interval: 1000,
});

export async function enqueueAgentTask(task: AgentTask) {
  if (queue.isSaturated && queue.size > config.orchestrator.maxPendingTasks) {
    throw new Error('Scheduler overloaded: backpressure active');
  }

  return queue.add(
    ({ signal }) => runAgentTask(task, { signal }),
    { id: task.id, priority: task.priority, signal: task.abortSignal }
  );
}
```

#### Verificação do default = 5

`5` é adequado como baseline por três razões:

1. **Simplicidade + medição primeiro** (princípio arquitetural).
2. **Custo de coordenação cresce com escala** (risco já documentado no projeto).
3. **Perfil de workload**: parte relevante das tarefas compete por recursos compartilhados (merge, SQLite writes, refs git).

Ou seja: começar em 5, medir, e só então subir.

### CRITÉRIO DE ACEITAÇÃO

- Sem configuração explícita, scheduler usa **5**.
- Valor é configurável por config/CLI/ENV e validado.
- Teste de concorrência prova que `runningAgents` nunca excede cap.
- Sistema ativa backpressure ao saturar fila e não explode backlog.
- Cancelamento e shutdown não deixam promises pendentes.

### RISCOS + mitigação

- **Deadlock por limiter aninhado** → não reutilizar mesmo limiter dentro da própria função limitada.
- **Starvation de baixa prioridade** → aging ou teto de diferença de prioridade.
- **Cap muito alto degradando throughput real** → alertar e sugerir retorno ao baseline com base em métricas.

## 2) Coordination overhead metrics visíveis na TUI

### O QUE é

Medir e exibir, por sessão e por tarefa, quanto tempo o sistema gasta **coordenando** versus **executando trabalho útil**.

Métrica principal:

`coordination_overhead_ratio = coordination_ms / (coordination_ms + execution_ms)`

### O QUE NÃO é

- Não é métrica de qualidade semântica do código gerado.
- Não é proxy de custo de token isolado.
- Não é apenas contador de mensagens sem contexto temporal.

### COMO implementar

1. Instrumentar eventos mínimos no runtime:
   - `task_queued_at`
   - `task_dispatched_at`
   - `agent_started_at`
   - `agent_done_at`
   - `merge_queued_at`
   - `merge_started_at`
   - `merge_done_at`
   - `lock_wait_ms` (refs/worktree/merge)
2. Persistir como eventos auditáveis (reuso de `audit_log` ou tabela derivada `task_runtime_metrics`).
3. Calcular métricas derivadas:
   - `queue_wait_ms = agent_started_at - task_dispatched_at`
   - `execution_ms = agent_done_at - agent_started_at`
   - `merge_wait_ms = merge_started_at - merge_queued_at`
   - `merge_exec_ms = merge_done_at - merge_started_at`
   - `coordination_ms = queue_wait_ms + merge_wait_ms + lock_wait_ms + scheduler_decision_ms`
4. Exibir no `[C]ost` tab (3.4) com semáforo:
   - Verde `< 0.25`
   - Amarelo `0.25..0.40`
   - Vermelho `> 0.40`
5. Expor também séries: p50/p95 de queue wait, backlog, tasks/s concluídas.

```sql
SELECT
  session_id,
  SUM(coordination_ms) AS coordination_ms,
  SUM(execution_ms) AS execution_ms,
  ROUND(
    1.0 * SUM(coordination_ms) / NULLIF(SUM(coordination_ms) + SUM(execution_ms), 0),
    4
  ) AS coordination_overhead_ratio
FROM task_runtime_metrics
WHERE session_id = ?
GROUP BY session_id;
```

### CRITÉRIO DE ACEITAÇÃO

- TUI mostra `coordination_overhead_ratio` em tempo real da sessão.
- Métrica é rastreável até eventos brutos (auditabilidade).
- Consulta SQL retorna dados estáveis com 10k+ eventos.
- Alertas de faixa (verde/amarelo/vermelho) funcionam e são testados.

### RISCOS + mitigação

- **Métrica “bonita”, mas não acionável** → manter decomposição em submétricas (queue/merge/lock).
- **Eventos faltantes gerando distorção** → validação de integridade por tarefa (campos obrigatórios).
- **Custo de instrumentação alto** → batch insert e agregação incremental.

## 3) SQLite query optimization para histórico grande

### O QUE é

Conjunto de tuning de schema + runtime para manter consultas e gravações previsíveis com histórico longo (10k+ até 1M+ registros).

### O QUE NÃO é

- Não é trocar SQLite por outro banco.
- Não é indexar todas as colunas indiscriminadamente.
- Não é tuning “cego” sem `EXPLAIN QUERY PLAN` e medição.

### COMO implementar

1. **WAL + PRAGMAs base** no bootstrap:
   - `journal_mode = WAL`
   - `synchronous = NORMAL` (ou `FULL` via config se prioridade for durabilidade máxima)
   - `busy_timeout = 5000`
   - `temp_store = MEMORY`
   - `cache_size` em KB negativo (ex.: `-65536` para ~64MB)
2. **Checkpoint strategy**:
   - manter autocheckpoint (`wal_autocheckpoint`) e monitorar tamanho do `-wal`;
   - ao exceder limite configurado + janela segura, executar `wal_checkpoint(RESTART)` / `TRUNCATE`.
3. **Índices orientados a query real**:
   - índices compostos para filtros/ordenação frequentes (ex.: `(session_id, created_at)`, `(type, created_at)`, `(agent, timestamp)`).
4. **Planejador e estatísticas**:
   - executar `PRAGMA optimize;` periodicamente e após mudanças de schema/índices.
   - usar `EXPLAIN QUERY PLAN` para confirmar `SEARCH USING INDEX` nas queries quentes.
5. **Proteção contra configuração inválida**:
   - PRAGMA desconhecida pode ser ignorada silenciosamente; ler de volta e validar valor aplicado.

```ts
db.pragma('journal_mode = WAL');
db.pragma('synchronous = NORMAL');
db.pragma('busy_timeout = 5000');
db.pragma('temp_store = MEMORY');
db.pragma('cache_size = -65536');
db.pragma('wal_autocheckpoint = 1000');
db.pragma('optimize');
```

### CRITÉRIO DE ACEITAÇÃO

- Queries críticas usam índice (validado por `EXPLAIN QUERY PLAN`).
- p95 de leitura/escrita em volume alvo permanece dentro do SLO definido pelo projeto.
- `-wal` não cresce sem controle em carga prolongada.
- PRAGMAs aplicadas são verificadas por leitura explícita e logadas.

### RISCOS + mitigação

- **Over-indexing piorando write throughput** → medir custo de escrita antes/depois e remover índice inútil.
- **Checkpoint starvation** → monitorar `-wal` + checkpoint coordenado.
- **Lock contention** → `busy_timeout`, transações curtas e batch writes.

## 4) Worktree reuse (pool/recycle)

### O QUE é

Reutilizar worktrees já criadas para reduzir custo de `add/remove` por tarefa, com isolamento preservado por branch/worktree e reset completo entre leases.

### O QUE NÃO é

- Não é compartilhar uma mesma worktree entre tarefas simultâneas.
- Não é pular limpeza de estado entre tarefas.
- Não é desativar regras de segurança de refs/merge.

### COMO implementar

1. Introduzir `WorktreePool` com estados:
   - `idle`, `leased`, `cleaning`, `quarantined`.
2. Fluxo de aquisição:
   - tentar reaproveitar `idle`;
   - fallback para `git worktree add` quando pool vazio;
   - respeitar cap de pool (normalmente próximo ao cap de concorrência).
3. Fluxo de liberação:
   - `git reset --hard <base>`
   - `git clean -fdx`
   - validação de limpeza (`status --porcelain` vazio)
   - retorno ao estado `idle` ou `quarantined`.
4. Segurança:
   - merge/ref updates protegidos por mutex/semaphore.
   - uma instância `SimpleGit` por worktree.
   - usar `git worktree list --porcelain` para reconciliação e `prune` de metadados stale.
5. Otimização prática:
   - symlink de `node_modules` quando seguro.
   - prewarm do pool no startup (opcional via config).

```ts
const release = await worktreePool.acquire(task.agentId);
try {
  await runTaskInWorktree(task, release.worktreePath);
} finally {
  await worktreePool.recycle(release.worktreeId, task.baseRef);
}
```

### CRITÉRIO DE ACEITAÇÃO

- Reuso reduz latência de provisionamento de worktree vs. caminho cold.
- Nenhuma tarefa inicia com worktree “suja”.
- Em concorrência, não há colisão de refs nem corrupção de estado.
- Pool se recupera de worktree desaparecida/corrompida (quarentena + recriação).

### RISCOS + mitigação

- **Estado residual** → limpeza obrigatória + validação pós-clean.
- **Corrupção por path movido/removido** → reconciliação no startup (`list/prune/repair`) e quarentena.
- **Contenção em refs compartilhadas** → serialização explícita de operações críticas.
