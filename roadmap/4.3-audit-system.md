# 4.3 Audit System

> Phase 4 — Intelligence (Act 3: Resolution)

Trust but Verify: máxima autonomia operacional com rastreabilidade completa, detecção de risco e prestação de contas por sessão, feature, agente e modelo.

## Objetivo da etapa

Definir e implementar o sistema de auditoria fim-a-fim do HUU para:

1. Registrar 100% das ações relevantes (tool calls e chamadas LLM),
2. Detectar comportamentos suspeitos sem bloquear o fluxo normal por padrão,
3. Produzir relatórios pós-sessão auditáveis e reproduzíveis,
4. Medir custo real em granularidade suficiente para otimização (sessão/feature/agente/fase/modelo),
5. Sustentar explicitamente a filosofia arquitetural **"Trust but Verify"** descrita em `ARCHITECTURE.md`.

## Coerência com `ARCHITECTURE.md`

Esta etapa está alinhada com as decisões arquiteturais:

- **Decisão 3 (SQLite WAL)**: auditoria centralizada no mesmo SQLite unificado.
- **Decisão 5 (Model tiering)**: custo por modelo/agente para validar estratégia Opus/Sonnet/Haiku.
- **Decisão 7 (Context-Curator)**: eventos e custos do curador entram no trilho de auditoria.
- **Decisão 8 (CoVe seletivo)**: custo e trilha de verificação crítica rastreáveis.
- **Decisão 10 (TUI Cost tab)**: dados auditáveis alimentam visualização de custo.
- **Decisão 12 (Trust but Verify)**: sem bloqueios excessivos no caminho feliz, com verificação total pós-fato e escalonamento baseado em risco.

## 4.3.1 Complete tool call logging

### O QUE é

Um trilho de auditoria **append-only** para todas as execuções de ferramentas e chamadas LLM, com contexto suficiente para reconstruir cronologia, causalidade e impacto:

- `session_id`, `run_id`, `trace_id`, `span_id`, `parent_span_id`,
- agente, tarefa/feature (`task_id`, `feature_id`, `beat_id`),
- ação executada (`event_type`, `tool_name`),
- status, duração, erro, resumo do resultado,
- tokens e custo estimado (quando aplicável).

### O QUE NÃO é

- Não é log bruto irrestrito de prompts/outputs completos com risco de vazamento.
- Não é mecanismo de bloqueio preventivo em toda ação (isso pertence a políticas de execução/gate específico).
- Não é substituto de logs operacionais de debug; é trilha de auditoria orientada a governança e forense.

### COMO implementar

1. **Instrumentação obrigatória** nos pontos:
   - pré e pós execução de qualquer tool,
   - pré e pós chamada LLM (incluindo CoVe, context-curator e AI merge resolver),
   - eventos de escalonamento humano.
2. **Sanitização** antes de persistir:
   - redigir segredos por padrão (`api_key`, `token`, `password`, `authorization`, etc.),
   - armazenar `params_hash` (SHA-256) + resumo textual curto,
   - limitar payload textual para evitar explosão de armazenamento.
3. **Integridade/tamper evidence**:
   - campos `prev_hash` e `entry_hash` para hash-chain por sessão,
   - sem UPDATE/DELETE em eventos históricos (somente append).
4. **Esquema recomendado** (extensão de `audit_log` da etapa 0.2):

```sql
CREATE TABLE IF NOT EXISTS audit_log (
  id INTEGER PRIMARY KEY,
  ts_ms INTEGER NOT NULL,
  session_id TEXT NOT NULL,
  run_id TEXT NOT NULL,
  trace_id TEXT NOT NULL,
  span_id TEXT NOT NULL,
  parent_span_id TEXT,
  agent_id TEXT NOT NULL,
  phase TEXT NOT NULL,
  feature_id TEXT,
  task_id TEXT,
  beat_id TEXT,
  event_type TEXT NOT NULL,      -- tool_call_start|tool_call_end|llm_call_start|llm_call_end|escalation
  tool_name TEXT,
  model_name TEXT,
  success INTEGER,
  error_code TEXT,
  duration_ms INTEGER,
  params_sanitized TEXT,
  params_hash TEXT,
  result_summary TEXT,
  input_tokens INTEGER,
  output_tokens INTEGER,
  cache_creation_input_tokens INTEGER,
  cache_read_input_tokens INTEGER,
  estimated_cost_usd REAL,
  risk_score INTEGER DEFAULT 0,
  risk_flags_json TEXT,
  prev_hash TEXT,
  entry_hash TEXT NOT NULL
);
```

5. **Indices mínimos** para análise operacional:

```sql
CREATE INDEX IF NOT EXISTS idx_audit_session_ts ON audit_log(session_id, ts_ms);
CREATE INDEX IF NOT EXISTS idx_audit_agent_ts ON audit_log(agent_id, ts_ms);
CREATE INDEX IF NOT EXISTS idx_audit_event_ts ON audit_log(event_type, ts_ms);
CREATE INDEX IF NOT EXISTS idx_audit_failed_tools ON audit_log(tool_name, ts_ms) WHERE success = 0;
CREATE INDEX IF NOT EXISTS idx_audit_high_risk ON audit_log(session_id, ts_ms) WHERE risk_score >= 70;
```

### CRITÉRIO DE ACEITAÇÃO

- 100% das tool calls e LLM calls geram eventos `start` + `end`.
- Toda linha de auditoria contém correlação (`session_id`, `trace_id`, `span_id`).
- Segredos não aparecem em claro nos logs (teste automatizado com payloads sensíveis).
- Hash-chain válida por sessão (recomputável e verificável).

### RISCOS

- **Vazamento de dados sensíveis em log** → sanitização central + testes de redaction.
- **Alto volume de escrita degradar SQLite** → WAL + índices seletivos + retenção.
- **Perda de causalidade** por IDs inconsistentes → geração padronizada no runtime base.

## 4.3.2 Post-session audit report generation

### O QUE é

Geração automática de relatório pós-sessão em **JSON + Markdown** com visão executiva e trilha técnica detalhada, cobrindo:

- resumo da sessão (sucessos, falhas, escalonamentos),
- timeline de ações,
- eventos suspeitos e severidade,
- custo por agente/modelo/fase/feature,
- recomendações objetivas de melhoria.

### O QUE NÃO é

- Não é relatório exclusivamente narrativo gerado por LLM sem base determinística.
- Não é dashboard em tempo real (isso é responsabilidade da TUI).
- Não é substituto de investigação humana em incidentes críticos.

### COMO implementar

1. Pipeline determinístico de relatório:
   - agregações SQL -> estrutura JSON canônica -> render Markdown.
2. Persistir ambos:
   - `audit_reports` (metadata + json),
   - artefato `.md` exportável.
3. Se sumarização por LLM for usada, ela deve ser **opcional**, sempre derivada de fatos já estruturados.
4. Estrutura mínima do relatório:
   - Sessão e escopo,
   - KPIs (ações, falhas, custo total, p95 duração),
   - Top 10 eventos de maior risco,
   - Top 10 eventos de maior custo,
   - divergências/políticas violadas,
   - ações recomendadas.

### CRITÉRIO DE ACEITAÇÃO

- Relatório é gerado ao final de 100% das sessões finalizadas.
- Mesmo `session_id` + mesmo banco => mesmo relatório (reprodutibilidade).
- Relatório contém links/IDs navegáveis para eventos-fonte.
- Exportação Markdown abre corretamente em viewer comum e TUI.

### RISCOS

- **Relatório incompleto por dependência de joins frágeis** → contrato fixo de schema + testes de snapshot.
- **Narrativa divergente dos fatos** → separar camada factual (SQL) da camada textual (opcional).
- **Arquivo muito grande** → versão detalhada + versão executiva compacta.

## 4.3.3 Suspicious action flagging

### O QUE é

Motor heurístico de detecção de atividade suspeita para agentes, com score e flags por evento/sessão, orientado aos riscos de aplicações agentic (prompt injection indireta, abuso de tool, exfiltração, DoW/loops, etc.).

### O QUE NÃO é

- Não é SIEM corporativo completo.
- Não é classificador ML black-box obrigatório para fase inicial.
- Não é bloqueio automático universal (padrão é sinalizar; bloquear apenas regras críticas explícitas).

### COMO implementar

1. **Rule engine** declarativo com pesos:
   - cada regra gera `risk_points` + `flag_code`.
2. Heurísticas mínimas obrigatórias:
   - acesso a path fora do workspace esperado,
   - picos anormais de escrita/deleção de arquivos,
   - repetição de mesma tool+params hash em janela curta (loop/DoW),
   - sequência de falhas acima de limiar,
   - custo anormal por minuto/agente,
   - indícios de goal hijack/prompt injection em conteúdo externo,
   - uso de ferramenta fora do perfil do agente.
3. Escalonamento por severidade:
   - `low` (apenas log),
   - `medium` (flag no relatório),
   - `high` (alerta TUI),
   - `critical` (requer confirmação humana para continuar pipeline específico).
4. Persistência:
   - `risk_score` e `risk_flags_json` em `audit_log`,
   - opcional `audit_flags` para normalização.

Exemplo simplificado:

```ts
if (event.toolName === "bash" && event.pathOutsideWorkspace) {
  addFlag(event, "PATH_OUTSIDE_WORKSPACE", 40);
}
if (event.repeatCount5m >= 8) {
  addFlag(event, "POSSIBLE_LOOP_DOW", 35);
}
if (event.estimatedCostUsd > dynamicThreshold(agentId)) {
  addFlag(event, "COST_SPIKE", 30);
}
```

### CRITÉRIO DE ACEITAÇÃO

- Regras obrigatórias acima implementadas e testadas com casos sintéticos.
- Eventos `high/critical` aparecem na TUI e no relatório pós-sessão.
- Taxa de falso positivo monitorada e ajustável por configuração.
- Nenhum evento crítico deixa de ser registrado.

### RISCOS

- **Falso positivo alto** → thresholds por baseline de sessão/agente + calibração contínua.
- **Falso negativo em ataques novos** → rule registry extensível + revisão periódica.
- **Regra cara em runtime** → heurísticas O(1)/O(log n) no hot path; análises pesadas no pós-sessão.

## 4.3.4 Cost reporting per session, per feature, per agent

### O QUE é

Sistema de contabilização e atribuição de custo com granularidade operacional e financeira:

- por sessão,
- por feature/tarefa (beat/task/feature_id),
- por agente,
- por fase e por modelo,
- incluindo custo de cache (read/write), CoVe, context-curator e AI merge resolver.

### O QUE NÃO é

- Não é estimativa simplificada por tarifa única por modelo.
- Não é somente soma de `input_tokens` e `output_tokens` sem cache/modificadores.
- Não é relatório mensal agregado sem drill-down.

### COMO implementar

1. **Catálogo de preços versionado** (`model_pricing`) com vigência temporal e dimensões:
   - `model_name`, `input_per_mtok`, `output_per_mtok`,
   - `cache_write_per_mtok`, `cache_read_per_mtok`,
   - `tool_unit_cost` (ex.: web search, code execution).
2. **Registrar uso bruto** por chamada:
   - `input_tokens`,
   - `output_tokens`,
   - `cache_creation_input_tokens`,
   - `cache_read_input_tokens`.
3. **Cálculo de custo por evento**:

```ts
const cost =
  (inputTokens / 1_000_000) * price.inputPerMTok +
  (outputTokens / 1_000_000) * price.outputPerMTok +
  (cacheWriteTokens / 1_000_000) * price.cacheWritePerMTok +
  (cacheReadTokens / 1_000_000) * price.cacheReadPerMTok +
  extraToolCosts;
```

4. **Atribuição obrigatória de escopo**: cada evento de custo deve ter `session_id`, `agent_id`, `phase`, e sempre que houver `feature_id`/`task_id`.
5. **Cobertura explícita de pontos de gasto**:

| Ponto de gasto | Como capturar | Campo obrigatório |
|---|---|---|
| Agents (todos) | `llm_call_end` por execução | `agent_id`, `model_name`, tokens, custo |
| CoVe (4 passos) | `event_type = cove_step` por passo | `cove_step`, tokens, custo |
| Context-curator | `event_type = curator_run` | `agent_id=context-curator`, tokens, custo |
| Merger AI resolver (Tier 4) | `event_type = merge_tier4_llm` | `merge_tier=4`, tokens, custo |
| Web search / code execution (quando houver cobrança) | `event_type = tool_billing` | `tool_name`, `quantity`, `unit_cost` |

6. **Views de consumo**:
   - `cost_by_session`,
   - `cost_by_feature`,
   - `cost_by_agent_model_phase`.

### CRITÉRIO DE ACEITAÇÃO

- Custo total por sessão é reconciliável com custo somado por evento.
- CoVe, context-curator e Tier 4 merger aparecem separadamente nos relatórios.
- TUI `[C]ost` mostra breakdown por agente/modelo/fase e por feature.
- Mudança de preços não altera retrospectivamente sessões antigas (snapshot/versionamento).

### RISCOS

- **Subcontagem por caminhos sem instrumentação** → wrapper único obrigatório para toda chamada LLM/tool.
- **Supercontagem por dupla contabilização** → contrato claro entre campos de uso + testes de reconciliação.
- **Deriva de preços ao longo do tempo** → tabela versionada com `effective_from`/`effective_to`.

## Padrões de query SQLite para análise de auditoria

1. **Timeline de sessão (forense):**

```sql
SELECT ts_ms, agent_id, event_type, tool_name, success, duration_ms, estimated_cost_usd, risk_score
FROM audit_log
WHERE session_id = ?
ORDER BY ts_ms;
```

2. **Top custos por agente na sessão:**

```sql
SELECT agent_id, SUM(estimated_cost_usd) AS total_cost
FROM audit_log
WHERE session_id = ?
GROUP BY agent_id
ORDER BY total_cost DESC;
```

3. **Detecção de loop (mesma tool+hash em janela curta):**

```sql
WITH ordered AS (
  SELECT session_id, agent_id, tool_name, params_hash, ts_ms,
         LAG(ts_ms) OVER (
           PARTITION BY session_id, agent_id, tool_name, params_hash
           ORDER BY ts_ms
         ) AS prev_ts
  FROM audit_log
  WHERE event_type = 'tool_call_end'
)
SELECT *
FROM ordered
WHERE prev_ts IS NOT NULL AND (ts_ms - prev_ts) < 5000;
```

4. **Anomalia de custo por minuto (baseline local):**

```sql
WITH minute_cost AS (
  SELECT session_id,
         (ts_ms / 60000) AS minute_bucket,
         SUM(estimated_cost_usd) AS cost_min
  FROM audit_log
  GROUP BY session_id, minute_bucket
),
stats AS (
  SELECT session_id,
         AVG(cost_min) AS avg_cost,
         MAX(cost_min) AS max_cost
  FROM minute_cost
  GROUP BY session_id
)
SELECT m.*, s.avg_cost,
       CASE WHEN m.cost_min >= s.avg_cost * 3 THEN 1 ELSE 0 END AS is_spike
FROM minute_cost m
JOIN stats s USING (session_id);
```

> Operacional: usar `EXPLAIN QUERY PLAN` durante desenvolvimento e rodar `ANALYZE` após migrações de índice para manter performance de consultas históricas.
