# 3.3 Human Intervention

> Phase 3 — TUI (Act 2 → Act 3: Resolution)

A camada de intervenção humana mantém o operador no controle durante execução real: redirecionar (`steer`), enfileirar próxima instrução (`follow-up`), abortar com cleanup determinístico (`abort`) e promover aprendizado explícito (`promote`).

## Status de auditoria da etapa

**Situação atual:** incompleta.  
O arquivo anterior descrevia apenas intenção funcional e não trazia semântica operacional, contrato de dados, regras de concorrência, nem critérios testáveis de pronto.

**Coerência com `ARCHITECTURE.md`:** parcialmente coerente (ações `[S][F][A][P]` existem na decisão #10), porém faltavam detalhes para manter consistência com:
- comunicação tipada via SQLite WAL (decisão #3),
- fluxo de orquestração/estado do agente (decisões #1 e #6),
- merge FIFO e escalonamento humano (decisão #4),
- TUI com Detail View e logs em tempo real (decisão #10).

Esta versão fecha essas lacunas e define contrato implementável ponta a ponta.

## Contrato operacional da etapa (cross-cutting)

### 1) Semântica de prioridade e ordering

Prioridade de comandos concorrentes para o mesmo `agent_run_id`:

1. `abort` (prioridade máxima)
2. `steer`
3. `follow_up`

Regras obrigatórias:
- `abort` **cancela** qualquer `steer` pendente e esvazia fila de `follow_up`.
- `steer` é aplicado no próximo **safe interrupt point** (fim da tool atual / antes do próximo turno do modelo).
- `follow_up` é FIFO por `agent_run_id` e **não** interrompe turno em andamento.
- Comandos devem ter `command_id` único para idempotência (at-least-once no transporte, exactly-once no efeito).

### 2) Contrato de mensagem (SQLite mail system)

Usar a tabela `messages` já prevista no 0.2, com payload JSON tipado (sem criar subsistema paralelo nesta etapa).

```ts
type InterventionKind = 'steer' | 'follow_up' | 'abort' | 'promote';
type InterventionState = 'queued' | 'accepted' | 'applied' | 'rejected' | 'canceled' | 'failed';

interface InterventionPayload {
  commandId: string;            // UUID
  kind: InterventionKind;
  taskId: string;
  agentId: string;
  agentRunId: string;
  requestedBy: 'human';
  text?: string;                // steer/follow_up
  requestedAt: string;          // ISO timestamp
  state: InterventionState;
  reason?: string;              // rejected/canceled/failed
  meta?: {
    replacesCommandId?: string; // steer last-write-wins opcional
    followUpPosition?: number;
    promotedInstinctId?: string;
  };
}
```

Tipos de mensagem recomendados:
- `intervention_requested`
- `intervention_ack`
- `intervention_applied`
- `intervention_rejected`
- `intervention_failed`

> Nota: manter `task_progress`, `task_done`, `escalation`, `merge_ready` etc. sem alteração; 3.3 apenas adiciona eventos de intervenção.

### 3) Entrega em tempo real para agents em execução

Padrão recomendado para esta etapa:
- **Durabilidade/replay:** SQLite `messages` (fonte de verdade).
- **Tempo real em processo único:** `EventEmitter`/bus interno publica novo evento imediatamente para o Detail View.
- **Fallback cross-process:** polling incremental no SQLite por `id > lastSeenId` (200-500ms), sem varredura completa.

```ts
async function streamAgentMessages(agentRunId: string, lastSeenId: number) {
  return db.prepare(`
    SELECT id, type, payload, created_at
    FROM messages
    WHERE agent_run_id = ? AND id > ?
    ORDER BY id ASC
  `).all(agentRunId, lastSeenId);
}
```

Esse padrão mantém baixa latência de UI e preserva recuperação pós-crash (replay por `lastSeenId`).

## Item 1 — `[S]teer`

### O QUE é

Comando humano de **redirecionamento imediato** para um agent já em execução, com efeito no próximo ponto seguro de interrupção.

Semântica alvo:
- interrompe fluxo planejado atual do agent (sem matar processo),
- injeta nova instrução com prioridade sobre backlog lógico do turno,
- registra ACK/aplicação para observabilidade.

### O QUE NÃO é

- Não é kill de processo (`abort` cobre isso).
- Não é edição retroativa de mensagens anteriores.
- Não é replanejamento global do beat sheet (isso continua no orquestrador/planner).
- Não é garantia de interrupção no meio de uma syscall bloqueante; interrupção é cooperativa no boundary seguro.

### COMO implementar

1. **Captura na TUI (Detail View):**
   - Atalho `S` seleciona modo steer.
   - Campo textual recebe instrução.
   - Enter envia `intervention_requested(kind=steer)`.

2. **Roteamento no orquestrador:**
   - Valida se `task.status === Running`.
   - Rejeita steer para task terminal (`Done`, `Failed`, `Review`).
   - Publica comando no canal de controle do runtime.

3. **Aplicação no runtime:**
   - Armazena steer pendente por `agentRunId`.
   - No próximo safe point, injeta instrução no loop do agent.
   - Emite `intervention_applied`.

4. **Política de múltiplos steer:**
   - Recomendado: **last-write-wins** enquanto ainda pendente.
   - Comando substituído recebe `state=canceled` e `reason='superseded_by_newer_steer'`.

Snippet (TUI + input):

```tsx
useInput((input, key) => {
  if (input === 's' && !isComposing) setMode('steer');
}, { isActive: detailViewFocused });

<TextInput
  value={value}
  onChange={setValue}
  onSubmit={(text) => dispatchIntervention({ kind: 'steer', text })}
  focus={mode === 'steer'}
  placeholder="Steer: redirecione o agente..."
/>;
```

### CRITÉRIO DE ACEITAÇÃO

- Dado um agent `Running`, ao enviar steer:
  - comando vira `accepted` em até 1 ciclo de monitoramento;
  - próximo output do agent reflete nova direção;
  - card permanece em `Running` (não transita para `Failed`);
  - log exibe `requested -> accepted -> applied` com `commandId`.
- Steer em task terminal é rejeitado com mensagem explícita na UI (`rejected` + motivo).

### RISCOS

- **Race com conclusão natural da task** (steer chega ao mesmo tempo do `task_done`).
  - Mitigar com verificação atômica de estado antes de aplicar.
- **Steer storm** (muitos steer seguidos).
  - Mitigar com debounce/throttle e last-write-wins.
- **Ambiguidade de instrução humana**.
  - Mitigar com preview/confirm opcional para comandos destrutivos.

## Item 2 — `[F]ollow-up`

### O QUE é

Comando humano de **continuação pós-turno**, enfileirado para execução após a conclusão natural do turno atual do agent.

Semântica alvo:
- não interrompe tool/turno em progresso,
- preserva ordem de chegada (FIFO),
- permite orientar próximos passos com baixa fricção.

### O QUE NÃO é

- Não é steer disfarçado (não interrompe agora).
- Não é backlog global do projeto (escopo é o `agentRunId` atual).
- Não é fila infinita sem governança.

### COMO implementar

1. **Fila por execução:**
   - estrutura `Map<agentRunId, FollowUpQueue>`.
   - cada item com `commandId`, `text`, `createdAt`, `position`.

2. **Enqueue via TUI:**
   - Atalho `F` abre modo follow-up.
   - Enter persiste `intervention_requested(kind=follow_up)` e adiciona à fila.

3. **Drain da fila:**
   - Trigger em evento `turn_end`.
   - Consome 1 item por turno (modo conservador).
   - Se fila vazia: sem ação.

4. **Governança mínima:**
   - limite recomendado: `MAX_FOLLOW_UP_QUEUE = 10` por agent.
   - excedente -> `rejected` com motivo `queue_limit_reached`.

```ts
function onTurnEnd(run: AgentRun) {
  const next = followUpQueue.dequeue(run.id);
  if (!next) return;
  runtime.injectNextUserMessage(run.id, next.text);
  emitApplied(next.commandId);
}
```

### CRITÉRIO DE ACEITAÇÃO

- Follow-up enviado durante execução não altera resposta em progresso.
- Primeiro turno após `turn_end` usa o follow-up mais antigo.
- Três follow-ups enviados em sequência executam na mesma ordem.
- Ao abortar o agent, fila é esvaziada e itens restantes recebem `canceled`.

### RISCOS

- **Fila longa com instruções obsoletas**.
  - Mitigar com limite de tamanho e opção de limpar fila.
- **Conflito semântico com steer posterior**.
  - Mitigar com regra explícita: steer não remove follow-ups, mas abort remove ambos.
- **Baixa visibilidade do que está enfileirado**.
  - Mitigar exibindo “Pending follow-ups (N)” no Detail View.

## Item 3 — `[A]bort`

### O QUE é

Cancelamento humano **graceful e determinístico** de uma execução em andamento, com cleanup completo de recursos:
- runtime (execução e streaming),
- worktree/branch temporária,
- estado no SQLite (mensagens e task),
- merge queue (remoção de entradas pendentes).

### O QUE NÃO é

- Não é apenas `controller.abort()` sem pós-processamento.
- Não é “best effort silencioso”; falha de cleanup deve ser registrada e visível.
- Não é hard-kill imediato como caminho primário (usar hard-kill apenas fallback controlado).

### COMO implementar

Pipeline recomendado:

1. **Sinal de cancelamento:**
   - `AbortController.abort(new Error('human_abort'))`.
   - Propagar `signal` para tool calls, subprocessos e IO (`spawn/exec` com `signal`).

2. **Transição de estado:**
   - `Running -> Aborting -> Failed`.
   - Motivo padronizado: `failed_reason='human_abort'`.

3. **Cleanup idempotente (ordem fixa):**
   1. Persistir `intervention_applied(kind=abort)`.
   2. Cancelar steer/follow-up pendentes.
   3. Marcar task como `Failed`.
   4. Remover entradas da merge queue da task/branch.
   5. Remover worktree (`WorktreeManager.remove(agentId)`), e branch temporária quando aplicável.
   6. Persistir `agent_aborted`/`cleanup_done`.

4. **Fallback controlado:**
   - se processo não encerrar após timeout curto, aplicar kill explícito;
   - ainda assim executar cleanup e registrar fallback.

5. **Sem swallow de erro:**
   - erro de cleanup deve produzir `intervention_failed` + `escalation`.

Snippet de referência:

```ts
async function abortAgentRun(ctx: AbortContext): Promise<void> {
  if (ctx.state.isTerminal()) return; // idempotência

  ctx.transition('aborting');
  ctx.controller.abort(new Error('human_abort'));

  try {
    await ctx.runtime.waitForStop({ timeoutMs: 5000 });
  } finally {
    await ctx.db.transaction(() => {
      ctx.messages.markPendingControlsCanceled(ctx.agentRunId);
      ctx.tasks.markFailed(ctx.taskId, 'human_abort');
      ctx.mergeQueue.removeByTaskId(ctx.taskId);
      ctx.messages.insertAbortApplied(ctx);
    });

    await ctx.worktree.remove(ctx.agentId, { force: true });
  }
}
```

Padrões Node.js obrigatórios nesta etapa:
- `AbortSignal.any([...])` para compor cancelamento do usuário + timeout;
- listeners `abort` com `{ once: true }` para evitar leak;
- `throwIfAborted()` em loops longos/CPU-bound;
- diferenciar erro de abort (`AbortError`) de erro operacional.

### CRITÉRIO DE ACEITAÇÃO

- Ao abortar agent `Running`:
  - task vai para coluna `Failed` com razão `human_abort`;
  - worktree do agent não existe mais;
  - não há item da task na merge queue;
  - mensagens de `abort requested/applied/completed` existem no SQLite;
  - Detail View para de receber stream daquela execução.
- Repetir abort no mesmo `agentRunId` não quebra estado (idempotente).

### RISCOS

- **Race condition** entre `task_done` e `abort`.
  - Mitigar com lock/mutex por `agentRunId` + compare-and-set de estado.
- **Worktree órfã** por falha de remoção.
  - Mitigar com cleanup retry + stale-cleanup job na fase 5.2.
- **Entrada fantasma em merge queue**.
  - Mitigar com remoção transacional e verificação pós-op.
- **Cancelamento parcial de subprocessos**.
  - Mitigar propagando `signal` nativamente para `child_process` e registrando timeout fallback.

## Item 4 — `[P]romote`

### O QUE é

Ação humana para transformar aprendizado de uma task `Done` em registro explícito em `instincts` (com provenance), sem depender de inferência automática.

### O QUE NÃO é

- Não substitui pipeline completo de aprendizado da fase 4.2.
- Não promove automaticamente toda task concluída.
- Não publica conhecimento global por padrão sem validação humana.

### COMO implementar

1. **Pré-condições:**
   - task em `Done`;
   - evidência disponível (resumo, diff, decisão, teste associado).

2. **Fluxo de UI:**
   - Atalho `P` no card/task `Done`.
   - Abre formulário: título, descrição, tags, confiança inicial (0.3-0.85), escopo.
   - Confirmação explícita do operador.

3. **Persistência:**
   - Inserir em `instincts` com `source_task_id`, `source_agent_id`, `created_by='human'`.
   - Registrar mensagem `intervention_applied(kind=promote)`.

4. **Deduplicação mínima:**
   - hash de `(title + normalized_content + source_task_id)` para evitar duplicatas triviais.

```ts
interface PromoteInput {
  taskId: string;
  agentId: string;
  title: string;
  content: string;
  tags: string[];
  confidence: number; // 0.3..0.85
}
```

### CRITÉRIO DE ACEITAÇÃO

- `P` só aparece habilitado para tasks `Done`.
- Após confirmação, existe novo registro em `instincts` com provenance completo.
- A UI confirma sucesso e mostra `instinct_id`.
- Duplicata detectada retorna aviso explícito, sem inserir linha redundante.

### RISCOS

- **Promoção de aprendizado fraco/ruidoso**.
  - Mitigar exigindo evidência textual e revisão humana.
- **Explosão de instintos duplicados**.
  - Mitigar com hash de dedupe + busca aproximada por título.
- **Confiança inflada manualmente**.
  - Mitigar com defaults conservadores e histórico auditável.

## Plano de testes de aceitação (E2E)

1. **Steer básico:** agent em execução recebe steer e muda direção no próximo turno.
2. **Follow-up FIFO:** enfileirar 3 follow-ups e validar ordem exata.
3. **Abort completo:** abortar agent e verificar:
   - status `Failed(human_abort)`,
   - worktree removida,
   - merge queue limpa,
   - mensagens persistidas.
4. **Abort idempotente:** abort duplicado não gera inconsistência.
5. **Promote válido:** task `Done` gera linha em `instincts` com provenance.
6. **Promote inválido:** task não-`Done` é rejeitada com feedback claro.
7. **Realtime + replay:** abrir Detail View após reconectar e recuperar histórico por `lastSeenId`.

## Não-objetivos explícitos desta etapa

- Não implementar broker externo (Kafka/Redis/NATS) nesta fase.
- Não substituir SQLite WAL como backbone de mensagens.
- Não implementar auto-promoção sem humano.
- Não resolver toda política de governança de memória da fase 4.2.
- Não alterar estratégia de merge tier 3/4 além da remoção de fila no abort.
