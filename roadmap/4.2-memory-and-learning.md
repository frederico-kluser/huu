# 4.2 Memory & Learning

> Phase 4 — Intelligence (Act 3: Resolution)

This phase turns raw execution history into reliable behavior improvements with a closed loop:
`observation -> analysis -> instinct -> reinforcement/decay -> promotion -> session summary -> next-session loading`.

---

## Completeness and architecture coherence check

### Current diagnosis (before implementation)

The previous version of this file was not enough for implementation because it did not define:

1. Event schema and hook contract for deterministic observation coverage.
2. End-to-end loop control (who triggers each stage, when, and with which guardrails).
3. Confidence scoring math and explicit decay/reinforcement rules.
4. Promotion criteria from instinct to durable project knowledge.
5. Summary format and context-loading selection strategy.
6. Why threshold `20 observations` exists and whether it is configurable.

### Alignment with `ARCHITECTURE.md`

This spec is intentionally aligned with Decision 7 and related principles:

- **SQLite unified memory**: all memory artifacts remain in the same SQLite database (`observations`, `instincts`, `sessions`, `entities`, `relations`).
- **Context-curator over blind accumulation**: promotion and pruning are explicit; no uncontrolled growth.
- **Agents stateless, scratchpad stateful**: learned behavior is persisted outside any single agent context.
- **Retrieval just-in-time**: startup loads only relevant summaries/instincts, never full history blindly.
- **Model tiering**: Haiku is default for pattern analysis and summary drafting (mechanical tasks).

## Closed learning loop (mandatory)

Implementation is only considered complete when all transitions below are live:

1. **Observe**: every tool call emits structured events (`pre` + `post`).
2. **Aggregate**: events are normalized and grouped by project/session/tool/domain.
3. **Analyze**: analyzer evaluates windows that meet minimum evidence thresholds.
4. **Instinct upsert**: new/updated instinct candidate is created with bounded confidence.
5. **Apply during runtime**: active instincts are injected as guidance for relevant tasks.
6. **Reinforce/contradict**: outcomes increase or decrease confidence with explicit math.
7. **Decay + prune**: aging and contradiction penalties lower confidence over time.
8. **Promote**: mature instincts are promoted to project knowledge (`entities/relations`).
9. **Summarize session**: durable outcomes and pending state are recorded in `sessions`.
10. **Load next startup context**: last N days are selectively retrieved and injected.

If any transition is missing, the loop is incomplete.

## Recommended configuration (defaults + configurability)

The value `20 observations` is a **good default**, not a hardcoded constant. It must be configurable.

```yaml
memoryLearning:
  observation:
    enabled: true
    preToolHook: true
    postToolHook: true
    sanitizePii: true
  analysis:
    model: haiku
    minObservations: 20           # default floor, configurable
    minUniqueSessions: 3          # avoids single-session overfit
    cooldownMinutes: 15           # avoids over-analysis
    maxCandidatesPerRun: 10
  instinct:
    confidenceMin: 0.30
    confidenceMax: 0.85
    reinforceDelta: 0.04
    contradictionDelta: 0.08
    decayHalfLifeDays: 14
    deleteBelow: 0.20
  promotion:
    minConfidence: 0.78
    minSupportingSessions: 5
    minSupportingObservations: 50
    requireHumanApproval: false
  sessions:
    summaryModel: haiku
    loadWindowDays: 7
    maxSummariesToLoad: 20
```

### Why `20` is acceptable as default

- It is consistent with ECC V2 practice for low-cost first signal extraction.
- It reduces noisy pattern mining compared with very small windows.
- It is still cheap enough for frequent projects.

### Why `20` must be configurable

- Different repositories have very different tool-call volumes.
- High-noise projects may need `30-50`; low-volume projects may need `10-15`.
- Dynamic-threshold practices (industry observability) favor adaptive baselines over fixed constants.

## 1) Observation logging via tool call hooks (100% coverage)

### O QUE E

Deterministic telemetry ingestion for every tool call lifecycle, capturing both intent (`pre`) and outcome (`post`) with correlation IDs.

### O QUE NAO E

- Not probabilistic skill-based observation.
- Not best-effort logging that can silently fail.
- Not free-form text logs without typed schema.

### COMO IMPLEMENTAR

1. Intercept tool execution in Agent Runtime (`pre` and `post` hooks).
2. Write structured rows to `observations` in a single DB transaction per event.
3. Persist at least:
   - `project_id`, `session_id`, `agent_id`, `task_id`
   - `tool_name`, `tool_phase` (`pre`/`post`)
   - `event_ts`, `latency_ms`, `success`, `error_code`
   - `input_hash`, `output_hash`, `trace_id`
4. Enforce sanitization before persistence (strip secrets/PII from params/output summary).
5. Use fallback queue only if DB busy; flush queue deterministically.

```ts
export async function onToolPost(event: ToolPostEvent): Promise<void> {
  const row = normalizeObservation(event); // sanitize + hash + trace_id
  await db.insertObservation(row);         // no silent drop
}
```

### CRITERIO DE ACEITACAO

- For a test session with 100 tool calls, `observations` contains exactly expected `pre/post` rows.
- Every `post` can be joined to a `pre` by `trace_id`.
- No raw secrets appear in stored params/results.
- Failure to persist observation is surfaced (log + metric), not ignored.

### RISCOS

- **Risk:** high write volume in long runs.  
  **Mitigation:** WAL mode + prepared statements + batched flush.
- **Risk:** sensitive data leakage in logs.  
  **Mitigation:** sanitizer + denylist + integration tests with synthetic secrets.
- **Risk:** missing events on crash.  
  **Mitigation:** append-only queue and replay on startup.

## 2) Pattern detection (threshold: 20+ observations -> Haiku analysis)

### O QUE E

A periodic/background analyzer that turns raw observations into candidate behavioral patterns.

### O QUE NAO E

- Not real-time LLM call on every observation.
- Not single-session overfitting.
- Not static regex-only heuristics without evidence scoring.

### COMO IMPLEMENTAR

1. Scheduler scans projects with enough new data since last analysis.
2. Gate by `minObservations` **and** `minUniqueSessions`.
3. Build compacted evidence bundle (counts, success rates, latency deltas, contradiction events).
4. Ask Haiku for candidate instincts in strict JSON schema.
5. Reject low-signal candidates before writing instincts.

```ts
if (window.totalObs < cfg.analysis.minObservations) return;
if (window.uniqueSessions < cfg.analysis.minUniqueSessions) return;
const candidates = await analyzeWithHaiku(window);
await persistQualifiedCandidates(candidates);
```

### CRITERIO DE ACEITACAO

- Analyzer does not run below configured evidence floor.
- Analyzer produces deterministic JSON payload schema.
- Re-running analyzer over same immutable window yields stable candidates.
- Analysis frequency respects cooldown and budget caps.

### RISCOS

- **Risk:** noisy patterns become instincts.  
  **Mitigation:** session-diversity gate + support threshold + contradiction checks.
- **Risk:** too many analyses increase cost.  
  **Mitigation:** cooldown + delta-based windows + max candidates.
- **Risk:** fixed threshold misfit across projects.  
  **Mitigation:** per-project configurable threshold (default 20).

## 3) Instinct generation with confidence scores (0.3-0.85)

### O QUE E

Conversion of qualified pattern candidates into atomic, scoped instincts with bounded confidence and metadata.

### O QUE NAO E

- Not unconstrained confidence values.
- Not global knowledge by default.
- Not opaque free-text recommendations without machine-usable fields.

### COMO IMPLEMENTAR

1. Define typed instinct schema:
   - `id`, `project_id`, `domain`, `statement`
   - `confidence_raw`, `confidence_effective`
   - `support_count`, `support_sessions`, `last_evidence_at`
   - `status` (`active`, `inactive`, `promoted`, `archived`)
2. Compute confidence with bounded formula:

```ts
const base = 0.30;
const supportBoost = Math.min(0.25, Math.log2(supportCount + 1) * 0.06);
const consistencyBoost = consistencyRatio * 0.20;
const sessionBoost = Math.min(0.10, supportSessions * 0.02);
const confidence = clamp(base + supportBoost + consistencyBoost + sessionBoost, 0.30, 0.85);
```

3. Store rationale/evidence references (`observation_ids`) for auditability.
4. Keep instincts project-scoped unless explicit promotion occurs.

### CRITERIO DE ACEITACAO

- All new instincts persist with confidence inside `[0.30, 0.85]`.
- Every instinct links to supporting evidence references.
- Duplicate instinct statements are merged/upserted, not duplicated.
- Confidence computation is deterministic and unit-tested.

### RISCOS

- **Risk:** overconfident instincts from narrow evidence.  
  **Mitigation:** cap confidence and require multi-session evidence.
- **Risk:** duplicate semantic instincts.  
  **Mitigation:** canonicalization + similarity merge pass.
- **Risk:** brittle natural-language statements.  
  **Mitigation:** structured fields (`domain`, `conditions`, `action`) + statement linting.

## 4) Instinct decay when contradicted by evidence

### O QUE E

Lifecycle mechanism that continuously adjusts instinct confidence with reinforcement, contradiction penalties, and time decay.

### O QUE NAO E

- Not binary keep/delete logic only.
- Not immediate hard deletion without audit trail.
- Not decay that ignores recent positive evidence.

### COMO IMPLEMENTAR

1. Update on new evidence:
   - success aligned with instinct -> `+reinforceDelta`
   - contradiction -> `-contradictionDelta`
2. Apply time decay (half-life):

```ts
const decayFactor = Math.pow(0.5, ageDays / cfg.instinct.decayHalfLifeDays);
effectiveConfidence = rawConfidence * decayFactor;
```

3. Transition rules:
   - `< deleteBelow` -> `inactive` (soft state)
   - periodic cleanup can archive after retention window.
4. Persist decay events in audit trail (`instinct_events` or equivalent log stream).

### CRITERIO DE ACEITACAO

- Contradictory evidence lowers confidence predictably.
- Reinforcement can recover confidence if recent evidence supports behavior.
- Instincts below floor are not injected into runtime context.
- Lifecycle transitions are queryable and explainable.

### RISCOS

- **Risk:** oscillation from noisy outcomes.  
  **Mitigation:** smoothing window + minimum event count before major state change.
- **Risk:** permanent stale instincts.  
  **Mitigation:** half-life + archival policy.
- **Risk:** aggressive deletion of useful but rare instincts.  
  **Mitigation:** soft-inactive first, archive later, allow manual restore.

## 5) Instinct promotion to project-level knowledge

### O QUE E

A controlled pipeline that promotes mature instincts into durable project knowledge (`entities`/`relations`) and optional skill-level abstractions.

### O QUE NAO E

- Not automatic global promotion for every high-confidence instinct.
- Not bypassing context-curator review logic.
- Not replacing instinct table; promotion complements it.

### COMO IMPLEMENTAR

1. Promotion eligibility:
   - `confidence_effective >= minConfidence`
   - `support_sessions >= minSupportingSessions`
   - `support_count >= minSupportingObservations`
   - no severe contradiction in recent window
2. Promotion transaction:
   - create/update entity (`type=pattern`/`practice`)
   - create relation `derived_from` instinct
   - mark instinct `promoted=true` and keep link
3. Optional human gate (`requireHumanApproval`) via TUI `[P]romote`.
4. Keep reversible metadata to demote if later contradicted.

### CRITERIO DE ACEITACAO

- Promotion only occurs when all configured gates pass.
- Promoted knowledge appears in `entities` and is retrievable by curator.
- Every promoted entity can be traced back to source instinct and observations.
- Demotion path exists and is tested.

### RISCOS

- **Risk:** knowledge-base pollution.  
  **Mitigation:** strict gates + curator policy + demotion support.
- **Risk:** irreversible bad promotion.  
  **Mitigation:** provenance metadata + reversible status.
- **Risk:** promotion storms in high-activity sessions.  
  **Mitigation:** rate limits and batch approval mode.

## 6) Session summary generation on completion

### O QUE E

High-signal session distillation persisted to `sessions`, capturing completed work, unresolved items, learned instincts, contradictions, and next actions.

### O QUE NAO E

- Not raw full transcript dump.
- Not chain-of-thought persistence.
- Not one-pass lossy summarization without structure.

### COMO IMPLEMENTAR

1. Trigger on session stop (and pre-compact checkpoint if needed).
2. Use two-pass summarization:
   - Pass A (recall): collect all critical events.
   - Pass B (precision): remove low-signal noise/tool verbosity.
3. Persist structured sections:
   - `completed`, `failed_attempts`, `open_risks`, `next_steps`
   - `key_files`, `instinct_updates`, `promotion_events`
4. Store summary quality metadata (`source_events_count`, `compression_ratio`, `token_budget`).

```md
### Completed
### Failed Attempts
### Instinct Updates
### Open Risks
### Next Session Bootstrap
```

### CRITERIO DE ACEITACAO

- Summary is generated for every completed session.
- Summary always includes unresolved items and bootstrap actions.
- Random audit of summaries shows no critical decision loss.
- Summaries are small enough for startup injection budget.

### RISCOS

- **Risk:** over-compression dropping crucial context.  
  **Mitigation:** recall-first strategy + quality checks.
- **Risk:** verbose summaries wasting context budget.  
  **Mitigation:** hard token cap + section-based compaction.
- **Risk:** accidental leakage of sensitive content.  
  **Mitigation:** same sanitizer policy used for observations.

## 7) Context loading: last 7 days of sessions on startup

### O QUE E

Selective startup hydration that loads recent high-value memory artifacts (sessions + active instincts) into runtime context.

### O QUE NAO E

- Not eager loading all historical sessions.
- Not loading stale/inactive instincts blindly.
- Not replacing just-in-time retrieval with monolithic prompt stuffing.

### COMO IMPLEMENTAR

1. On startup, fetch sessions from last `loadWindowDays` (default 7).
2. Rank summaries by relevance:
   - recency
   - overlap with current objective/files
   - unresolved-risk weight
3. Load top N summaries + active instincts only.
4. Keep references for on-demand deep retrieval (JIT), not full expansion.

```sql
SELECT * FROM sessions
WHERE project_id = ?
  AND ended_at >= datetime('now', '-7 days')
ORDER BY ended_at DESC
LIMIT ?;
```

### CRITERIO DE ACEITACAO

- Startup context loads within token budget target.
- Loaded memory improves task continuity in integration tests.
- No inactive/low-confidence instincts are injected.
- Retrieval remains JIT for deeper history access.

### RISCOS

- **Risk:** startup prompt bloat.  
  **Mitigation:** top-N ranking + strict token budget.
- **Risk:** old summaries overriding current user intent.  
  **Mitigation:** explicit precedence rule: latest user instruction always wins.
- **Risk:** missing important but older context.  
  **Mitigation:** searchable archive references for manual or automatic pull-on-demand.
