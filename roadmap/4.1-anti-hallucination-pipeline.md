# 4.1 Anti-Hallucination Pipeline

> Phase 4 - Intelligence (Act 3: Resolution)

Defesa em 4 camadas para todo output + Chain-of-Verification (CoVe) seletivo para caminhos criticos.  
Princípio central: **verificacao proporcional ao risco**.

---

## 1) Estado da atividade e alinhamento arquitetural

### Status atual (diagnostico)

A versao anterior deste arquivo estava **incompleta para implementacao** (apenas checklist curto, sem contratos, fluxos, criterios e cobertura de risco).  
Esta versao converte a atividade 4.1 em especificacao tecnica executavel.

### Coerencia com `ARCHITECTURE.md`

Esta etapa implementa diretamente a **Decisao 8 (Anti-Alucinacao: 4 camadas + CoVe seletivo)** e se integra a:

- **Decisao 5 (Extended Squad)**: usa `reviewer` para L3.
- **Decisao 6 (Beat Sheet)**: usa metadados do beat sheet para classificar criticidade.
- **Decisao 4 (Merge Tier 4)**: marca conflitos semanticos de merge como candidatos a `critical: true`.
- **Decisao 12 (Audit)**: todos os passos de verificacao devem ser auditaveis.

Em resumo: 4.1 nao substitui planejamento, merge nem auditoria; ela adiciona um **gate de confiabilidade factual e de aderencia** dentro do loop de orquestracao.

---

## 2) Escopo da etapa

### O que esta etapa entrega

1. Pipeline de verificacao L1-L4 aplicado ao output de agentes (especialmente `builder`).
2. CoVe em 4 passos para saidas marcadas como criticas.
3. Criterio objetivo para marcar tarefa/output como `critical: true`.
4. Contratos de entrada/saida (tipos, status, reasons, metadados de auditoria).
5. Criterios de aceitacao e metricas operacionais.

### O que esta etapa NAO entrega

- Treino/fine-tuning de modelo (incluindo treino real de Self-RAG).
- Substituicao de guardrails de seguranca/prompt-injection (isso pertence a 4.3 e politicas de security).
- Criacao de novos frameworks de eval externos; usar infraestrutura existente.
- Resolver governanca de produto/compliance fora do escopo tecnico da verificacao.

```ts
export type VerificationLayer = "L1" | "L2" | "L3" | "L4" | "COVE";

export type VerificationStatus =
  | "pass"
  | "fail"
  | "retryable_fail"
  | "not_applicable";

export interface VerificationFinding {
  code:
    | "NO_EVIDENCE"
    | "OUT_OF_SCOPE_SOURCE"
    | "REQUIREMENT_MISMATCH"
    | "TEST_FAILURE"
    | "UNSUPPORTED_CLAIM"
    | "CRITICAL_POLICY_TRIGGER";
  message: string;
  evidence?: string[];
}

export interface VerificationResult {
  layer: VerificationLayer;
  status: VerificationStatus;
  findings: VerificationFinding[];
  metadata: Record<string, string | number | boolean>;
}

export interface VerificationDecision {
  accepted: boolean;
  requiresHumanReview: boolean;
  requiresRetry: boolean;
  results: VerificationResult[];
}
```

## 6) Fluxo de execucao (alto nivel)

```txt
agent output
  -> classify criticality
  -> L1 (prompt policy check / source policy)
  -> L2 (quote-first, se document-heavy)
  -> L3 (reviewer evaluator-optimizer, max 3)
  -> L4 (tests gate, quando aplicavel)
  -> CoVe (somente se critical=true)
  -> decision: accept | retry | escalate human
```

Regras de curto-circuito:

- Falha nao-retryable em L1/L2/L3/L4: rejeita output.
- Falha retryable: retorna para novo ciclo do builder com feedback estruturado.
- Falha em CoVe de output critico: `requiresHumanReview = true`.

## 7) Especificacao detalhada por item do checklist

## 7.1 L1 - Prompt design templates ("nao sei" + restricao de fonte)

### O QUE e

Camada de design de prompt que:

1. permite abstencao explicita ("nao sei"/"informacao insuficiente");
2. restringe o modelo a fontes autorizadas;
3. exige citacao/evidencia para afirmacoes factuais.

### O QUE NAO e

- Nao e classificador de seguranca completo.
- Nao e substituto de quote-first (L2) para documentos longos.
- Nao e CoVe (nao faz verificacao independente por claim).

### COMO implementar

Template base (system/developer prompt) com blocos fixos:

```xml
<source_policy>
Use APENAS informacoes de: {{allowed_sources}}.
Nao use conhecimento geral fora das fontes fornecidas.
</source_policy>

<uncertainty_policy>
Se a evidencia for insuficiente, responda explicitamente:
"Nao tenho informacao suficiente nas fontes fornecidas."
</uncertainty_policy>

<citation_policy>
Toda afirmacao factual deve referenciar evidencia.
Sem evidencia => nao afirmar.
</citation_policy>
```

Implementar um validador sintatico simples para confirmar que o output:

- contem citacoes quando requerido;
- nao usa frases proibidas de certeza sem suporte;
- inclui abstencao quando faltam evidencias.

### CRITERIO DE ACEITACAO

- Prompt template centralizado e reutilizavel por orchestrator/reviewer.
- 100% dos prompts de verificacao incluem `source_policy` + `uncertainty_policy`.
- Casos sem evidencia resultam em abstencao explicita, nao em invencao.

### RISCOS e mitigacao

- **Risco:** over-refusal (modelo recusa demais).  
  **Mitigacao:** calibrar exemplos few-shot de respostas validas.
- **Risco:** falso senso de grounding.  
  **Mitigacao:** obrigar evidencia estruturada e validar no L3.

## 7.2 L2 - Quote-first para tarefas document-heavy

### O QUE e

Pipeline em dois passos para tarefas orientadas a documento:

1. extrair quotes verbatim relevantes;
2. responder apenas com base nesses quotes.

### O QUE NAO e

- Nao e RAG completo com indexacao vetorial.
- Nao e sumarizacao generica livre.
- Nao deve responder com inferencia sem quote.

### COMO implementar

Ativar L2 quando:

- tarefa marcada como `document_heavy = true`, ou
- contexto documental acima de threshold (ex.: > 20k tokens), ou
- requisito explicito de grounding documental.

Contrato sugerido:

```json
{
  "quotes": [
    { "text": "...", "source": "file.md#L120-L130" }
  ],
  "answer": "...",
  "no_evidence": false
}
```

Regras:

- `answer` so pode usar informacoes presentes em `quotes`.
- `quotes` vazio => `no_evidence = true` e resposta de abstencao.
- Sempre persistir `source` para auditoria.

### CRITERIO DE ACEITACAO

- Em tarefas document-heavy, 100% das respostas incluem bloco de quotes.
- Se nao houver quote relevante, sistema retorna `NO_EVIDENCE` sem inventar.
- Auditoria consegue rastrear claim -> quote -> fonte.

### RISCOS e mitigacao

- **Risco:** quote irrelevante selecionado.  
  **Mitigacao:** reviewer (L3) valida relevancia e suporte.
- **Risco:** custo/latencia em documentos extensos.  
  **Mitigacao:** limitar top-k quotes + deduplicacao por source.

## 7.3 L3 - Reviewer agent loop (evaluator-optimizer)

### O QUE e

Loop de refinamento com agente `reviewer` separado avaliando:

- aderencia a requisitos;
- consistencia com evidencias;
- clareza/completeza do output.

Padrao: **evaluator-optimizer** com limite de 3 iteracoes.

### O QUE NAO e

- Nao e loop infinito de "melhoria subjetiva".
- Nao e code-review estilistica irrelevante.
- Nao substitui testes automatizados (L4).

### COMO implementar

Contrato do reviewer:

```ts
interface ReviewerVerdict {
  verdict: "PASS" | "FAIL_RETRYABLE" | "FAIL_HARD";
  feedback: string[];
  missingEvidence: string[];
  requirementMismatches: string[];
}
```

Pseudocodigo:

```ts
let attempt = 0;
let candidate = draft;
while (attempt < 3) {
  const verdict = await reviewerEvaluate(candidate, requirements, evidence);
  if (verdict.verdict === "PASS") break;
  if (verdict.verdict === "FAIL_HARD") return reject(verdict);
  candidate = await builderRevise(candidate, verdict.feedback);
  attempt += 1;
}
if (attempt === 3) return escalateHuman("L3_MAX_ITERATIONS");
```

### CRITERIO DE ACEITACAO

- Maximo de 3 iteracoes enforced por codigo.
- Feedback sempre estruturado (nao texto livre sem campos).
- Em falha hard ou max-iter, output nao e aceito silenciosamente.

### RISCOS e mitigacao

- **Risco:** reviewer e builder compartilham o mesmo vies.  
  **Mitigacao:** prompts separados e, quando possivel, modelo/config diferente.
- **Risco:** loop caro sem ganho.  
  **Mitigacao:** metricar ganho por iteracao e abortar em retorno decrescente.

## 7.4 L4 - Automated test gate

### O QUE e

Gate deterministico de testes antes de aceitar output de codigo.

### O QUE NAO e

- Nao cria suite de testes nova nesta etapa.
- Nao mascara falhas com fallback silencioso.
- Nao substitui revisao humana em mudancas criticas.

### COMO implementar

1. Resolver comando de teste oficial do repositorio (ex.: `npm test`, `vitest`, etc.).
2. Executar em ambiente do task/worktree.
3. Interpretar exit code + resumo.
4. Anexar resultado ao `VerificationResult` de L4.

Contrato:

```ts
interface TestGateResult {
  status: "pass" | "fail" | "not_applicable";
  command: string;
  exitCode?: number;
  summary: string;
}
```

`not_applicable` so e permitido com justificativa explicita (ex.: tarefa sem alteracao de codigo executavel).

### CRITERIO DE ACEITACAO

- Todo output de codigo passa por gate de teste ou justificativa `not_applicable`.
- Falha de teste bloqueia aceite automatico.
- Logs de teste ficam auditaveis.

### RISCOS e mitigacao

- **Risco:** testes flake geram falso negativo.  
  **Mitigacao:** politica de retry limitada e etiquetada como flake.
- **Risco:** tempo excessivo de pipeline.  
  **Mitigacao:** smoke tests no hot path + full suite em checkpoints.

## 7.5 CoVe seletivo para outputs criticos

### O QUE e

Chain-of-Verification de 4 passos para outputs `critical: true`:

1. gerar rascunho;
2. gerar perguntas de verificacao por claim;
3. responder perguntas **independentemente** do rascunho;
4. revisar rascunho com base nas verificacoes.

### O QUE NAO e

- Nao roda para todo output (custo/latencia proibitivos).
- Nao e Chain-of-Thought publico obrigatorio ao usuario.
- Nao substitui L3/L4; complementa para alto risco.

### COMO implementar

Separar chamadas:

- `planVerificationQuestions(draft)`  
- `answerQuestionIndependently(question, sources)` (sem acesso ao draft completo)
- `reviseWithVerification(draft, verifiedAnswers)`

Pseudocodigo:

```ts
const draft = await generateDraft(task);
const questions = await planVerificationQuestions(draft);
const verifiedAnswers = await Promise.all(
  questions.map((q) => answerQuestionIndependently(q, allowedSources))
);
const revised = await reviseWithVerification(draft, verifiedAnswers);
```

Regra obrigatoria: se claim critica ficar sem suporte, marcar `UNSUPPORTED_CLAIM` e bloquear aceite automatico.

### CRITERIO DE ACEITACAO

- CoVe e executado apenas quando `critical=true`.
- Etapa 3 e realmente independente (prompt/entrada separados do draft).
- Mudancas do rascunho para versao revisada ficam registradas.

### RISCOS e mitigacao

- **Risco:** pseudo-independencia (verificador contaminado pelo draft).  
  **Mitigacao:** isolamento de contexto na chamada da etapa 3.
- **Risco:** custo alto.  
  **Mitigacao:** gatilho seletivo por criticidade + limites de tamanho.

## 7.6 Criterios para `critical: true`

### O QUE e

Politica objetiva para ativar CoVe e regime de verificacao reforcado.

### O QUE NAO e

- Nao e decisao subjetiva ad hoc por agente.
- Nao depende de "feeling" textual sem regra.
- Nao deve marcar tudo como critico.

### COMO implementar

#### 1) Regras hard (critico automatico)

Marcar `critical: true` se qualquer condicao for verdadeira:

- toca autenticacao, autorizacao, criptografia, segredos, permissao;
- altera infraestrutura/deploy/config de producao;
- resolve conflito semantico de merge Tier 4;
- altera decisao arquitetural (interfaces centrais, contratos cross-agent, schema core);
- envolve risco de compliance/legal explicitamente citado.

#### 2) Score de risco (quando nao houver hard rule)

```txt
riskScore = impact(1-5) + irreversibility(0-3) + uncertainty(0-2)
critical = riskScore >= 7
```

Campos obrigatorios no beat sheet task:

```ts
interface CriticalityMeta {
  critical: boolean;
  criticalReasons: string[]; // enum reason codes
  riskScore?: number;
}
```

### CRITERIO DE ACEITACAO

- Toda tarefa possui `critical` explicitamente definido (true/false, sem null).
- Quando `critical=true`, possui `criticalReasons`.
- CoVe dispara automaticamente para todo output critico.

### RISCOS e mitigacao

- **Risco:** under-classification (deixar critico passar como normal).  
  **Mitigacao:** hard rules + auditoria de amostragem.
- **Risco:** over-classification (tudo critico).  
  **Mitigacao:** score + revisao periodica de thresholds.

## 8) Cobertura de cenarios de risco do orquestrador

| Cenario de risco | Camada principal | Cobertura | Risco residual |
|---|---|---|---|
| Resposta factual sem evidencia | L1 + L2 | Alta | Medio (se fonte ruim) |
| Interpretacao errada de documento longo | L2 + L3 | Alta | Medio |
| Desalinhamento com requisitos da tarefa | L3 | Alta | Baixo-medio |
| Codigo funcional mas quebrando comportamento | L4 | Alta | Medio (flake/coverage) |
| Mudanca critica (seguranca/deploy/merge Tier 4) | Critical + CoVe + L3/L4 | Alta | Medio |
| Claim critico sem suporte explicito | CoVe | Alta | Baixo |
| Prompt injection em fonte externa | Fora de 4.1 (4.3/security) | Parcial | Alto se nao houver guardrail externo |

Conclusao: as 4 camadas + CoVe seletivo cobrem os riscos nucleares de alucinacao e validacao de output do orquestrador; riscos de seguranca de entrada e governance ficam como responsabilidades complementares (4.3/auditoria e politicas de tool security).

## 9) Self-RAG e reflection tokens (adaptacao pragmatica)

Referencia tecnica: Self-RAG usa tokens RET/REL/SUP/USE com treino especifico.  
Nesta etapa, **nao faremos treino de modelo**, mas aplicaremos o principio via tags estruturadas no pipeline:

- `RET`: o agente precisa buscar evidencia adicional?
- `REL`: a evidencia recuperada e relevante?
- `SUP`: a claim esta suportada por evidencia?
- `USE`: a evidencia foi util para a resposta final?

Uso recomendado: incluir esses campos no output do reviewer/CoVe para melhorar rastreabilidade e decisao de retry.

## 10) Prompt templates anti-hallucination (baseline)

Template minimo recomendado:

```txt
Voce deve responder APENAS com base nas fontes fornecidas.
Para cada afirmacao factual, inclua evidencia (quote + source).
Se nao houver evidencia suficiente, responda literalmente:
"Nao tenho informacao suficiente nas fontes fornecidas."
Nao use conhecimento externo.
```

Template para tarefa documental:

```txt
Passo 1: extraia quotes verbatim relevantes.
Passo 2: responda usando apenas esses quotes.
Se nenhum quote suportar a resposta, marque NO_EVIDENCE.
```

Template evaluator (L3):

```txt
Avalie o output contra requisitos e evidencias.
Responda com PASS, FAIL_RETRYABLE ou FAIL_HARD.
Inclua lista objetiva de mismatches e evidencias ausentes.
```

## 11) Observabilidade, metricas e criterios de pronto global

Metricas minimas:

- `%outputs_with_abstention_when_no_evidence`
- `%document_tasks_with_quote_block`
- `reviewer_pass_rate_first_try`
- `avg_l3_iterations`
- `test_gate_fail_rate`
- `%critical_outputs_with_cove`
- `unsupported_claim_rate_after_cove`

Criterio de pronto global da etapa 4.1:

1. Todos os 6 itens do checklist implementados com teste/validacao.
2. Logs auditaveis por camada (entrada, saida, findings, decisao).
3. Criticidade deterministica e reproduzivel.
4. Sem aceite silencioso de output sem evidencia em caminho critico.

---

## 12) Notas de custo e operacao

- Caminho normal (L1-L4): custo baixo/moderado por output.
- Caminho critico: CoVe adiciona chamadas extras; deve ser seletivo.
- Limites recomendados:
  - max 3 iteracoes em L3;
  - max 1 rodada CoVe por output (salvo override humano);
  - timeout e fail-fast em tools de teste.
