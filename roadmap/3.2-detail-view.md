## 1) Live log streaming from agent execution

**O QUE e**
- Painel de log em tempo real do **card selecionado** no Kanban, com atualizacao continua durante execucao do agent.
- Deve mostrar timestamp, severidade/tipo da mensagem e conteudo (progress, info, warn, error, escalation).
- Deve suportar modo "follow tail" (auto-scroll no fim) e scroll manual do historico.

**O QUE NAO e**
- Nao e o agregador global de logs de todos os agents (isso pertence a 3.4 `[L]ogs`).
- Nao e exportador de logs para arquivo.
- Nao e viewer de auditoria completa (`audit_log` da fase 4.3).

**COMO implementar**
1. **Fonte de dados (SQLite / arquitetura):**
   - Ler `messages` filtrando por `task_id` + tipos relevantes (`task_progress`, `escalation`, `task_done`, `merge_result`).
   - Polling inicial: 250-500ms (suficiente para TUI responsiva sem saturar render).
2. **Modelo de estado recomendado:**
   - `lines: LogLine[]` (ring buffer com limite, ex.: 2000 linhas).
   - `followTail: boolean`.
   - `scrollOffset: number` (0 = fim).
3. **Scroll + auto-scroll (best practice Ink):**
   - Usar `useInput` para `up/down/pageUp/pageDown/home/end`.
   - Se usuario rolar para cima, desligar `followTail`.
   - Se voltar ao fim (`End`), reativar `followTail`.
4. **Renderizacao:**
   - Lista windowed (`slice`) para viewport visivel.
   - `Text wrap="truncate-end"` para evitar quebra destrutiva.
   - Cor por severidade (`error=red`, `warn=yellow`, `info=white`, `progress=cyan`).

```tsx
type LogLine = {
  id: string;
  ts: string;
  level: 'progress' | 'info' | 'warn' | 'error' | 'escalation';
  message: string;
};

function visibleWindow(lines: LogLine[], height: number, scrollOffset: number) {
  const end = Math.max(lines.length - scrollOffset, 0);
  const start = Math.max(end - height, 0);
  return lines.slice(start, end);
}
```

**CRITERIO DE ACEITACAO**
- Novas linhas aparecem em tempo real sem travar o input.
- Scroll manual funciona e pausa auto-scroll.
- `End` retorna ao modo follow-tail.
- Com 2k linhas no buffer, UI permanece usavel (sem flicker severo).

**RISCOS**
- **Risco:** re-render excessivo e flicker em terminais lentos.  
  **Mitigacao:** throttle de polling + windowing + limite de buffer.
- **Risco:** perda de navegacao do usuario por auto-scroll agressivo.  
  **Mitigacao:** follow-tail explicito, desligado ao primeiro scroll manual.

## 2) Diff preview (files changed by agent)

**O QUE e**
- Preview de mudancas do agent selecionado no formato diff unificado, com codificacao de cor por tipo de linha.
- Exibir cabecalho de arquivo, hunks (`@@`) e linhas adicionadas/removidas/contexto.

**O QUE NAO e**
- Nao e merge/conflict resolver (fase 2.4).
- Nao e visualizador completo de review por comentarios inline.
- Nao e editor de codigo dentro da TUI.

**COMO implementar**
1. **Pre-requisito funcional:** usar o tracking de mudancas da 1.2 (arquivos criados/modificados/deletados).
2. **Geracao do patch:**
   - `git --no-pager diff --no-color --unified=3 <baseRef>..<headRef> -- <file>`.
   - Parsear linhas do diff e classificar por prefixo (`+`, `-`, `@@`, `diff --git`, `index`).
3. **Renderizacao terminal (best practice):**
   - `+` em verde, `-` em vermelho, `@@` em ciano, headers em negrito/dim.
   - Preservar marcadores `+/-` (nao usar apenas cor para semantica).
4. **Controle de volume:**
   - Limite por arquivo (ex.: 400 linhas renderizadas).
   - Mostrar aviso: "diff truncado" com total real.
   - Navegacao por arquivo e por hunk.

```tsx
function colorForDiffLine(line: string): {color?: string; dimColor?: boolean} {
  if (line.startsWith('+++') || line.startsWith('---')) return {color: 'blue'};
  if (line.startsWith('@@')) return {color: 'cyan'};
  if (line.startsWith('+')) return {color: 'green'};
  if (line.startsWith('-')) return {color: 'red'};
  return {dimColor: true};
}
```

**CRITERIO DE ACEITACAO**
- Mudancas de pelo menos 3 arquivos sao navegaveis sem perda de responsividade.
- Linhas adicionadas/removidas/contexto ficam visualmente distintas e legiveis.
- Diffs grandes sao truncados de forma previsivel e transparente.

**RISCOS**
- **Risco:** diffs muito grandes degradam render.  
  **Mitigacao:** truncamento + paginacao/hunks + lazy render por arquivo.
- **Risco:** ambiguidade visual para daltomismo.  
  **Mitigacao:** manter prefixos `+/-` e headers textuais, nao depender apenas de cor.

## 3) Metrics panel: tokens used, context %, cost, elapsed

**O QUE e**
- Painel de metricas operacionais da tarefa selecionada: tokens de entrada/saida, total, contexto usado (%), custo estimado e tempo decorrido.
- Atualizacao continua junto com o log.

**O QUE NAO e**
- Nao e dashboard global de custos por fase/agent (fase 3.4 `[C]ost`).
- Nao e faturamento oficial do provedor (e estimativa orientativa de runtime).

**COMO implementar**
1. **Contrato de dados minimo por task:**
   - `inputTokens`, `outputTokens`, `contextUsedTokens`, `contextWindowTokens`, `startedAt`, `updatedAt`, `model`.
2. **Calculo de custo:**
   - `cost = (inputTokens/1e6 * inputRate) + (outputTokens/1e6 * outputRate)`.
   - Tabela de rates centralizada (`src/cost/modelRates.ts`).
3. **Layout recomendado:**
   - Blocos compactos de 1 linha por metrica (facil scan no terminal).
   - Destaque visual para `context %` e `elapsed`.
4. **Fallbacks explicitos:**
   - Se faltar metrica: exibir `N/A` (nao inventar zero silencioso).

```ts
type ModelRate = {inputPerM: number; outputPerM: number};
function estimateCostUSD(input: number, output: number, rate: ModelRate) {
  return (input / 1_000_000) * rate.inputPerM + (output / 1_000_000) * rate.outputPerM;
}
```

**CRITERIO DE ACEITACAO**
- Tokens, custo e elapsed atualizam em runtime para tarefa em `Running`.
- Ao finalizar tarefa, valores permanecem estaveis (snapshot final).
- Nenhuma metrica critica desaparece silenciosamente (N/A visivel quando ausente).

**RISCOS**
- **Risco:** divergencia entre custo estimado e custo real do provider.  
  **Mitigacao:** rotular como "estimated", guardar formula/rates em configuracao versionada.
- **Risco:** telas poluidas com excesso de numeros.  
  **Mitigacao:** limitar a metricas decisorias e mover detalhes para aba `[C]ost`.

## 4) Context usage bar (visual)

**O QUE e**
- Barra visual dedicada ao uso de contexto da tarefa selecionada (`used / window` + percentual).
- Deve incluir thresholds de risco para orientar decisao humana.

**O QUE NAO e**
- Nao executa compactacao automaticamente.
- Nao altera prompt/contexto do agent por conta propria.

**COMO implementar**
1. **Componente dedicado:** `ContextUsageBar.tsx`.
2. **Escala e semaforo sugeridos:**
   - `<= 70%`: verde (seguro)
   - `71-85%`: amarelo (atencao)
   - `> 85%`: vermelho (risco alto)
3. **Texto sempre junto da barra:**
   - Ex.: `78% (156k / 200k tokens)`.
4. **Acessibilidade terminal:**
   - Barra ASCII/Unicode (`█░`) + texto numerico.
   - Nunca depender somente de cor.

```tsx
function contextColor(pct: number) {
  if (pct > 85) return 'red';
  if (pct > 70) return 'yellow';
  return 'green';
}
```

**CRITERIO DE ACEITACAO**
- Percentual e numeros absolutos aparecem juntos.
- Mudanca de faixa (verde/amarelo/vermelho) ocorre automaticamente.
- Em terminais sem cor, a leitura numerica continua suficiente.

**RISCOS**
- **Risco:** indicador pouco confiavel por dados incompletos.  
  **Mitigacao:** exibir origem e timestamp da ultima medicao.
- **Risco:** "falso alarme" frequente em tarefas longas.  
  **Mitigacao:** usar thresholds claros + tendencia (subindo/estavel/caindo) no resumo.

## 5) ESC to return to Kanban

**O QUE e**
- Navegacao de volta Detail -> Kanban via `ESC`, preservando contexto de selecao no board.

**O QUE NAO e**
- Nao encerra a aplicacao.
- Nao cancela tarefa em execucao.

**COMO implementar**
1. `useInput` no container do Detail para capturar `key.escape`.
2. Ao retornar, restaurar:
   - `selectedTaskId`
   - coluna/scroll anterior no Kanban.
3. Prioridade de ESC:
   - se houver modal local aberto no Detail, ESC fecha modal;
   - sem modal, ESC volta para Kanban.

```tsx
useInput((_input, key) => {
  if (!key.escape) return;
  if (isModalOpen) return closeModal();
  navigate({view: 'kanban', restoreTaskId: taskId});
});
```

**CRITERIO DE ACEITACAO**
- ESC sempre retorna ao Kanban em no maximo 1 tecla (quando sem modal).
- Card anteriormente aberto volta focado no board.
- Nenhum efeito colateral no runtime do agent.

**RISCOS**
- **Risco:** perda de foco/selecionado ao voltar.  
  **Mitigacao:** estado de navegacao centralizado com `selectedTaskId`.
- **Risco:** conflitos de shortcut com outros componentes.  
  **Mitigacao:** prioridade de input por foco ativo.

## 6) Transition animation Kanban -> Detail View

**O QUE e**
- Transicao curta para orientar o operador na troca de tela (ENTER no card -> Detail).
- Animacao deve ser **discreta e opcional**, sem comprometer performance.

**O QUE NAO e**
- Nao e framework de animacao complexo estilo web.
- Nao exige 60fps constante no terminal.

**COMO implementar**
1. **Estrategia recomendada para Ink:**
   - Estado de transicao com poucos frames (ex.: 3-5 passos, 120-200ms total).
   - Fallback para troca instantanea se terminal lento.
2. **Feature flag / preferencia:**
   - `ui.transitions = "auto" | "off" | "on"`.
3. **Performance:**
   - Limitar fps efetivo (ex.: 15-20 durante transicao).
   - Evitar re-render de subcomponentes pesados durante animacao (`React.memo`).

```ts
type ViewState = 'kanban' | 'transitioning' | 'detail';
// transitioning dura poucos ticks e depois monta DetailView real
```

**CRITERIO DE ACEITACAO**
- Com transicao ligada, troca de tela permanece responsiva e sem flicker forte.
- Com transicao desligada, comportamento continua correto.
- Em ambiente lento, modo `auto` cai para instantaneo.

**RISCOS**
- **Risco:** flicker em terminais menos capazes.  
  **Mitigacao:** animacao curta + fallback automatico + possibilidade de desativar.
- **Risco:** complexidade desnecessaria para pouco ganho.  
  **Mitigacao:** manter implementacao minima e isolada em hook/utilitario.

## 7) Intervention-readiness summary (sinalizacao para decisao humana)

**O QUE e**
- Bloco sintetico no Detail View dizendo se o humano deve monitorar, aguardar ou intervir.
- Foco em **decisao**, nao em executar a acao (acoes pertencem a 3.3).

**O QUE NAO e**
- Nao executa `[S]teer`, `[F]ollow-up`, `[A]bort`, `[P]romote`.
- Nao substitui julgamento humano.

**COMO implementar**
1. **Sinais minimos (com severidade):**
   - heartbeat atrasado (ex.: sem `task_progress` por > N segundos),
   - contexto > 85%,
   - erro recente/escalation aberta,
   - retries acima do limite,
   - diff muito grande/arriscado (heuristica simples).
2. **Resumo textual curto:**
   - Ex.: `Status: ATENCAO — heartbeat 45s atrasado, contexto 88%`.
3. **Hints de proxima acao (somente orientacao):**
   - Exibir teclas da 3.3 como dica: `[S] [F] [A]`.

```ts
type InterventionLevel = 'ok' | 'watch' | 'act-now';
// derivado de thresholds de contexto, heartbeat e erros recentes
```

**CRITERIO DE ACEITACAO**
- Operador consegue decidir em segundos se intervem ou nao.
- Quando houver escalation real, resumo sinaliza no mesmo ciclo de polling.
- Sem sinais criticos, resumo permanece "OK" de forma estavel (sem oscillation ruidosa).

**RISCOS**
- **Risco:** muitos alertas falsos (fadiga).  
  **Mitigacao:** thresholds calibrados + histerese simples para evitar piscar de status.
- **Risco:** resumo esconder detalhes importantes.  
  **Mitigacao:** resumo sempre acompanhado de links visuais para log/diff/metricas.
