# 2.2 Orchestrator Loop

> Phase 2 — Orchestration (Act 2: Confrontation)

The showrunner's brain. The main loop that decomposes tasks, assigns them to agents, monitors progress, and advances the beat sheet.

## Máquina de estados obrigatória do loop (cobertura completa)

Para evitar lacunas entre decompose/assign/monitor/collect/merge, o loop deve ser modelado como uma FSM explícita.

| Estado | Disparo de entrada | Ação principal | Saída possível |
|---|---|---|---|
| `DECOMPOSE` | novo objetivo ou checkpoint exige replanning | gerar/atualizar tarefas atômicas e dependências | `ASSIGN`, `ESCALATED`, `FAILED` |
| `ASSIGN` | há tarefas `ready` e capacidade disponível | selecionar agente por role + score + fairness | `MONITOR`, `ASSIGN` (mais tarefas), `ESCALATED` |
| `MONITOR` | existem agentes `running` | polling de mensagens + health checks | `COLLECT`, `ESCALATED`, `MONITOR` |
| `COLLECT` | chegou `task_done`, `task_progress`, `escalation` ou timeout | consolidar resultado, atualizar estado de tarefa | `MERGE`, `ASSIGN`, `ESCALATED`, `FAILED` |
| `MERGE` | tarefa marcada para integração | enfileirar/chamar 1.3 (Tier 1-2 agora; 3-4 em 2.4) | `ADVANCE_BEAT`, `ASSIGN`, `ESCALATED` |
| `ADVANCE_BEAT` | bloco/beat concluído | mover ponteiro beat/sequence/act + checkpoint | `DECOMPOSE`, `ASSIGN`, `COMPLETED` |
| `ESCALATED` | bloqueio sem resolução automática | registrar incidente + pausar escopo afetado | `ASSIGN` (após resolução), `FAILED` |
| `FAILED` | política de retry esgotada ou erro fatal | persistir diagnóstico e encerrar ciclo afetado | terminal parcial/global |
| `COMPLETED` | objetivo global entregue | flush final e handoff | terminal |

### Invariantes obrigatórias

1. **Sem estado implícito em memória volátil**: progresso do loop é recuperável por SQLite após crash.
2. **Idempotência de consumo**: mesma mensagem não pode causar transição dupla.
3. **Sem merge paralelo no `main`**: merge serializado por mutex/semaphore.
4. **Sem bloqueio do event loop**: nada de `spawnSync`, loops CPU-bound longos, ou polling com callback sobreposto.
5. **Sem falha silenciosa**: toda exceção gera evento auditável (`escalation` ou `error` estruturado).

## Contratos de dados mínimos para 2.2

> Esta etapa depende da 0.2 para schema base; aqui define apenas os contratos que o loop exige para funcionar.

### `messages` (uso operacional pelo loop)

Campos mínimos esperados:
- `id` (monotônico, usado como watermark)
- `type` (`task_assigned`, `task_progress`, `task_done`, `merge_ready`, `merge_result`, `escalation`, `health_check`, `broadcast`)
- `task_id`, `agent_id`, `correlation_id`
- `payload_json` (JSON tipado)
- `created_at`

Consultas base:

```sql
SELECT id, type, task_id, agent_id, correlation_id, payload_json, created_at
FROM messages
WHERE id > :last_seen_id
  AND type IN ('task_progress','task_done','merge_result','escalation')
ORDER BY id ASC
LIMIT :batch_size;
```

### Watermark de consumo

Persistir em `beat_state` (ou tabela dedicada) o `last_seen_message_id` para replay seguro após reinício.

### Índices recomendados

```sql
CREATE INDEX IF NOT EXISTS idx_messages_type_id ON messages(type, id);
CREATE INDEX IF NOT EXISTS idx_messages_task_id ON messages(task_id, id);
CREATE INDEX IF NOT EXISTS idx_messages_agent_id ON messages(agent_id, id);
```

## Design detalhado por item do checklist

## 1) Main orchestrator loop: decompose → assign → monitor → collect → merge

### O QUE é
Loop principal não-bloqueante que coordena o ciclo de vida completo das subtarefas, com transições explícitas e checkpoints.

### O QUE NÃO é
- Não é um while infinito sem estado.
- Não é execução síncrona de um agente por vez.
- Não é responsável por resolver conflito de merge (isso é da 1.3/2.4).

### COMO implementar
1. Implementar um `tick()` idempotente.
2. Dirigir transições por estado persistido (FSM).
3. Usar `setTimeout` recursivo (não `setInterval`) para evitar ticks sobrepostos.
4. Tratar erro por fase com classificação `retryable` vs `fatal`.

```ts
export async function runOrchestratorLoop(ctx: LoopContext, signal: AbortSignal) {
  while (!signal.aborted) {
    const startedAt = Date.now();
    try {
      await tick(ctx, signal);
    } catch (err) {
      await ctx.escalations.raiseLoopError(err);
    }
    const delayMs = ctx.backpressure.nextDelayMs(Date.now() - startedAt);
    await ctx.timers.sleep(delayMs, signal);
  }
}
```

### CRITÉRIO DE ACEITAÇÃO
- Loop executa ao menos 1 ciclo completo `DECOMPOSE → ASSIGN → MONITOR → COLLECT → MERGE`.
- Reinício do processo não duplica merge nem perde mensagem (watermark funcional).
- `AbortController` encerra loop de forma graciosa sem corromper estado.

### RISCOS
- **Loop overlapping** por timer mal configurado -> usar `setTimeout` recursivo.
- **Starvation por microtasks** (`process.nextTick` recursivo) -> evitar em path crítico.
- **Estado zumbi após crash** -> persistir `last_seen_message_id` e estados de tarefa.

## 2) Task assignment: match subtasks to agents by role

### O QUE é
Scheduler dependency-aware que escolhe qual subtarefa pronta vai para qual agente, respeitando papel, capacidade e fairness.

### O QUE NÃO é
- Não é alocação aleatória.
- Não é um algoritmo de otimização global caro (MILP/RL) nesta etapa.
- Não ignora dependências do beat sheet.

### COMO implementar
1. Construir conjunto `readyTasks` (dependências concluídas).
2. Filtrar agentes compatíveis por `role`.
3. Calcular score determinístico por par `task-agent`.
4. Aplicar aging para evitar starvation.
5. Alocar até `concurrencyCap`.

Score inicial recomendado:

```ts
score =
  roleMatch * 100 +
  priorityWeight * 30 +
  agingMinutes * 2 -
  retryCount * 10 -
  agentCurrentLoad * 5;
```

### CRITÉRIO DE ACEITAÇÃO
- Tarefa nunca é atribuída se dependência estiver pendente.
- Dois runs com mesmo estado geram atribuições equivalentes (determinismo).
- Nenhuma tarefa pronta fica indefinidamente sem tentativa (aging funcional).

### RISCOS
- **Starvation** de tarefas de baixa prioridade -> aging progressivo.
- **Sobrecarga de um agente** -> load penalty + per-role cap.
- **Coordination theater** (overhead > trabalho) -> limitar complexidade do scheduler e medir throughput.

## 3) Parallel execution: spawn multiple agents concurrently

### O QUE é
Execução concorrente controlada de vários agents, com limite global e limites por role.

### O QUE NÃO é
- Não é paralelismo ilimitado.
- Não é spawn sem rastreamento de PID/sessão.
- Não é concorrência no merge de `main`.

### COMO implementar
1. Introduzir semáforo global (`maxConcurrentAgents`, default recomendado: 5).
2. Introduzir limites por role (`builder=2`, `tester=1`, etc. configurável).
3. Registrar `agent_run_id`, `started_at`, `abort_controller`.
4. Garantir cleanup em `finally`.

```ts
await semaphore.acquire();
try {
  await runtime.spawn(task, { signal });
} finally {
  semaphore.release();
}
```

### CRITÉRIO DE ACEITAÇÃO
- Nunca excede `maxConcurrentAgents`.
- Cancelamento de um agente não bloqueia os demais.
- Falha de spawn gera `escalation` auditável e não derruba o loop inteiro.

### RISCOS
- **Backpressure inexistente** -> explosão de fila e memória.
- **Leak de processos** -> obrigar handler em `exit`, `close`, `error`.
- **Deadlock de recursos compartilhados** -> não compartilhar instâncias mutáveis sem lock.

## 4) Progress monitoring: poll SQLite for `task_progress` messages

### O QUE é
Monitoramento de progresso e heartbeat via polling de mensagens no SQLite WAL.

### O QUE NÃO é
- Não é busy-waiting agressivo.
- Não é depender de notificações nativas SQLite (não há mecanismo cross-process robusto).
- Não é interpretar ausência de mensagem como conclusão.

### COMO implementar
1. Polling incremental por `id > last_seen_message_id`.
2. Batch limitado (`LIMIT 100-500`).
3. Intervalo adaptativo:
   - `runningAgents > 0`: 250-500ms
   - `idle`: 1000-2000ms
4. Long-polling HTTP/SSE é opcional para TUI; não substitui polling interno do loop.
5. Monitorar tamanho de WAL e aplicar checkpoint controlado quando necessário.

Padrão de consulta:

```sql
SELECT id, type, task_id, agent_id, payload_json, created_at
FROM messages
WHERE id > :lastSeen
ORDER BY id
LIMIT :limit;
```

### CRITÉRIO DE ACEITAÇÃO
- Latência de detecção de progresso dentro do SLA interno definido (ex.: < 1s em modo ativo).
- Watermark sempre avança monotonicamente.
- Monitor continua funcional após restart (replay sem perda).

### RISCOS
- **Checkpoint starvation WAL** com leitores longos -> checkpoints periódicos (`wal_checkpoint(RESTART)`) sob limite de tamanho.
- **Polling muito curto** -> custo de CPU desnecessário.
- **Polling muito longo** -> UX ruim e detecção tardia de stuck.

## 5) Completion handling: trigger merge when `task_done` received

### O QUE é
Pipeline de finalização: ao receber `task_done`, validar output, registrar resultado e disparar integração via merge workflow.

### O QUE NÃO é
- Não é merge imediato sem validação mínima.
- Não é tratar `task_done` como sucesso incondicional.
- Não é bypass da fila FIFO de merge.

### COMO implementar
1. Validar contrato de `task_done` (task_id, artifacts, status).
2. Persistir transição de tarefa para `ready_for_merge`.
3. Emitir `merge_ready`.
4. Chamar módulo 1.3 para Tier 1-2.
5. Consumir `merge_result` e atualizar estado final.

```ts
if (msg.type === 'task_done') {
  await tasks.markReadyForMerge(msg.taskId, msg.payload);
  await messages.emit('merge_ready', { taskId: msg.taskId, agentId: msg.agentId });
}
```

### CRITÉRIO DE ACEITAÇÃO
- Cada `task_done` gera no máximo um `merge_ready` (idempotência).
- Resultado de merge (`success`/`conflict`/`failed`) reflete na tarefa e no beat.
- Falha de merge não perde artifacts e gera escalation quando necessário.

### RISCOS
- **Duplicação de merge** em retry/restart -> chave idempotente por `task_id + run_id`.
- **Condição de corrida** entre done e cancel -> regra de precedência de estado (cancel vence se timestamp anterior ao done commit).
- **Acoplamento indevido com 2.4** -> nesta etapa só Tier 1-2.

## 6) Escalation handling: respond to agent escalations

### O QUE é
Mecanismo estruturado para lidar com bloqueios técnicos/semânticos sem travar o sistema inteiro.

### O QUE NÃO é
- Não é log textual sem workflow.
- Não é pausar todo o orquestrador para qualquer erro pequeno.
- Não é esconder erro e seguir como se nada tivesse ocorrido.

### COMO implementar
1. Definir severidades (`low`, `medium`, `high`, `critical`).
2. Definir categorias (`missing_context`, `tool_failure`, `merge_conflict`, `dependency_deadlock`, `agent_crash_loop`).
3. Matriz de ação:
   - low: retry local + follow_up
   - medium: reroute para `debugger`/`researcher`
   - high/critical: pausar escopo afetado + notificar humano (TUI/CLI)
4. Persistir `escalation` com status (`open`, `acked`, `resolved`, `failed`).

### CRITÉRIO DE ACEITAÇÃO
- Toda escalation vira registro rastreável com owner/status.
- Escalation crítica pausa apenas escopo necessário (não o sistema inteiro por padrão).
- Após resolução, tarefa retorna para fluxo normal (`ASSIGN`/`MERGE`).

### RISCOS
- **Falso positivo em excesso** -> calibrar thresholds e categorias.
- **Escalation flood** -> rate limit + deduplicação por assinatura.
- **Sem dono claro** -> preencher `assigned_to` (humano/agent) no registro.

## 7) Health check: periodic pings to detect stuck agents

### O QUE é
Watchdog ativo para detectar processo vivo mas sem progresso (stuck), crash silencioso, ou atraso anormal.

### O QUE NÃO é
- Não é apenas checar PID.
- Não é timeout fixo sem considerar tipo de tarefa.
- Não é restart infinito.

### COMO implementar
1. Emitir `health_check` ping em intervalo fixo (ex.: 10s).
2. Exigir resposta em `task_progress` com `heartbeat_seq`.
3. Detectar stuck por ausência de heartbeat/progresso (`stuckAfterMs`, ex.: 45s).
4. Remediação em camadas:
   - camada 1: `steer`/follow-up solicitando status
   - camada 2: abort gracioso (`SIGTERM` + grace timeout)
   - camada 3: retry com backoff exponencial + jitter (limite de tentativas)
5. Após limite, abrir escalation `agent_crash_loop`.

Padrão Node.js:
- usar APIs assíncronas de `child_process` (`spawn`/`fork`), nunca `spawnSync`
- tratar eventos `message`, `exit`, `close`, `error`
- suportar cancelamento via `AbortController`

### CRITÉRIO DE ACEITAÇÃO
- Processo travado é detectado dentro de janela configurada.
- Retry respeita `maxRetries` e não entra em loop infinito.
- Após falha definitiva, sistema continua operando para outras tarefas.

### RISCOS
- **Timeout agressivo** mata tarefas válidas longas -> timeout por tipo de tarefa.
- **Timeout permissivo** atrasa recuperação -> SLO por role.
- **Reinício em tempestade** -> backoff + circuit breaker.

## 8) Beat sheet advancement: move to next beat/sequence/act

### O QUE é
Atualização formal do progresso narrativo/técnico quando tarefas de um beat estão concluídas e integradas.

### O QUE NÃO é
- Não é avanço baseado em “sensação”.
- Não é ignorar checkpoints (Catalyst, Midpoint, All Is Lost, Break Into Three, Final Image).
- Não é avançar com pendências críticas abertas.

### COMO implementar
1. Definir `isBeatComplete(beatId)`:
   - todas tarefas atômicas `done`
   - merges concluídos
   - sem escalations críticas abertas
2. Persistir transição em `beat_state`.
3. Avaliar checkpoint e decidir se precisa `DECOMPOSE` adicional.
4. Emitir evento para TUI/CLI com novo estado.

### CRITÉRIO DE ACEITAÇÃO
- Beat só avança com critérios objetivos satisfeitos.
- Checkpoints são registrados e auditáveis.
- Final do Act 2 dispara transição correta para fase subsequente do arco.

### RISCOS
- **Advance prematuro** -> gate estrito com verificação de merge+escalation.
- **Advance bloqueado para sempre** -> diagnóstico de dependência cíclica + escalation de deadlock.
- **Drift entre beat e tasks** -> transação única ao atualizar ambos.

## Padrões técnicos obrigatórios nesta etapa (pesquisa web + Context7)

1. **Event loop Node.js**
   - Não bloquear event loop.
   - Preferir callbacks/ticks curtos e trabalho pesado fora do loop principal.
   - Evitar abuso de `process.nextTick` que pode causar starvation.
2. **SQLite polling**
   - Polling incremental por watermark.
   - WAL mode habilitado.
   - Checkpoint policy explícita para evitar crescimento do `-wal`.
   - Reconhecer que SQLite não oferece push cross-process nativo robusto.
3. **Scheduling**
   - Ready queue baseada em DAG/dependências.
   - Prioridade + fairness (aging).
   - Limite de concorrência por recursos.
4. **Health check de child process**
   - Heartbeat IPC + timeouts + restart com limite.
   - Supervisão por eventos `exit/close/error`.
5. **Backpressure e concorrência**
   - Semáforo global/per-role.
   - Limites de fila.
   - Delay adaptativo do loop.
   - Abort explícito de trabalho em sobrecarga.
