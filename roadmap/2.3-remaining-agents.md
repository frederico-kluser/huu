## 4) Contrato técnico padrão para definição de agent

### 4.1 Estrutura mínima de definição (TypeScript)

```ts
export interface AgentDefinition {
  id:
    | "planner"
    | "tester"
    | "reviewer"
    | "researcher"
    | "merger"
    | "refactorer"
    | "doc-writer"
    | "debugger"
    | "context-curator";
  role: string;
  model: "haiku" | "sonnet" | "opus";
  tools: {
    allow: string[];   // allowlist explícita
    deny?: string[];   // denylist opcional para segurança adicional
  };
  inputContract: string[];     // quais artefatos recebe
  outputContract: string[];    // quais artefatos deve produzir
  completionCriteria: string[];// quando pode emitir task_done
  escalationPolicy: string[];  // quando abrir escalation
  systemPrompt: string;        // prompt por papel
}
```

### 4.2 Template base de prompt por papel (recomendado)

```xml
<background_information>
Você é o agente {ROLE}. Seu foco é {MISSION}.
</background_information>

<instructions>
- Execute somente tarefas dentro do seu papel.
- Se faltar informação crítica, emita escalation com perguntas objetivas.
- Não extrapole para responsabilidades de outros agentes.
</instructions>

<tool_guidance>
- Use apenas tools autorizadas para este papel.
- Prefira a menor ação necessária (least privilege).
- Registre decisões e evidências no output.
</tool_guidance>

<output_description>
- Entregue output estruturado no contrato definido.
- Inclua riscos, limitações e próximos handoffs.
</output_description>
```

### 4.3 Estratégia de model tiering (padrão operacional)

- **Haiku**: tarefas mecânicas, varredura, síntese simples, baixo risco.
- **Sonnet**: implementação e coordenação de trabalho principal (default operacional).
- **Opus**: análise crítica, debugging profundo, revisão de alto risco, segurança.

**Regras de upgrade para Opus** (trigger):

1. 1ª tentativa falhou com evidência de insuficiência de raciocínio.
2. Decisão arquitetural ou conflito semântico crítico.
3. Código/resultado security-critical.
4. Alto impacto e baixa tolerância a falso positivo.

### 4.4 Princípio de least privilege para tools

1. Cada agent recebe **apenas** tools necessárias ao papel.
2. Ferramentas de escrita devem ser negadas por default.
3. Reviewer é estritamente read-only.
4. Tools de rede/web só para papéis de pesquisa com justificativa.
5. Todas as chamadas de tool devem ser auditáveis no SQLite.

## 5) Checklist detalhado por agent (implementável)

### [ ] `planner` — Beat Sheet decomposition (Sonnet)

### O QUE é

Agente responsável por decompor objetivos em beat sheet fractal (objetivo → atos → sequências → tarefas atômicas), incluindo dependências e checkpoints narrativos (Catalyst, Midpoint, All Is Lost, Break Into Three, Final Image).

### O QUE NÃO é

- Não implementa código.
- Não executa testes.
- Não decide merge.

### COMO implementar

- **Arquivo**: `src/agents/definitions/planner.ts`
- **Modelo**: `sonnet`
- **Tools**: `Read`, `Grep`, `Glob`
- **Input esperado**:
  - objetivo da feature
  - contexto arquitetural relevante
  - restrições de escopo
- **Output obrigatório**:
  - JSON com atos, sequências, tarefas atômicas
  - grafo de dependências entre tarefas
  - critérios de aceitação por tarefa
  - flag `critical` para subtarefas de risco

Snippet de output recomendado:

```json
{
  "objective": "Implementar autenticação",
  "acts": [{ "id": "act-1", "sequences": [] }],
  "atomicTasks": [{ "id": "task-1", "ownerRole": "builder" }],
  "dependencies": [{ "from": "task-2", "to": "task-1" }],
  "checkpoints": ["Catalyst", "Midpoint"]
}
```

### CRITÉRIO DE ACEITAÇÃO

- Decomposição contém os 4 níveis hierárquicos.
- Toda tarefa atômica tem owner role e verificação.
- Dependências permitem identificar paralelização segura.
- Saída é parseável e persistível no `beat_state`.

### RISCOS

- **Over-decomposition** gerando overhead de coordenação.
  - Mitigação: limite máximo de granularidade e regra de valor mensurável por tarefa.
- **Ambiguidade de handoff** para outros agentes.
  - Mitigação: output schema obrigatório + validação no orchestrator.

### [ ] `tester` — TDD + test execution (Sonnet)

### O QUE é

Agente que valida comportamento via testes e ciclo red-green-refactor, garantindo que requisitos planejados estejam cobertos por testes automatizados executáveis.

### O QUE NÃO é

- Não deve fazer refatoração ampla de produção sem demanda explícita.
- Não é responsável por decisões arquiteturais.

### COMO implementar

- **Arquivo**: `src/agents/definitions/tester.ts`
- **Modelo**: `sonnet`
- **Tools (baseline)**: `Read`, `Bash(test)`
- **Tools opcionais e restritas** (se TDD estrito for habilitado): `Write`/`Edit` somente em `tests/**`
- **Input esperado**:
  - requisitos da tarefa
  - diff atual do builder
  - suíte de testes existente
- **Output obrigatório**:
  - relatório de testes (pass/fail)
  - gaps de cobertura por requisito
  - recomendação de correção quando falhar

### CRITÉRIO DE ACEITAÇÃO

- Toda execução gera relatório estruturado.
- Cada requisito possui ao menos um teste mapeado (quando aplicável).
- Falhas retornam diagnóstico reproduzível (com comando e evidência).

### RISCOS

- **Flaky tests** produzindo falso negativo/positivo.
  - Mitigação: rerun controlado + marcação explícita de flakiness.
- **Escopo excessivo** (tester começa a “implementar feature”).
  - Mitigação: prompt com limites rígidos de papel.

### [ ] `reviewer` — code review, read-only tools (Opus)

### O QUE é

Agente de revisão técnica de alta criticidade, focado em bugs, riscos de segurança, inconsistência lógica e impactos de manutenção. Opera exclusivamente em modo leitura.

### O QUE NÃO é

- Não edita código.
- Não resolve conflitos de merge.
- Não substitui testes automatizados.

### COMO implementar

- **Arquivo**: `src/agents/definitions/reviewer.ts`
- **Modelo**: `opus`
- **Tools**: `Read`, `Grep`, `Glob` (**sem** `Write`, `Edit`, `Bash`)
- **Padrão de revisão**:
  - checklist estruturado (corretude, segurança, testes, regressão)
  - severidade obrigatória (`CRITICAL`, `HIGH`, `MEDIUM`, `LOW`)
  - evidência por finding (arquivo/linha/racional)
- **Output obrigatório**:

```json
{
  "summary": "2 issues relevantes",
  "findings": [
    {
      "severity": "HIGH",
      "category": "security",
      "file": "src/auth.ts",
      "evidence": "token sem validação de expiração",
      "recommendation": "validar exp e issuer"
    }
  ],
  "approve": false
}
```

### CRITÉRIO DE ACEITAÇÃO

- Zero chamadas de tools de escrita em auditoria.
- Findings com severidade, evidência e recomendação acionável.
- Sinal alto: evitar comentários de estilo sem impacto.

### RISCOS

- **Ruído alto** (baixa precisão).
  - Mitigação: checklist de impacto real + threshold mínimo de evidência.
- **Falso senso de segurança**.
  - Mitigação: manter gate de testes e não substituir validação automatizada.

### O QUE é

Agente responsável por integrar branches/worktrees com segurança, aplicando estratégia progressiva de resolução e registrando decisões de merge.

### O QUE NÃO é

  - Mitigação: limites de heurística + fallback explícito.

  - Mitigação: proibir comandos destrutivos no prompt/policy.

### [ ] `context-curator` — post-activity memory curation (Haiku)

### O QUE é

Agente pós-atividade que decide o que entra, atualiza ou sai da memória central (SQLite), preservando sinal alto e evitando acúmulo cego de contexto.

### O QUE NÃO é

- Não roda durante execução ativa da tarefa (somente pós-atividade).
- Não modifica código de produto.
- Não substitui o orchestrator na decisão de fluxo.

### COMO implementar

- **Arquivo**: `src/agents/definitions/context-curator.ts`
- **Modelo**: `haiku`
- **Tools**: `Read`, `Grep`, `SQLite`
- **Input esperado**:
  - output final do agente executor
  - diff/artefatos relevantes
  - estado atual do scratchpad/memória
- **Output obrigatório**:
  - operações de memória propostas (`upsert`, `link`, `decay`, `discard`)
  - justificativa por operação
  - confiança da informação (ex.: `0.3` a `0.85` para instincts)

Prompt mínimo recomendado para curadoria:

```xml
<mission>
Você é o context-curator. Preserve conhecimento de alto valor e descarte ruído.
</mission>

<rules>
- Curar, não acumular.
- Registrar somente fatos verificáveis e úteis para tarefas futuras.
- Evitar duplicatas semânticas.
- Se evidência for fraca, marcar baixa confiança ou descartar.
</rules>

<memory_targets>
- entities, relations, observations, sessions, instincts
</memory_targets>
```

Implementação recomendada:

1. Normalizar artefatos de entrada.
2. Detectar novidade vs duplicação.
3. Aplicar política de retenção (observations 30 dias, sessions 7 dias carregadas).
4. Persistir com idempotência (chave `session_id + task_id + agent_id`).

### CRITÉRIO DE ACEITAÇÃO

- Curadoria ocorre após cada `task_done`.
- Sem duplicação desnecessária de memória.
- Atualizações no SQLite são rastreáveis e auditáveis.
- Recuperação de contexto melhora relevância nas tarefas seguintes.

### RISCOS

- **Poluição de memória** (dados irrelevantes).
  - Mitigação: critérios de entrada estritos + confidence scoring.
- **Over-pruning** (perda de contexto útil).
  - Mitigação: descarte progressivo + revisão por checkpoints.
- **Contaminação por output incorreto**.
  - Mitigação: priorizar fatos com evidência e validação cruzada.
