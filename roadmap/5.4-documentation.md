# 5.4 Documentation

> Phase 5 — Polish (Final Image)

Esta etapa transforma o projeto em algo usável e contribuível por terceiros sem suporte síncrono do time core.
O objetivo não é “ter docs”, e sim ter documentação operacional, verificável e alinhada à arquitetura do HUU.

---

## Objetivo da etapa

Entregar documentação para **dois públicos**:

1. **Developer-user (uso):** quer instalar, executar, entender fluxos e diagnosticar problemas.
2. **Developer-contributor (contribuição):** quer montar ambiente, alterar código, validar localmente e abrir PR.

---

## Coerência arquitetural obrigatória (ARCHITECTURE.md)

A documentação desta etapa **deve espelhar explicitamente** as decisões arquiteturais já adotadas:

- Showrunner model e orquestração multi-agent (Decisões 1, 5, 6)
- Isolamento por Git worktrees e merge progressivo em tiers (Decisões 2, 4)
- SQLite WAL como comunicação, memória e auditoria (Decisões 3, 7, 12)
- CLI + TUI Kanban/Detail View como interface humana (Decisão 10)
- Stack Node.js + TypeScript + vitest + Ink (Decisão 11)
- Verificação proporcional ao risco (Decisão 8)

Se qualquer documento contradizer isso, o documento está incorreto mesmo que “bem escrito”.

## Checklist detalhado

### 1) API documentation
- [ ] API documentation

### O QUE É
Documentação técnica da API TypeScript pública (e semi-pública, quando necessário) para uso programático, integração interna e manutenção de contratos.
Deve cobrir:
- módulos exportados;
- tipos e interfaces relevantes;
- classes/funções públicas;
- invariantes e contratos de erro;
- exemplos de uso com foco no runtime do orquestrador.

### O QUE NÃO É
- Não é tutorial de CLI para iniciantes.
- Não é transcrição automática de todo símbolo interno sem curadoria.
- Não é documento de arquitetura (arquitetura segue em `ARCHITECTURE.md`).

### COMO implementar
1. **Padronizar comentários TSDoc** (`/** ... */`) no código exportado.
2. **Gerar documentação com TypeDoc** (fonte da verdade para referência de API).
3. **Configurar arquivo de configuração** (`typedoc.json` ou `typedocOptions` no `tsconfig`/`package.json`).
4. **Habilitar plugin de cobertura de documentação** para medir superfície documentada.
5. **Adicionar validação em CI** para detectar regressão de cobertura e drift da referência.
6. **Publicar saída consolidada** em `docs/api.md` (ou índice equivalente) com links para detalhes.
7. **Validar snippets contra documentação viva** (Context7 quando disponível + docs oficiais do ecossistema).

Snippet recomendado (`typedoc.json`):

```json
{
  "$schema": "https://typedoc.org/schema.json",
  "entryPoints": ["src/index.ts"],
  "out": "docs/api-site",
  "tsconfig": "tsconfig.json",
  "excludePrivate": true,
  "excludeProtected": true,
  "excludeInternal": true,
  "plugin": ["typedoc-plugin-coverage"],
  "requiredToBeDocumented": ["Class", "Interface", "Function", "Method", "TypeAlias", "Property"]
}
```

Scripts recomendados:

```json
{
  "scripts": {
    "docs:api": "typedoc",
    "docs:api:check": "typedoc --logLevel Verbose"
  }
}
```

Padrão de comentário mínimo por símbolo público:
- resumo curto;
- `@remarks` quando houver nuance;
- `@param`/`@returns`;
- `@example` com uso realista;
- tags de release quando aplicável (`@public`, `@beta`, `@internal`).

### CRITÉRIO DE ACEITAÇÃO
- `docs:api` executa sem erro.
- API pública principal está documentada com exemplos úteis.
- Cobertura mínima de documentação definida e validada automaticamente.
- Mudanças de API sem documentação passam a falhar no gate da etapa.

### RISCOS
- **Drift entre código e docs geradas:** mitigar com geração automática em CI.
- **Cobertura superficial (“comentário vazio”):** mitigar com review semântico + exemplos obrigatórios.
- **Exposição de internos:** mitigar com exclusão explícita (`excludeInternal`, release tags).

### 2) Contributing guide
- [ ] Contributing guide

### O QUE É
Guia prático para contribuição em `CONTRIBUTING.md`, com fluxo completo:
setup local, convenções, testes, estilo, abertura de issue/PR e critérios de review.

### O QUE NÃO É
- Não é um manifesto de arquitetura (isso pertence ao `ARCHITECTURE.md`).
- Não é um dump de regras genéricas sem comandos reais.
- Não é apenas “abra um PR”.

### COMO implementar
Estrutura mínima recomendada para `CONTRIBUTING.md`:

1. **Welcome + escopo de contribuições aceitas**
2. **Pré-requisitos de ambiente** (Node LTS, Git, etc.)
3. **Setup local passo a passo** (instalação, scripts de dev/test)
4. **Fluxo de trabalho** (branch, commit, push, PR)
5. **Padrões de qualidade** (testes, lint, tipagem, documentação)
6. **Como reportar bugs e propor features** (com templates)
7. **Code of Conduct + canais de comunicação**

Incluir checklists acionáveis:

```md
## PR Checklist
- [ ] Mudança cobre requisito definido
- [ ] Testes relevantes passam localmente
- [ ] Documentação atualizada (quando aplicável)
- [ ] Commits claros e foco único por PR
```

Alinhar explicitamente com o modelo do projeto:
- multi-agent orchestration;
- SQLite como estado/telemetria;
- worktrees por agente;
- TUI/CLI como superfícies oficiais de uso.

### CRITÉRIO DE ACEITAÇÃO
- Um contribuidor novo consegue fazer setup, rodar validações e abrir PR sem ajuda externa.
- O guia reduz PRs inválidos por ausência de contexto/processo.
- O arquivo é descobrível pelo GitHub (`CONTRIBUTING.md` no root, `.github` ou `docs/`).

### RISCOS
- **Guia desatualizado conforme scripts evoluem:** mitigar com revisão obrigatória em mudanças de DX.
- **Overload de informação:** mitigar com estrutura por seções e “quick path” no topo.
- **Ambiguidade de qualidade:** mitigar com checklists objetivos e comandos executáveis.

### 3) Examples: common workflows with expected output
- [ ] Examples: common workflows with expected output

### O QUE É
Biblioteca de exemplos reais em `examples/` mostrando fluxos típicos fim-a-fim com:
- contexto do cenário;
- comando(s) executado(s);
- saída esperada;
- resultado observável (arquivos alterados, estado no SQLite, status no Kanban/TUI).

### O QUE NÃO É
- Não é uma coleção aleatória de snippets sem contexto.
- Não é documentação “feliz” apenas de sucesso; incluir erros comuns e troubleshooting.
- Não é benchmark de performance.

### COMO implementar
Aplicar padrão **example-driven** com organização por tarefa do usuário:

Estrutura sugerida:

```text
examples/
  01-single-task.md
  02-multi-agent-feature.md
  03-merge-conflict-escalation.md
  04-recovery-after-crash.md
  05-doc-update-flow.md
```

Template obrigatório por exemplo:

```md
# <Nome do fluxo>

## Objetivo
## Pré-condições
## Comandos
## Saída esperada
## Resultado esperado no sistema
## Falhas comuns e correção
```

Boas práticas:
- priorizar cenários de alto valor e alta frequência;
- usar dados determinísticos (fixtures) para evitar saída não reprodutível;
- manter exemplos curtos, mas completos;
- separar exemplos por papel: operador do sistema vs contribuidor.

Adotar taxonomia de documentação para navegabilidade (Diátaxis):
- tutorial (onboarding),
- how-to (resolver tarefa),
- reference (API/comandos),
- explanation (racional técnico).

### CRITÉRIO DE ACEITAÇÃO
- Cada exemplo pode ser seguido por alguém novo com resultado equivalente.
- Saída esperada está explícita e comparável.
- Existe cobertura de fluxos positivos e de recuperação de erro.

### RISCOS
- **Exemplos quebram com mudanças de UX/CLI:** mitigar com revisão em toda alteração de comandos.
- **Exemplo não reproduzível:** mitigar com fixtures e observáveis estáveis.
- **Cobertura enviesada para casos felizes:** mitigar com seção obrigatória de falhas comuns.

### 4) Video demo
- [ ] Video demo

### O QUE É
Demonstração visual do fluxo end-to-end do HUU, focada em compreensão rápida do produto por usuários e contribuidores.

### O QUE NÃO É
- Não é material de marketing sem conteúdo técnico.
- Não é screencast improvisado sem script reproduzível.
- Não é substituto da documentação textual.

### COMO implementar
Recomendação principal para CLI/TUI:
- **VHS** para demos determinísticas e versionáveis (“demo as code”).
- **asciinema** para capturas interativas leves e incorporáveis.
- **terminalizer** apenas como alternativa quando GIF customizado for prioridade.

Estratégia recomendada:
1. Escrever roteiro em arquivo `.tape` (VHS).
2. Gerar saída (`.gif`/`.mp4`) com resolução legível.
3. Publicar artefato em local estável (`docs/demo/` ou release assets).
4. Linkar no `README.md` e nos exemplos relevantes.

Exemplo mínimo (`demo.tape`):

```text
Output docs/demo/huu-e2e.gif
Set Width 1200
Set Height 700
Type "huu run \"Implementar feature X\""
Enter
Sleep 3s
Type "huu status"
Enter
Sleep 2s
```

### CRITÉRIO DE ACEITAÇÃO
- Demo executa fluxo principal sem edição manual frame-a-frame.
- Leitura de comandos e saídas é legível.
- Link de demo está presente e funcional na documentação principal.

### RISCOS
- **Demo fica obsoleta com evolução da interface:** mitigar com roteiro versionado e regeneração automática.
- **Arquivo pesado ou ilegível:** mitigar com controle de resolução e compressão adequada.
- **Dependência de ferramenta específica:** mitigar documentando fallback (asciinema/terminalizer).

### 5) README badges e discoverability (cross-cutting)
- [ ] README badges e discoverability

### O QUE É
Curadoria de badges relevantes no `README.md` para comunicar estado do projeto rapidamente.

### O QUE NÃO É
- Não é “coleção de badges decorativos”.
- Não é substituir documentação por indicadores visuais.

### COMO implementar
Manter só badges com sinal objetivo:
- build/CI status;
- test/coverage;
- versão;
- licença.

Usar Shields.io com consistência de estilo e links válidos.
Evitar clutter: no máximo conjunto pequeno no topo, em uma linha lógica.

### CRITÉRIO DE ACEITAÇÃO
- Badges principais funcionam e apontam para fontes reais.
- README permanece escaneável, sem poluição visual.

### RISCOS
- **Badge quebrado/desatualizado:** mitigar com revisão periódica e automação.
- **Ruído visual:** mitigar com política de mínimo necessário.
